# 2026-07-13 函数分片并发 CPG 构图提案

## 背景

当前目录级分析已经支持文件级有界并发，但对超大单文件，主要耗时仍集中在单文件分析内部。

现状大致是：

- 文件之间可以并发分析
- 单文件主链仍以整文件为分析单元
- 单文件会先构建整文件 CPG，再进入后续规则分析
- 对超大文件，整文件构图和整文件上下文准备会形成明显长尾

本提案只处理这一类问题：

- 保留文件作为语义上下文边界
- 对超大单文件，在 CPG 构图阶段引入函数分片并发
- 并发构建函数级子图后，再统一合并为文件级总图

本提案不直接改动规则阶段的执行单元，不把 mark / propagate / lift / decide / rewrite 改成函数级独立执行。

## 目标

### 主要目标

- 降低超大单文件的 CPG 构图长尾时间
- 保持现有文件级分析语义不变
- 保持后续规则阶段继续消费文件级统一图

### 非目标

- 不把整条删除规则链改成函数级执行
- 不把 rewrite 改成函数级独立提交
- 不在第一版处理跨文件调度策略重写

## 核心思路

把单文件分析拆成三层：

1. 文件级骨架阶段
2. 函数级子图阶段
3. 文件级合并阶段

其中：

- 文件级骨架阶段串行
- 函数级子图阶段并发
- 文件级合并阶段串行或低并发收口

这不是“函数级分析替代文件级分析”，而是“文件级上下文下的函数级并发构图”。

## 分阶段设计

## 1. 文件级骨架阶段

该阶段负责生成不能安全切分的共享上下文，只做轻量骨架工作，不做重型局部图构造。

建议包含：

- `SyntaxTree`
- `SemanticModel`
- 文件根节点
- 文件级根图节点
- 非函数成员骨架节点
- 文件级 symbol / type / declaration 索引
- 函数分片清单

这一阶段的职责是准备共享分析上下文，而不是构完整文件图。

### 非函数成员

以下结构不属于函数级分片，应继续由文件级骨架阶段负责：

- namespace
- type declaration
- field
- property declaration 外层
- event
- indexer declaration 外层
- delegate declaration
- base type / interface list
- attribute / generic constraints

## 2. 函数级子图阶段

这一阶段只针对超大文件启用，并发构建局部 CPG 子图。

### 第一版建议的分片单位

- `MethodDeclarationSyntax`
- `ConstructorDeclarationSyntax`
- `AccessorDeclarationSyntax`

### 第一版建议暂不独立切分的单位

- `LocalFunctionStatementSyntax`
- lambda / anonymous method

第一版建议把 local function 归属到外层方法分片中处理，避免过早引入 capture 和归属关系复杂度。

### 每个分片应输出的内容

- 局部 syntax nodes
- 局部 token nodes
- 局部 operation nodes
- 局部 method abstraction nodes
- 局部 intraprocedural CFG
- 局部 intraprocedural DataFlow / DDG 风格边
- 对共享 symbol / type / method 节点的引用需求
- 一个可合并的 partition manifest

### 分片阶段约束

- 分片内部不直接写主图
- 分片内部不直接竞争全局节点字典
- 分片内部只生成局部节点、局部边、共享引用需求

## 3. 文件级合并阶段

该阶段负责把多个函数分片的局部子图合并回文件级总图。

建议职责：

- 节点归并
- 边归并
- `ORDER` 统一
- symbol / type / method 共享节点去重
- partition 到文件根的连接
- 跨分片调用和引用关系补边
- 全图 validator 收口

### 合并后再执行的能力

以下能力不建议在 partition 内闭合，建议放到文件级合并后：

- call graph
- method return 到 callsite 的跨函数连接
- member access linking
- 其他依赖全图完整性的补边

## 为什么不直接做函数级完整分析单元

当前删除规则链里，很多决策不是函数封闭的。

例如：

- 方法签名收缩
- public / non-private 方法参数收缩
- delegate 签名联动
- local function 调用点联动
- field / property / event / indexer 删除
- if / else if / switch 等结构重写
- 最终 rewrite span 冲突收口

这些能力都依赖文件级统一视角。

因此本提案明确不采用以下做法：

- 每个函数单独建完整 `RuleContext`
- 每个函数单独跑 mark / propagate / lift / decide / rewrite
- 最后再把 rewrite 结果拼回文件

这样做会同时破坏：

- 决策冲突收口
- 跨函数调用一致性
- rewrite span 合并
- 声明级改写归属

## 共享节点策略

并发阶段最大的风险不是线程数本身，而是共享节点竞争。

以下共享结构不建议在 partition 内直接写入：

- symbol nodes
- type declaration nodes
- method nodes
- method parameter nodes
- method return nodes
- method entry nodes
- method exit nodes

建议策略：

- partition 只产出稳定 key 和引用需求
- 文件级合并阶段统一 `GetOrCreate`

这样可以避免：

- 重复节点
- 锁竞争
- 节点 id 不稳定

## 触发条件

不建议对所有文件一律启用函数分片并发。

建议只对“大文件”启用。

第一版可采用简单阈值：

- `lineCount >= N`

后续可扩展为组合阈值：

- `lineCount`
- `syntaxNodeCount`
- `methodCount`

第一版建议先只用 `lineCount`，降低实现复杂度。

## 第一版建议范围

第一版不追求全 pass 并发化，建议只做到：

- 文件级骨架串行
- 顶层 method / constructor / accessor 分片并发
- partition 内构建：
  - syntax subtree
  - operation subtree
  - method abstraction
  - intraprocedural CFG
  - intraprocedural DataFlow
- 文件级合并
- 合并后统一补：
  - call graph
  - member access
  - cross-method edges
  - validation

## 预期收益

该方案主要针对以下场景：

- 单个超大文件包含大量相对独立的方法
- 当前整文件构图长尾明显
- 文件级并发已经不足以继续降低总耗时

预期收益点：

- 超大单文件构图时间下降
- 大文件 tail latency 下降
- 后续规则阶段可继续消费文件级统一图，无需先重写规则契约

## 主要风险

### 1. 图合并语义一致性

重点风险包括：

- method child 顺序
- syntax node id 稳定性
- shared symbol / type 去重
- operation owner 归属
- cross-partition 边补全
- validator 预期结构不一致

### 2. 分片边界不合理

若 local function、accessor、constructor 的归属处理不当，容易引入：

- capture 关系错位
- containing method 错位
- callsite owner 错位

### 3. 并发收益不及合并成本

如果文件规模不够大，分片和合并本身可能抵消收益。

因此必须保守启用，只针对超大文件。

## 实施建议

建议按以下顺序推进：

1. 定义 partition 模型和 merge manifest
2. 先在分析阶段引入“超大文件开关”
3. 先把 syntax / operation / intraprocedural CFG / intraprocedural DataFlow 下放到 partition
4. 把共享节点统一留在合并阶段归并
5. 合并后再统一执行 call graph / cross-partition link / validator
6. 再决定是否扩大到更多 pass

## 不建议的实现方式

以下做法不建议采用：

- 先把 `DeletionApplicationService` 改成函数级任务分发
- 先把 rule 阶段改成函数级 `RuleContext`
- 先把 rewrite 改成函数级提交
- 先把 local function 拆成独立 partition

这些做法都会在第一版把复杂度抬得过高。

## 推荐结论

推荐采用：

- 文件级上下文边界不变
- 超大文件启用函数分片并发构图
- 分片只负责局部子图生成
- 文件级合并后继续走现有规则阶段

这是当前体系下风险最低、最可能先看到真实收益的路线。
