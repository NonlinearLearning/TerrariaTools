# 边界约束

本页说明当前仓库的层间边界。你在移动代码前先读这一页。

## 依赖方向

建议按下面的方向理解依赖：

1. `src/Core` 定义稳定模型和规则服务。
2. `src/Application/Ports` 定义应用层公共契约。
3. `src/Application` 组织阶段和用例。
4. `src/Adapters` 实现端口。
5. `apps/*` 负责组合根、宿主和入口。

换句话说，越靠近 `apps/`，越接近入口；越靠近 `Core/`，越接近领域模型。

## 各层应该放什么

### 放到 `src/Core`

当代码满足下面任一条件时，优先放到 `Core`：

- 它表达的是稳定领域概念
- 它不需要 Roslyn、文件系统或 JSON 细节
- 它会被多个流程共享
- 它更像规则或计划逻辑，而不是 I/O 逻辑

### 放到 `src/Application`

当代码满足下面任一条件时，优先放到 `Application`：

- 它负责串联多个步骤
- 它定义请求、结果或端口接口
- 它属于用例流程，而不是底层能力实现

### 放到 `src/Adapters`

当代码满足下面任一条件时，优先放到 `Adapters`：

- 它直接调用 Roslyn
- 它直接读写文件或目录
- 它直接执行构建进程
- 它直接做 JSON 序列化或反序列化

### 放到 `apps`

当代码满足下面任一条件时，优先放到 `apps`：

- 它是对外宿主入口
- 它负责默认依赖装配
- 它负责 CLI 参数解析或命令分发

## 当前仓库中的重要边界

- `Core` 不应该依赖 `Adapters`。
- `Application.Ports` 不应该暴露 Roslyn 语义对象。
- 运行时流程和 shadow extraction 使用各自的 pipeline，不把专用阶段塞回标准流程。
- `Dome.Cli` 只负责解析和分发，不负责实现分析、规则或重写逻辑。

## 当前实现中的现实约束

当前实现里，运行时宿主和 shadow 宿主会复用标准 `Dome.Application` 的宿主工厂与共享服务。这说明它们仍共享一部分宿主层实现。你在调整这部分代码时，要先判断你是在复用标准能力，还是在把专用逻辑倒灌回标准主线。

## 改动前的检查清单

在你新建类型或移动代码前，先检查：

1. 这段代码是否依赖 Roslyn 或文件系统？
2. 这段代码是否会被多个流程共享？
3. 这段代码表达的是“领域规则”还是“执行步骤”？
4. 公开契约里是否出现了不该泄漏的基础设施类型？
