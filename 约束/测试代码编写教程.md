# 测试代码编写教程

这份教程面向 .NET / C# 项目，但大部分方法也适用于其他语言。

## 1. 先搞清楚：你到底要测什么

测试不是“把代码跑一遍”，而是验证**行为**。优先测这些维度：

| 维度 | 该测什么 | 示例 |
| --- | --- | --- |
| 正常流程 | 输入合法时，输出是否正确 | `Add("1,2") == 3` |
| 分支逻辑 | `if / switch / 早返回` 是否都覆盖 | 折扣规则、权限判断 |
| 边界值 | 最小值、最大值、空值、空集合 | `null`、`""`、`0`、`int.MaxValue` |
| 异常路径 | 非法输入是否抛出正确异常 | `ArgumentException`、`InvalidOperationException` |
| 状态变化 | 调用前后对象状态是否变化 | `IsCompleted`、`Balance` |
| 协作对象 | 外部依赖是否被正确调用 | 仓储、HTTP 客户端、消息队列 |
| 副作用 | 文件、数据库、缓存、日志、消息是否发生 | 写文件成功、发消息成功 |
| 回归点 | 以前出过 bug 的场景 | 某个输入曾导致崩溃 |

## 2. 单元测试的基本原则

优质单元测试通常满足这 5 点：

1. 快
2. 独立
3. 可重复
4. 自动判定结果
5. 写起来成本低

## 3. 代码层面应该怎么写

### 3.1 测试 public 行为，不测 private 方法

不要直接测私有方法。应该从 public 方法进入，验证最终行为。

```csharp
public class TextParser
{
    public string Parse(string input)
    {
        return Trim(input);
    }

    private string Trim(string input) => input.Trim();
}
```

测试应写成：

```csharp
[Fact]
public void Parse_ReturnsTrimmedText()
{
    var parser = new TextParser();

    var actual = parser.Parse(" a ");

    Assert.Equal("a", actual);
}
```

### 3.2 用 AAA 结构

- Arrange：准备数据和依赖
- Act：执行一个动作
- Assert：断言结果

```csharp
[Fact]
public void Add_TwoNumbers_ReturnsSum()
{
    // Arrange
    var calculator = new StringCalculator();

    // Act
    var actual = calculator.Add("1,2");

    // Assert
    Assert.Equal(3, actual);
}
```

### 3.3 一个测试只验证一个行为

不要把多个 Act 混在一起。拆成多个测试，或者用参数化测试。

```csharp
[Theory]
[InlineData("", 0)]
[InlineData("1", 1)]
[InlineData("1,2", 3)]
public void Add_ReturnsExpectedSum(string input, int expected)
{
    var calculator = new StringCalculator();

    var actual = calculator.Add(input);

    Assert.Equal(expected, actual);
}
```

### 3.4 避免写实现细节测试

不要测试“内部怎么写”，要测试“对外表现如何”。

不好：

```csharp
// 不建议：测试内部某个 helper 的存在感
```

好：

```csharp
[Fact]
public void Checkout_WhenCartHasItems_ReturnsTotalPrice()
{
    // 只验证用户能感知到的结果
}
```

## 4. 从哪些维度编写测试

### 4.1 输入维度

从输入数据的不同形态去测：

- `null`
- 空字符串 `""`
- 空集合 `[]`
- 单个元素
- 多个元素
- 边界值（`0`、`1`、`-1`、`int.MaxValue`）
- 超长输入

示例：

```csharp
[Theory]
[InlineData(null)]
[InlineData("")]
[InlineData("   ")]
public void Normalize_WhenInputIsEmpty_ThrowsArgumentException(string? input)
{
    var validator = new InputValidator();

    Assert.Throws<ArgumentException>(() => validator.Normalize(input!));
}
```

### 4.2 分支维度

代码里有 `if / else / switch / 三元表达式`，每条分支都要测。

```csharp
public decimal CalculatePrice(decimal price, bool vip)
{
    return vip ? price * 0.9m : price;
}
```

对应测试：

```csharp
[Theory]
[InlineData(100, true, 90)]
[InlineData(100, false, 100)]
public void CalculatePrice_ReturnsExpectedPrice(decimal price, bool vip, decimal expected)
{
    var service = new PricingService();

    var actual = service.CalculatePrice(price, vip);

    Assert.Equal(expected, actual);
}
```

### 4.3 异常维度

当输入非法时，测试应该明确验证异常类型，而不是只看“会不会炸”。

```csharp
[Fact]
public void Withdraw_WhenAmountIsNegative_ThrowsArgumentOutOfRangeException()
{
    var account = new BankAccount(100);

    Assert.Throws<ArgumentOutOfRangeException>(() => account.Withdraw(-1));
}
```

### 4.4 状态变化维度

当方法会改对象状态时，验证调用前后状态。

```csharp
[Fact]
public void Complete_MarksTaskAsCompleted()
{
    var task = new TodoItem();

    task.Complete();

    Assert.True(task.IsCompleted);
}
```

### 4.5 协作对象维度

当类依赖外部服务时，测试“有没有正确调用依赖”。

```csharp
public interface IEmailSender
{
    Task SendAsync(string to, string subject, string body);
}

public class OrderService
{
    private readonly IEmailSender _emailSender;

    public OrderService(IEmailSender emailSender)
    {
        _emailSender = emailSender;
    }

    public async Task PlaceAsync(string customerEmail)
    {
        await _emailSender.SendAsync(customerEmail, "Order placed", "Your order was placed.");
    }
}
```

配合 Moq：

```csharp
[Fact]
public async Task PlaceAsync_SendsConfirmationEmail()
{
    var emailSender = new Mock<IEmailSender>();
    var service = new OrderService(emailSender.Object);

    await service.PlaceAsync("a@b.com");

    emailSender.Verify(x => x.SendAsync(
        "a@b.com",
        "Order placed",
        "Your order was placed."), Times.Once);
}
```

## 5. 什么时候用 mock，什么时候只断言返回值

- 只关心结果时，用状态断言
- 关心调用过程时，用 mock 验证交互
- 关心可替代实现时，用 stub/fake 提供固定返回值

示例：

```csharp
public interface IClock
{
    DateTime Now { get; }
}

public class DiscountService
{
    private readonly IClock _clock;

    public DiscountService(IClock clock)
    {
        _clock = clock;
    }

    public decimal GetPrice(decimal price)
    {
        return _clock.Now.DayOfWeek == DayOfWeek.Tuesday ? price * 0.5m : price;
    }
}
```

测试：

```csharp
[Fact]
public void GetPrice_OnTuesday_ReturnsHalfPrice()
{
    var clock = new Mock<IClock>();
    clock.Setup(x => x.Now).Returns(new DateTime(2026, 5, 5));

    var service = new DiscountService(clock.Object);

    var actual = service.GetPrice(100m);

    Assert.Equal(50m, actual);
}
```

## 6. 命名规则

推荐格式：`方法_场景_期望结果`

例如：

- `Add_EmptyString_ReturnsZero`
- `Withdraw_InsufficientBalance_ThrowsInvalidOperationException`
- `PlaceAsync_WhenEmailIsValid_SendsConfirmationEmail`

好名字能直接说明测试意图。

## 7. 断言怎么写更稳

优先断言：

1. 返回值
2. 抛出的异常类型和消息
3. 状态变化
4. 外部依赖调用次数和参数

尽量少断言：

- 实现细节
- 私有字段
- 无关对象

## 8. 哪些代码更该优先写测试

优先级从高到低：

1. 复杂业务规则
2. 过去出过 bug 的代码
3. 分支很多的函数
4. 计算逻辑
5. 状态机/流程控制
6. 外部调用封装

不必优先写测试的：

- 简单 DTO
- 只有属性的模型
- 纯样板代码

## 9. 常见坏味道

- 一个测试里写太多逻辑
- 测试依赖时间、随机数、网络、数据库
- 测试名字和内容不一致
- 断言过多但无重点
- 只测 happy path，不测失败路径
- 只追求覆盖率，不追求真实风险

## 10. 推荐的项目结构

```text
src/
  App/
tests/
  App.Tests/
```

建议让测试项目和生产项目一一对应：

- `src/Foo` -> `tests/Foo.Tests`
- 测试命名与生产代码保持一致

## 11. 测试代码模板 / 脚手架

下面是一套可以直接复制的测试文件结构，适合 .NET / xUnit / Moq。

### 11.1 推荐目录结构

```text
tests/
  App.Tests/
    App.Tests.csproj
    Fixtures/
      TestData.cs
    Services/
      OrderServiceTests.cs
    Models/
      PriceCalculatorTests.cs
```

### 11.2 测试项目模板

`App.Tests.csproj`：

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
    <IsPackable>false</IsPackable>
    <IsTestProject>true</IsTestProject>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.NET.Test.Sdk" Version="17.13.0" />
    <PackageReference Include="xunit" Version="2.9.3" />
    <PackageReference Include="xunit.runner.visualstudio" Version="3.1.1">
      <IncludeAssets>runtime; build; native; contentfiles; analyzers; buildtransitive</IncludeAssets>
      <PrivateAssets>all</PrivateAssets>
    </PackageReference>
    <PackageReference Include="Moq" Version="4.20.72" />
  </ItemGroup>

  <ItemGroup>
    <ProjectReference Include="..\..\src\App\App.csproj" />
  </ItemGroup>
</Project>
```

### 11.3 单元测试类模板

```csharp
using Moq;
using Xunit;

namespace App.Tests.Services;

public class OrderServiceTests
{
    [Fact]
    public async Task PlaceAsync_WhenCalled_SendsEmail()
    {
        // Arrange
        var emailSender = new Mock<IEmailSender>();
        var sut = new OrderService(emailSender.Object);

        // Act
        await sut.PlaceAsync("a@b.com");

        // Assert
        emailSender.Verify(x => x.SendAsync(
            "a@b.com",
            "Order placed",
            "Your order was placed."), Times.Once);
    }
}
```

### 11.4 数据驱动测试模板

```csharp
using Xunit;

namespace App.Tests.Models;

public class PriceCalculatorTests
{
    [Theory]
    [InlineData(0, 0)]
    [InlineData(1, 1)]
    [InlineData(10, 10)]
    public void Calculate_ReturnsExpectedValue(decimal input, decimal expected)
    {
        // Arrange
        var sut = new PriceCalculator();

        // Act
        var actual = sut.Calculate(input);

        // Assert
        Assert.Equal(expected, actual);
    }
}
```

### 11.5 测试数据模板

```csharp
namespace App.Tests.Fixtures;

public static class TestData
{
    public const string ValidEmail = "a@b.com";
    public const string ValidSubject = "Order placed";
    public const string ValidBody = "Your order was placed.";
}
```

### 11.6 复制后你要改的地方

1. 把 `App.Tests` 改成你的测试项目名
2. 把 `ProjectReference` 改成你的实际生产项目路径
3. 把 `namespace` 改成你的命名空间
4. 把 `IEmailSender`、`OrderService`、`PriceCalculator` 换成你的真实类型
5. 把 `InlineData` 改成你的边界值和分支值

## 12. 一个实用的写测试清单

每次写测试前，先问自己：

1. 这个方法的公开行为是什么？
2. 它有哪些输入边界？
3. 它有哪些分支？
4. 有哪些异常路径？
5. 它是否依赖时间、随机数、网络、数据库？
6. 是否需要 mock/stub/fake？
7. 这个测试是否只验证一个重点？
8. 测试名是否能直接看懂？

## 13. 结论

写测试的核心不是“覆盖更多代码”，而是“把最容易出问题的行为锁住”。

优先写：

- 业务规则
- 分支逻辑
- 边界值
- 异常路径
- 协作对象交互

少写：

- 实现细节
- 私有方法
- 和外部环境强耦合的代码
