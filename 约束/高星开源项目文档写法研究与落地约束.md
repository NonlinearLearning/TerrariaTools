# 高星开源项目文档写法研究与落地约束

## 1. 目的

本文不是泛泛总结“别人文档写得好”，而是把高星开源项目的项目文档写法拆成可执行规则，落到本仓库后续文档编写、重写、补齐、评审中。

目标只有一个：

**以后本仓库写项目文档时，优先复用高星开源项目已经验证过的结构、入口、分层和页面契约，而不是凭感觉堆内容。**

本文面向 AI 执行，也面向仓库文档作者执行。

---

## 2. 外部样本范围

本轮只选：

1. GitHub 星数高；
2. 官方文档入口清晰；
3. 文档结构成熟；
4. 不同技术栈都有代表性。

检索时间：**2026-04-28**

### 2.1 React

- GitHub：`facebook/react`
- 观察到的星数：约 `245k stars`
- 仓库入口：<https://github.com/facebook/react>
- 文档入口：<https://react.dev/learn>
- 参考页入口：<https://react.dev/reference/react>
- 安装页入口：<https://react.dev/learn/installation>

提取信号：

- README 里直接给出 `Installation`、`Documentation`、`Examples`、`Contributing`。
- 文档把“学习路径”和“参考手册”明确拆开。
- 新手页明确告诉读者“这页能学到什么”。
- 大量页面直接给可运行代码，不让用户只看抽象说明。

### 2.2 Next.js

- GitHub：`vercel/next.js`
- 观察到的星数：约 `137k stars`
- 仓库入口：<https://github.com/vercel/next.js>
- 文档入口：<https://nextjs.org/docs>

提取信号：

- 文档首页直接声明分成 `Getting Started`、`Guides`、`API Reference` 三大块。
- 文档先讲“如何使用这套文档”，而不是直接把信息砸给用户。
- 对不同路由体系给出明确切换入口，避免不同版本/模式混在一起。
- 先给最短上手路径，再给专题指南，再给细参考。

### 2.3 FastAPI

- GitHub：`fastapi/fastapi`
- 观察到的星数：约 `97.4k stars`
- 仓库入口：<https://github.com/fastapi/fastapi>
- 文档入口：<https://fastapi.tiangolo.com/>
- 教程入口：<https://fastapi.tiangolo.com/tutorial/>

提取信号：

- 首页先说产品价值，再把文档入口和源码入口放一起。
- 教程明确说明：既可以按顺序学，也可以直接跳到某个主题解决眼前问题。
- 代码块强调“可以直接复制运行”，降低读者试错成本。
- 文档既服务初学者，也服务查 API 的老用户。

### 2.4 Kubernetes

- GitHub：`kubernetes/kubernetes`
- 观察到的星数：约 `119k stars`
- 仓库入口：<https://github.com/kubernetes/kubernetes>
- 文档入口：<https://kubernetes.io/docs/>
- Concepts：<https://kubernetes.io/docs/concepts/>
- Tasks：<https://kubernetes.io/docs/tasks/>
- Reference：<https://kubernetes.io/docs/reference/>

提取信号：

- 首页按用户任务切分，而不是按仓库目录切分。
- 文档顶层清晰分成：`Concepts`、`Tutorials`、`Tasks`、`Reference`、`Contribute`。
- `Tasks` 页面明确约束：一页只讲一件事，给短步骤。
- `Reference` 与 `Concepts` 强拆，避免“原理页”和“查表页”互相污染。
- 明确提供版本与语言切换入口。

### 2.5 Django

- GitHub：`django/django`
- 观察到的星数：约 `87.3k stars`
- 仓库入口：<https://github.com/django/django>
- 文档入口：<https://docs.djangoproject.com/>
- API Reference：<https://docs.djangoproject.com/en/4.2/ref/>

提取信号：

- 首页把 `First steps`、`Tutorial`、`Advanced Tutorials` 摆在非常靠前的位置。
- 文档首页明确给版本选择器与语言切换。
- API Reference 独立成完整分区，不和教程内容混写。
- README 只做总入口，不把全部文档硬塞进仓库首页。

### 2.6 TypeScript

- GitHub：`microsoft/TypeScript`
- 观察到的星数：约 `109k stars`
- 仓库入口：<https://github.com/microsoft/TypeScript>
- 文档入口：<https://www.typescriptlang.org/docs/>
- Handbook：<https://www.typescriptlang.org/docs/handbook/intro>

提取信号：

- 文档首页按读者背景给不同起步入口。
- Handbook 明确写出“目标”“结构”“非目标”，先告诉读者这套文档覆盖什么、不覆盖什么。
- `Handbook` 和 `Reference` 有明确边界，不互相替代。
- 文档承认“不是语言规范本身”，主动声明范围，减少误解。

## 3. 从这些项目里提炼出的稳定共同点

下面不是主观喜好，而是多个高星项目重复出现的写法。

### 3.1 文档先按“读者任务”分，不按仓库目录分

高星项目很少让用户先理解仓库目录树，再去猜应该看哪一页。

它们更常见的切法是：

- 我是第一次接触：看 `Getting Started` / `Quick Start` / `First steps`
- 我想理解原理：看 `Concepts` / `Learn` / `Handbook`
- 我只想完成一件事：看 `Tasks` / `How-to` / `Guides`
- 我只想查参数、命令、API：看 `Reference` / `API Reference`
- 我想参与项目：看 `Contributing`

结论：

**项目文档的一级导航，应先映射用户目的，再映射代码结构。**

### 3.2 必须把“学习路径”和“参考手册”分开

React、Next.js、TypeScript、Kubernetes、Django 都在做这件事。

稳定模式是：

- 新手学习页：按顺序看更舒服
- 任务页：按问题跳转更高效
- 参考页：适合查，不适合从头读到尾

结论：

**不要把教程、设计说明、命令参考、API 说明塞进同一篇长文里。**

### 3.3 首页不是摘要垃圾场，而是导航入口

成熟项目的首页通常只做几件事：

1. 说明这是什么项目
2. 说明适合谁
3. 给最快的开始路径
4. 给后续分流入口
5. 给贡献与社区入口

它们一般不会把所有细节都堆首页。

结论：

**项目首页应该负责分流，不应该承担全部说明书职责。**

### 3.4 每个页面都要有自己的页面契约

高质量文档页通常会告诉读者：

- 这一页解决什么问题
- 需要什么前置知识
- 你会学到什么
- 这一页结束后该去哪

React 的 `You will learn`、Next.js 的 `How to use the docs`、TypeScript Handbook 的 `How is this Handbook Structured` 都在做这件事。

结论：

**每篇关键文档都应明确“本页边界”。**

### 3.5 示例必须可复制、可运行、可定位

FastAPI、React 这点尤其明显。

稳定写法包括：

- 代码块能直接复制
- 明确保存到哪个文件
- 明确如何运行
- 示例围绕单一概念，不做多主题混杂

结论：

**文档里的例子不是装饰品，而是最小可执行证明。**

### 3.6 概念、任务、参考、贡献这四类内容要强拆

Kubernetes 这点最清楚，Django 和 TypeScript 也很稳定。

四类内容本质不同：

- 概念：解释是什么、为什么
- 任务：一步一步做成一件事
- 参考：参数、API、命令、字段、边界定义
- 贡献：怎么参与、怎么提 PR、怎么验证

结论：

**内容类型不同，就不要假装它们能共享一种写法。**

### 3.7 版本边界必须显式

Next.js、Kubernetes、Django 都有明显的版本入口或版本提示。

结论：

**当文档内容会随版本变化时，必须显式标注版本、适用范围、过期状态。**

### 3.8 文档要承认非目标

TypeScript Handbook 明确写出 `Non-Goals`，这是很强的信号。

结论：

**好文档不只是说“讲什么”，还会说“这篇不讲什么”。**

这样能避免：

- 用户误读范围
- 一篇文档无限膨胀
- 本该拆页的内容继续挤在一页里

## 4. 这些项目没有采用的低质量写法

本轮样本里，几乎都在避免下面这些问题：

### 4.1 把首页写成超长流水账

坏味道：

- 项目简介、安装、教程、架构、FAQ、贡献、更新记录全塞一个 README

问题：

- 无法分流
- 读者找不到入口
- 更新时容易局部过期

### 4.2 用仓库目录替代用户视角

坏味道：

- `src`
- `core`
- `utils`
- `internal`

这种目录名对读者的任务没有帮助。

### 4.3 原理、步骤、参数混写

坏味道：

- 一段解释概念
- 紧接着贴安装步骤
- 再突然插 API 参数表

问题：

- 既不适合学习，也不适合查阅

### 4.4 没有页面边界

坏味道：

- 不知道这页给谁看
- 不知道要先懂什么
- 不知道看完后去哪里

### 4.5 例子不能直接跑

坏味道：

- 只有片段，没有上下文
- 不说明放在哪个文件
- 不说明怎么执行

## 5. 本仓库后续写项目文档时的落地约束

以后写本仓库的项目文档、模块文档、使用文档、架构文档，默认遵守下面这些规则。

### 5.1 先判定文档类型，再决定写法

落笔前先明确这篇属于哪一类：

1. `Getting Started / 快速开始`
2. `Concepts / 原理说明`
3. `How-to / Task / 操作步骤`
4. `Reference / API / 参数说明`
5. `Contributing / 开发与提交流程`

如果一篇文档同时想承担 2 类以上职责，优先拆文档，而不是继续加小标题。

### 5.2 项目总入口文档必须承担分流职责

项目总入口页至少包含：

1. 项目是什么
2. 适合谁
3. 最快开始入口
4. 进阶入口
5. 参考入口
6. 贡献入口

不要求首页解释完所有细节。

### 5.3 快速开始文档必须短

快速开始文档默认只回答：

1. 环境前置
2. 安装/启动
3. 第一个最小成功结果
4. 下一步去哪里

不要把架构总览、全部参数、全部边界条件塞进去。

### 5.4 原理文档必须显式说明边界

原理文档至少写清楚：

1. 这篇解释什么
2. 不解释什么
3. 需要什么前置知识
4. 与哪些代码、模块、流程相关

### 5.5 操作文档一页只做一件事

任务页默认约束：

- 一页只解决一个明确动作
- 步骤按顺序写
- 必要时给前置条件
- 给成功判断
- 给失败排查入口

### 5.6 参考文档不写成长教程

参考文档只做这些事：

- 定义字段/类型/API/命令/选项
- 说明输入输出
- 说明边界、默认值、限制
- 需要时补极小示例

不要把参考文档写成大段背景介绍。

### 5.7 所有关键文档都要有“读完下一步”

除纯参考页外，关键文档默认补一节：

- `下一步`

告诉读者下一篇应该看什么，而不是让用户自己猜。

### 5.8 文档示例必须贴近真实仓库

优先使用：

- 本仓库真实路径
- 本仓库真实命令
- 本仓库真实配置名
- 本仓库真实模块名

少写脱离仓库上下文的玩具例子。

### 5.9 版本/状态变化必须显式标注

当文档受版本、阶段、实验状态影响时，必须标：

- 适用版本
- 生效日期
- 是否草稿/实验/已过期

## 6. 推荐的信息架构模板

如果后续要补项目级文档，优先参考下面结构，而不是从零拍脑袋。

### 6.1 项目总入口模板

```md
# 项目名

一句话说明项目是什么。

## 适合谁

## 快速开始

## 核心概念

## 常见任务

## 参考资料

## 贡献与开发
```

### 6.2 快速开始模板

```md
# 快速开始

## 这篇解决什么问题

## 前置条件

## 步骤

## 成功结果

## 下一步
```

### 6.3 原理说明模板

```md
# 某原理 / 某模块说明

## 这篇解释什么

## 不解释什么

## 前置知识

## 核心概念

## 与代码的对应关系

## 常见误解

## 下一步
```

### 6.4 任务文档模板

```md
# 如何完成某任务

## 目标

## 前置条件

## 步骤

## 验证方式

## 常见问题

## 下一步
```

### 6.5 参考文档模板

```md
# 某 API / 命令 / 参数参考

## 适用范围

## 输入

## 输出

## 参数 / 字段定义

## 默认值与限制

## 最小示例
```

## 7. AI 执行这些文档任务时必须遵守的规则

### 7.1 先判断用户要的是哪类文档

禁止不分类就直接开写。

至少先判断是：

- 快速开始
- 原理说明
- 操作文档
- 参考文档
- 贡献文档

### 7.2 先补信息架构，再写正文

当用户要求“补项目文档”“重写文档”“参考优秀项目怎么写”时，AI 先给出拟采用的文档类型与章节结构，再写正文。

### 7.3 先参考官方成熟项目，再映射本仓库

默认优先参考：

- React
- Next.js
- FastAPI
- Kubernetes
- Django
- TypeScript

然后才落到本仓库具体内容。

### 7.4 不抄目录，要抄结构原则

允许学习的对象：

- 信息架构
- 页面分工
- 教程/参考分层
- 版本边界标法
- 页面契约写法

不应做的事：

- 直接照搬别人标题层级
- 直接照搬别人术语体系
- 直接照搬别人段落文案

### 7.5 文档必须能落到本仓库真实路径

如果 AI 输出的是文档，优先：

- 给出真实文件名
- 给出真实目录
- 给出真实代码锚点
- 给出真实后续补齐建议

不要只在聊天里讲抽象建议。

## 9. 验收标准

完成这类任务时，至少满足：

1. 使用的是官方项目入口或官方文档入口，而不是二手总结站。
2. 明确说明参考了哪些高星项目。
3. 提炼的是“结构规律”，不是只抄别人目录。
4. 输出能直接指导本仓库写文档，而不是停留在泛泛点评。
5. 给出至少一套本仓库可复用模板。
6. 文档已经实际落到 `约束/*.md`。

## 文档同步与实现约束

### 文档类型

本文属于：仓库约束文档。

### 代码对齐文档要求

- 后续若本仓库新增项目级 README、快速开始文档、参考文档、贡献文档，默认要参考本文结构。
- 若本文里的样本链接、观察结论、模板结构明显过期，应同步修正。
- 若后续 `AGENTS.md` 真的接入本文触发词，应同步更新本文的触发建议部分。

### 文档对齐代码要求

- 文档落地到仓库时，优先绑定真实模块、真实命令、真实路径。
- 文档若声称“可以这样运行 / 这样验证”，应尽量给出可执行命令或可核对路径。

### 默认代码锚点

- `AGENTS.md`
- `约束/*.md`
- `README*`
- `docs/**/*.md`

### 交付检查

- 本文已经给出样本来源；
- 本文已经提炼共同结构；
- 本文已经转化为本仓库可执行规则；
- 本文已经给出模板，而不是只给观点。
