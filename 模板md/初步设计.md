# 删除框架设计总览

## 1. 这份文档负责什么

这份文档是 `设计docs/` 的主入口，只负责四件事：

1. 说明删除框架的目标和总边界。
2. 固定系统级架构与全流程。
3. 说明文档之间的职责分工和阅读顺序。
4. 固定跨文档共用的术语。

这份文档不展开共享规则细节，不展开专项算法，也不逐字段解释内部对象和 C# 契约。

## 2. 设计目标

当前要设计的是一套基于 `NewJoern` 图事实的代码删除框架。

这套框架需要同时满足下面几条总约束：

1. 图分析、删除语义、源码回写必须分离。
2. 标记、传播、决策、计划、执行必须分阶段组织。
3. 同一目标允许承受多个标记，但最终只能产出一个动作。
4. 图证据不足、结构无法稳定回写、外部活引用无法排除时，默认保守跳过。
5. 未来允许宿主加载外部规则包，但外部规则包不能直接改源码。

## 3. 总体结论

当前采用的方向是：

```text
模块化单体 + 图分析先行 + 删除语义独立建模 + 源码回写后置
```

落到工程组织上，有三个结论：

1. 当前阶段先收清模块边界，不直接全量进入完整 DDD。
2. 图事实直接建立在 `NewJoern` 现有 API、schema 和 overlay 之上，不再平行发明第二套图模型。
3. `DeletionDecision` 是当前最值得单独演进的核心模块。

## 5. 总体架构

### 5.1 模块划分

```text
Host / CLI
  -> Workflow
    -> GraphAnalysis
    -> DeletionDecision
    -> SourceRewrite
    -> Reporting
    -> RuleHost

外围基础设施:
  - NewJoern Graph / Overlay
  - Roslyn Source Mapping
  - File System
  - Logging / Audit
```

### 5.2 模块职责

#### GraphAnalysis

- 负责加载和查询 `NewJoern` 图。
- 负责提供节点、边、源码位置、调用关系、数据依赖等事实。
- 不负责决定删什么。

#### DeletionDecision

- 负责标记、传播、决策、计划候选。
- 负责围绕“同一目标唯一最终动作”做语义裁决。
- 是规则语义最集中的模块。

#### SourceRewrite

- 负责把删除计划映射到源码位置。
- 负责执行表达式级、语句级、作用域级、定义级改写。
- 负责语法合法性和回写失败兜底。

#### Reporting

- 负责输出报告、审计记录、跳过原因和统计结果。
- 不反向控制主流程。

#### RuleHost

- 负责外部规则包加载、兼容性校验、注册和元数据治理。
- 不直接做源码删除。

## 6. 总体流程

### 6.1 全流程

```text
加载 -> 分析 -> 标记 -> 传播 -> 决策 -> 计划 -> 执行 -> 输出
```

### 6.2 各阶段职责

| 阶段 | 职责 | 产物 |
| --- | --- | --- |
| 加载 | 准备图、源码、配置、规则集 | 分析上下文 |
| 分析 | 查询图事实，不做删除裁决 | 图事实、候选输入 |
| 标记 | 产生原始待删标记 | `Mark` |
| 传播 | 基于已有标记扩展新标记 | 新 `Mark`、`PropagationTrace` |
| 决策 | 聚合同一目标上的多个标记，产出唯一最终动作 | `DecisionOutcome` |
| 计划 | 把语义动作转成技术可执行计划 | `DeletionPlan` |
| 执行 | 第一次真实改源码 | 重写结果 |
| 输出 | 发布结果、证据、报告 | 报告与审计记录 |

### 6.3 核心约束

这里固定一条跨文档共用约束：

```text
优先级 != 决策
优先级 ∈ 决策阶段
```

这条规则的正式定义放在 `2026-04-29-代码删除规则设计.md`。其他文档只引用，不重复展开整套解释。

## 7. 关键依赖边界

### 7.1 GraphAnalysis 与 NewJoern 的关系

GraphAnalysis 必须直接建立在 `NewJoern` 现有 API 和模型之上。

允许直接依赖的能力包括：

- `NewJoernGraph`
- `NewJoernNode`
- `NewJoernEdge`
- `NodeKind`
- `EdgeKind`
- `NodePropertyNames`
- `CallGraphOverlay`
- `ReachingDefOverlay`

禁止做法：

1. 再平行定义一套 `CallNode / MethodNode / DataFlowNode`。
2. 在删除框架里复制 `NewJoern` schema 语义。
3. 让图语义在 `NewJoern` 和删除框架里长期双轨并行。

### 7.2 外部规则边界

外部规则包可以提供：

- 标记规则
- 传播规则
- 决策策略建议

外部规则包不能直接做：

- 文件删除
- 源码回写
- 宿主一致性裁决

删除执行权保留在宿主框架内。具体 C# 契约见 `2026-04-29-外部规则接口-CSharp-设计稿.md`。

## 8. 统一术语

### 8.1 全流程术语

- 全流程：`加载 -> 分析 -> 标记 -> 传播 -> 决策 -> 计划 -> 执行 -> 输出`
- 规则语义子流程：`标记 -> 传播 -> 决策 -> 计划 -> 执行`

### 8.2 内部对象与外部契约

- `DeletionTarget`：内部删除语义目标。
- `Mark`：内部运行期标记结果。
- `DecisionOutcome`：内部决策阶段正式输出。
- `RuleManifest`、`IMarkRule`、`IPropagateRule`、`IDecisionPolicy`：外部规则包契约。

如果某份文档使用 `MarkTarget`、`RuleTargetKind` 这类类型，必须明确它属于外部契约层，而不是内部语义层。

### 8.3 模块名与局部组件名

- `SourceRewrite` 是模块名。
- `Planner`、`Rewriter`、`Remover` 只是模块内组件，不是和模块同级的新架构层。

## 9. 非功能总约束

全家文档共享下面这些非功能约束：

1. 删除必须基于图证据，不靠文本猜测。
2. 无法稳定裁决时，默认保守跳过。
3. 执行结果必须可解释、可审计、可回放。
4. 同一输入图和同一规则集，应得到稳定结果。
5. 同一源码重复执行，不应重复删除同一目标。

## 10. 演进方向

当前不直接全量采用完整 DDD。

更现实的演进路径是：

1. 先用模块化单体收清边界。
2. 先把内部语义对象和阶段结果对象收稳定。
3. 等 `DeletionDecision` 的复杂度继续上升，再局部演进成真正的 Domain。

优先升级对象仍然是：

- `DeletionDecision`

典型升级信号包括：

1. 标记来源链越来越复杂。
2. 传播规则越来越多。
3. 决策冲突越来越多。
4. “同一目标唯一最终动作”变成强一致性核心约束。
