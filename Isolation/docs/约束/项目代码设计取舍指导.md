# 项目代码设计取舍指导

## 1. 目的

本文约束 `Isolation` 仓库在**项目级 / 模块级 / 分层级**上的设计取舍，重点回答“一个能力应该落在哪个项目、哪个层、哪个边界”。

本文是 `docs/约束/DDD架构取舍指导.md` 的项目级展开，不替代它。

## 2. 设计依据

综合以下经验：

- Clean Architecture 的 Use Cases 层应保持薄而可测试。
- modular-monolith-with-ddd 的 ADR 强调富领域模型、战术模式和松耦合写模型。
- ABP 的模块架构强调 `Application.Contracts`、`Application`、`Domain`、`Infrastructure` 的清晰包边界。
- Thoughtworks 强调把架构规则落为 fitness functions。

结合本仓库现状，项目级设计优先解决的是：

- 分层是否继续保持清晰。
- 新能力应该落在现有层还是新增项目。
- `Application` 与 `Logic` 的边界是否继续收敛。
- 约束能否被 `tests/ArchitectureTests/Program.cs` 持续保护。

## 3. 现有项目级事实

当前主线结构：

- `src/Domain`
- `src/Logic`
- `src/Application`
- `src/Infrastructure`
- `tests/ArchitectureTests`

当前仓库已经证明有效的设计选择：

- `src/Application/Application.csproj` 只引用 `Domain` 和 `Logic`。
- `src/Logic/Logic.csproj` 只引用 `Domain`。
- `src/Infrastructure/Infrastructure.csproj` 负责 Roslyn、持久化和技术适配。
- `tests/ArchitectureTests/Program.cs` 已作为架构守门员存在。

因此，默认策略不是“再发明一层”，而是先把现有四层边界做干净。

## 4. 项目级落位规则

### 4.1 什么时候放进 Domain

满足以下任一条件时优先考虑 `Domain`：

- 代码表达的是业务概念、业务状态、业务不变量。
- 代码回答的是“这个领域对象本身如何变化才合法”。
- 代码应作为多个用例共享的核心业务语义。

例子：

- `src/Domain/Decision/RewriteDecision.cs`
- `src/Domain/Propagation/ChangeCandidate.cs`
- `src/Domain/Workspaces/WorkspaceContext.cs`

### 4.2 什么时候放进 Logic

满足以下条件时优先考虑 `Logic`：

- 代码是单阶段、可复用的推导能力。
- 它消费领域对象，但不定义领域不变量。
- 它被多个应用服务复用，或者应当从应用服务中剥离出来。

例子：

- `src/Logic/Workspaces/WorkspaceContextBuilder.cs`
- `src/Logic/Marking/RuleTargetCandidateBuilder.cs`
- `src/Logic/Decision/RewriteDecisionAssessmentBuilder.cs`

补充说明：

- 如果一段推导已经上升为领域合法性解释，继续下沉到 `Domain` 更合适。
- 当前 `RewriteDecisionAssessmentBuilder` 已收敛为事实转交壳，真正的评估解释位于 `Domain.Decision.RewriteDecisionAssessmentPolicy`。

注意：

- `Logic` 不等于“凡是不知道放哪就先丢进去”。
- 如果一个 `Logic` 类同时负责计划、执行、报告、事件主链，就要警惕它是否正在长成新的应用层。

规则相关补充：

- 规则对象语义与合法性校验放 `Domain.Rules`
- 规则目录、默认装配、场景预设、规则工厂优先放 `Logic.Rules`
- `WorkspaceRuleDefaultsBuilder` 这类类应逐步退化为薄适配器，避免同时承担目录、工厂、预设三种职责

### 4.3 什么时候放进 Application

只有在它属于**用例编排**时才放 `Application`：

- 组装请求。
- 组织调用顺序。
- 做 DTO 映射。
- 协调多个 `Domain` / `Logic` 能力形成一次完整用例。

例子：

- `src/Application/Services/WorkspaceContextAppService.cs`
- `src/Application/Services/PropagationAppService.cs`
- `src/Application/Services/DecisionAppService.cs`

特殊警示：

- `src/Application/Services/RewriteWorkflowAppService.cs` 代表当前最接近边界红线的位置。
- 当 `AppService` 开始直接构造规则、评分、候选或阶段性中间对象时，应优先考虑下沉到 `Logic` 或 `Domain Service`。

### 4.4 什么时候放进 Infrastructure

满足以下条件时放 `Infrastructure`：

- 依赖 Roslyn、文件系统、DI、持久化、程序集加载、外部系统。
- 是技术实现替换点。
- 不应该出现在业务主链语义中。

例子：

- `src/Infrastructure/Roslyn`
- `src/Infrastructure/Persistence`
- `src/Infrastructure/Analysis`

## 5. 新项目 / 新文件夹的取舍规则

### 5.1 默认先分文件夹，不新增项目

新增项目只有在以下条件满足至少一项时才考虑：

- 需要独立依赖图。
- 需要独立替换或独立发布。
- 现有项目边界无法表达真实依赖方向。
- 已经出现必须由编译器强制阻断的跨层污染。

否则优先：

- 在现有项目内分子目录。
- 通过命名空间和架构测试收敛。
- 通过抽离 `Logic` 能力或 `Domain` 行为减轻复杂度。

### 5.2 不要为了“更像标准模板”机械拆项目

ABP 的模块架构会拆 `Domain.Shared`、`Application.Contracts` 等包，但本仓库当前不应机械照搬。

当前更合理的演进顺序是：

1. 先减少 `Application.Contracts` 对 `Domain.*` 的直接泄露。
2. 再评估是否真的需要 `Domain.Shared` 一类稳定共享层。
3. 最后才讨论新增项目是否能降低真实复杂度。

## 6. Application.Contracts 的项目级约束

`Application.Contracts` 是项目边界，不是内部直通层。

强制规则：

- 输入输出优先使用契约枚举、契约 DTO、原始边界字段。
- 不要把 `Domain` 聚合、实体、值对象直接暴露为外部契约。
- DTO 内禁止逻辑。
- DTO 只表达这个用例真正需要的输入。

当前仓库的典型问题：

- 工作流相关契约主体已经切换到 `Contract*` 边界类型。
- 当前剩余关注点是边界命名是否继续泄露领域语义，以及应用内部中间模型是否被反推回契约层。

当前仓库的默认收敛策略：

1. 先从工作流和决策相关契约开始纯化。
2. 高频复用但又跨边界的语义，优先映射为契约层自有 enum / record。
3. 把映射成本留在 `Application.Mappers`，不要把它回推到 `AppService`。

## 7. 项目级协作规则

### 7.1 同模块内部不要互调 AppService 代替领域抽象

参考 ABP 经验，同一模块内一个 `AppService` 不应把另一个 `AppService` 当共享逻辑库。

优先顺序：

1. 如果是用例无关的核心规则，放 `Domain` 或 `Domain Service`。
2. 如果是单阶段可复用能力，放 `Logic`。
3. 如果只是边界映射，放 `Application.Mappers`。

### 7.2 Mapper 是边界转换层，不是逃避建模的垃圾桶

允许：

- `Domain -> DTO`
- `DTO -> Domain`
- 契约层语义与领域层语义的转换

不允许：

- 在 Mapper 中隐藏复杂规则推导。
- 用 Mapper 承担本应进入 `Domain` 或 `Logic` 的判断逻辑。

## 8. 架构测试策略

项目级规则应优先变成自动化检查。

当前建议继续增强：

1. 检查 `Application.Contracts` 是否直接引用禁止的 `Domain.*` 命名空间。
2. 检查 `Application.Services` 是否直接 new 低等级规则对象。
3. 检查 `Logic.Workflow` 下的类数量、职责聚集趋势或命名模式是否越界。
4. 检查 `IRewriteWorkflowArtifactAssembler` 是否保持单一公开装配入口。
5. 检查决策评估解释是否继续留在 `Domain.Decision`。

## 9. 默认优化顺序

当要优化项目级结构时，按以下顺序行动：

1. 清理 `Application.Contracts` 泄露。
2. 继续纯化 `RewriteWorkflowAppService` 一类编排服务。
3. 拆细 `Logic.Workflow` 的职责热点。
4. 只在确有独立依赖图需求时新增项目。
5. 把新共识写回 `tests/ArchitectureTests/Program.cs`。

## 10. 一句话决策法

当你犹豫一个能力放哪时，用下面的问题判断：

- 它是业务不变量吗。是，进 `Domain`。
- 它是可复用阶段能力吗。是，优先进 `Logic`。
- 它是一次用例的顺序编排吗。是，进 `Application`。
- 它依赖 Roslyn / IO / 持久化 / 外部实现吗。是，进 `Infrastructure`。

如果四个问题都答不清，先不要加新层，先把职责说清楚。

## 12. ABP 模块组织方式对 Isolation 的适配结论（2026-04）

本节基于对 ABP 文档与真实模块代码的交叉阅读，目标是提炼“可借鉴的边界纪律”，不是复制整套产品化模块矩阵。

已核对的资料包括：

- `abp-dev/docs/en/framework/architecture/best-practices/module-architecture.md`
- `abp-dev/docs/en/framework/architecture/best-practices/application-services.md`
- `abp-dev/docs/en/framework/architecture/best-practices/domain-services.md`
- `abp-dev/docs/en/framework/architecture/best-practices/data-transfer-objects.md`
- `abp-dev/docs/en/framework/fundamentals/modularity/basics.md`
- `abp-dev/docs/en/framework/architecture/domain-driven-design/application-layer.md`
- `abp-dev/docs/en/framework/architecture/domain-driven-design/domain-layer.md`
- `abp-dev/modules/identity/src`
- `abp-dev/modules/permission-management/src`
- `abp-dev/modules/tenant-management/src`
- `abp-dev/modules/feature-management/src`

### 12.1 已确认的 ABP 真实组织方式

ABP 的成熟模块通常采用一组稳定项目边界：

- `Domain.Shared`
- `Domain`
- `Application.Contracts`
- `Application`
- `HttpApi`
- `HttpApi.Client`
- `EntityFrameworkCore` / `MongoDB`
- 视场景再加 `Web` / `Blazor`

从实际 `csproj` 关系看，核心依赖方向很稳定：

- `Application.Contracts -> Domain.Shared`
- `Application -> Application.Contracts + Domain`
- `Domain -> Domain.Shared`
- 技术实现项目依赖 `Domain` 或边界契约

从实际代码看：

- `TenantAppService` 负责用例顺序、仓储调用、DTO 映射、事件触发
- `TenantManager` 负责创建租户、改名、校验等领域动作
- `PermissionAppService`、`FeatureAppService` 虽然也有流程代码，但共享能力依然围绕 manager / definition manager / repository 组织，而不是在同模块内部互调 AppService
- `Domain.Shared` 更常承载错误码、常量、定义记录、资源名、验证器工厂这类稳定共享语义

### 12.2 适合 Isolation 借鉴的部分

1. **先守依赖方向，再谈模块花样**
   - ABP 的价值首先体现在依赖纪律。
   - 对 Isolation 来说，这直接对应现有四层：
     - `Domain`
     - `Logic`
     - `Application`
     - `Infrastructure`

2. **Application Service 保持薄壳**
   - AppService 负责请求入口、顺序编排、DTO 边界、少量授权/策略判断。
   - 同模块内部共享能力优先下沉到 `Domain`、`Domain Service`、`Logic`。

3. **共享能力优先下沉到稳定语义层**
   - ABP 把“业务动作”压到 manager / domain service。
   - Isolation 对应做法是：
     - 领域不变量和合法性进 `Domain`
     - 可复用阶段能力、规则目录、preset、factory 进 `Logic`

4. **契约层只表达边界输入输出**
   - ABP 的 DTO 很细，职责单一。
   - Isolation 应继续让 `Application` 边界模型只表达用例输入输出，不回流内部中间模型。

5. **用自动化测试固化边界**
   - ABP 的项目切分把边界变成编译期事实。
   - Isolation 当前更适合继续用 `tests/ArchitectureTests/Program.cs` 把边界做成 fitness functions。

### 12.3 当前不应硬搬的部分

当前仓库的主矛盾是边界收口与职责回归，最优动作是把现有四层做干净。

现阶段不建议直接新增：

- `Domain.Shared` 独立项目
- `Application.Contracts` 独立项目
- `HttpApi` / `HttpApi.Client`
- 多数据库实现项目
- UI / installer 配套项目

原因：

1. 当前仓库还在收口 `Application` 与 `Logic` 的职责边界。
2. `Logic.Workflow` 仍是热点目录，主要复杂度来自用例主链，而不是交付面矩阵。
3. 真实收益当前来自“减少层内串味”，还不是“新增更多编译单元”。

### 12.4 对 Isolation 的直接指导

#### A. 继续保持四层主干，不急着新增项目

当前优先级：

1. 继续让 `Application` 退出稳定规则构造
2. 继续把阶段性复用能力收口到 `Logic`
3. 继续把决策合法性与不变量压回 `Domain`
4. 用架构测试守住上述边界

#### B. 把 `src/Application/Abstractions` 视作“轻量边界面”

当前仓库已经存在 `src/Application/Abstractions`。

这更像当前阶段的轻量版 `Application.Contracts`：

- 允许它承载应用层边界接口
- 继续避免把领域模型直接扩散成外部契约
- 等真正出现独立发布、远程调用、独立依赖图需求，再评估独立项目化

#### C. 把 `Logic` 当作当前仓库的“模块内复用层”

ABP 在模块内通过 Domain / Manager / DefinitionManager 组织复用。

Isolation 当前最适合把下面这类能力继续压进 `Logic`：

- 规则目录
- 规则 preset
- 规则 factory
- workflow stage
- assembler
- candidate builder

这样可以让 `RewriteWorkflowAppService`、`WorkspaceContextAppService`、`PropagationAppService` 继续变薄。

#### D. 只有在出现真实编译边界诉求时，才考虑 ABP 式项目拆分

满足以下任一条件，再评估新增项目：

- 需要独立发布 SDK / Contracts
- 需要真正的远程 API 边界
- 需要多交付面复用同一契约
- 需要编译器强制阻断当前命名空间级约束

### 12.5 当前最值钱的落地动作

结合 ABP 经验与仓库现状，当前优先顺序保持为：

1. 继续清理 `Application.Services` 中残留的稳定规则构造与局部规则字符串
2. 继续把 `Logic.Workflow` 主链拆成更稳定的阶段能力组合
3. 继续把决策不变量和状态合法性压回 `Domain` / `Domain Service`
4. 把这些边界补进 `tests/ArchitectureTests/Program.cs`
5. 等四层边界稳定后，再评估是否需要独立 `Application.Contracts` 或 `Domain.Shared`

### 12.6 一句话执行口径

对 Isolation 学 ABP，优先学边界纪律、薄应用层、稳定共享能力落位和自动化守边界，再按真实复杂度决定是否拆更多项目。

### 12.11 批次 A 已落地的 Application 层内垂直切片事实

截至当前代码状态，`RewriteWorkflowAppService` 已继续收口为更薄的 facade，
并把用例编排拆到 `Application` 层内部目录：

- `src/Application/Services/RewriteWorkflow/RewriteWorkflowUseCase.cs`
- `src/Application/Services/RewriteWorkflow/RewriteWorkflowStageInputFactory.cs`
- `src/Application/Services/RewriteWorkflow/RewriteWorkflowRunState.cs`
- `src/Application/Services/RewriteWorkflow/RewriteWorkflowRunDtoAssembler.cs`

这组事实的项目级意义：
