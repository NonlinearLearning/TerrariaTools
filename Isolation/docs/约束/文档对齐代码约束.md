# 文档对齐代码约束

## 1. 目的

本文约束 `Isolation` 仓库中“文档如何反向约束代码实现”。

这里的“文档对齐代码”指：

- 文档中的有效规则、ADR、架构约束、流程约束，会反向决定代码应该怎样实现。
- 代码与文档冲突时，必须显式处理冲突，不能长期并存。
- 文档不能只做说明书，文档要进入实现决策和回归守护。

本文是仓库级执行约束。

## 3. 外部资料给出的可执行结论

### 3.1 文档是交付的一部分

Write the Docs 和 GitLab 都把文档放进开发流程本体，而不是发布后补材料。

对本仓库的直接结论：

- 文档里的有效约束，属于实现输入。
- 编码前要先识别适用文档。

### 3.2 文档要和真实代码对象建立可追踪关系

Microsoft 明确推荐通过源码引用、命名片段、真实文件关联来降低漂移。

对本仓库的直接结论：

- 文档规则应尽量落到真实目录、真实类型、真实测试。
- 文档只写抽象口号，约束力会很弱。

### 3.3 自动生成文档不能替代规则判断

Microsoft 风格指南强调自动生成内容仍需人工审查。

对本仓库的直接结论：

- 代码注释、生成文档、AI 草稿都可以辅助实现。
- 最终约束判断仍由仓库规则文档和验证结果负责。

### 3.4 约束需要进入流水线

Thoughtworks 明确强调 fitness functions 把架构标准表达成代码，并放进 build / pipeline 监控漂移。

对本仓库的直接结论：

- 文档一旦稳定，就应继续转成测试、脚本或架构检查。
- 只停留在 Markdown 的规则，治理强度有限。

## 4. 本仓库哪些文档对代码有约束力

默认具有实现约束力的文档：

1. `AGENTS.md`
2. `docs/约束/*.md`
3. `.omx/plans/*.md` 中已经进入执行态、并被本轮任务引用的计划
4. 已有测试和架构测试中固化的规则

默认解释优先级：

1. `AGENTS.md`
2. `docs/约束/` 中更具体的文档
3. 当前任务引用的专项计划
4. 一般说明性文档

## 5. 代码实现时的强制规则

### 5.1 开工前必须识别适用文档

涉及以下主题时，必须先读相应文档：

- DDD / 分层 / 架构边界
- 项目落位 / 模块落位
- 局部代码设计
- 测试约束
- 文档同步约束

### 5.2 代码与文档冲突时必须二选一闭环

发现冲突后，只允许两种闭环路径：

1. **改代码对齐文档**
2. **改文档对齐代码，并补理由与验证**

不允许出现：

- 代码已经改了，文档以后再说
- 文档明知失效，先继续沿用
- 汇报里回避冲突

### 5.3 规则文档要尽量转成可执行守护

优先落地为：

1. `tests/ArchitectureTests/Program.cs`
2. 回归测试
3. 构建脚本检查
4. 文档引用检查

### 5.4 计划文档的验收标准要回流到代码验证

若计划文档中出现以下内容，代码落地时要同步映射：

- 状态迁移约束
- 分层依赖约束
- API / DTO 约束
- 关键错误路径
- 关键回归场景

映射对象优先级：

1. 自动化测试
2. ArchitectureTests
3. 构建日志
4. 手工验证记录

### 5.5 文档中的“已知问题”必须持续清理

如果代码已修复文档中的已知问题：

- 同一轮任务要同步清理或降级该问题描述。

如果代码尚未修复：

- 文档要准确反映当前缺口，不得把已知缺口写成现状能力。

## 6. 文档驱动代码的判断方法

### 6.1 先判断文档类型

- **约束文档**：直接影响实现
- **计划文档**：影响当前任务的验收与步骤
- **说明文档**：提供背景，约束力较弱

### 6.2 再判断文档状态

优先采用：

- 当前仓库仍在引用的规则
- 已被测试或计划反复印证的规则
- 与当前目录、当前类型直接相关的规则

### 6.3 最后判断动作

- 文档有效且代码不符：改代码
- 文档过时且代码已有更强证据：改文档
- 两边都不完整：先补专项计划，再一起收敛

## 7. 本仓库推荐的自动化方向

### 7.1 对架构文档

继续沉到 `tests/ArchitectureTests/Program.cs`：

- 层间依赖
- 应用层编排边界
- 公共契约泄露
- 领域解释落位

### 7.2 对计划文档

继续沉到回归测试：

- 生命周期不变量
- 失败路径
- 默认策略来源
- 工作流阶段边界

### 7.3 对文档同步规则

继续沉到流程检查：

- 代码改动是否伴随文档影响评估
- 关键约束文档是否同步更新
- 文档里的关键路径是否仍存在

当前已明确的首个自动化落点：

- `tests/ArchitectureTests/Program.cs` 持续校验 `docs/plans/2026-04-21-DDD证据清单-热点盘点-批次计划.md` 的必备章节与关键路径锚点。
- 后续新增的证据计划文档，默认沿用同样模式继续进入自动化守护。

## 8. Done Definition

一次“文档对齐代码”完成，至少满足：

1. 已识别适用文档。
2. 代码实现符合文档约束，或文档已同步修正。
3. 冲突已闭环。
4. 关键规则已尽量回流到自动化检查。
5. 最终汇报已说明本轮依据了哪些文档、更新了哪些文档、补了哪些验证。
