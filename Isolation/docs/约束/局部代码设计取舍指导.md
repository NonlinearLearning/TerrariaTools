# 局部代码设计取舍指导

## 1. 目的

本文约束 `Isolation` 仓库在**类、方法、DTO、仓储、值对象、局部抽象**层面的设计取舍，重点解决“这段代码该怎么写才不把局部复杂度再次推回架构边界”。

## 2. 设计依据

综合以下经验：

- Fowler 反对贫血模型，强调行为应进入领域模型。
- ABP 强调 DTO 无逻辑、仓储按聚合根定义、应用服务只处理用例。
- modular-monolith-with-ddd ADR 强调聚合边界、领域事件、领域服务。
- 本仓库已有 `docs/约束/Google-CSharp-Style-Guide-约束.md` 负责代码风格，本文只约束设计取舍。

## 3. 类级设计规则

### 3.1 聚合根和实体要持有行为，不要退化成属性容器

优先写法：

- 用公开业务方法改变状态。
- 用方法保护不变量。
- 在行为发生时登记领域事件。

例子：

- `src/Domain/Decision/RewriteDecision.cs`
- `src/Domain/Propagation/ChangeCandidate.cs`

禁止倾向：

- 公开大量 setter，然后让上层自由拼状态。
- 把业务规则留给 `AppService` 或 Mapper 判断。

### 3.2 值对象优先承载重复出现的原语组合

当以下情况出现时优先建值对象，而不是继续传裸参数：

- 一组字段总是一起出现。
- 它们有自己的合法性或比较语义。
- 它们在多个方法间反复成组传递。

但不要为一次性包装制造值对象噪音。值对象必须带来语义增量。

### 3.3 领域服务只处理“不自然属于单一聚合”的核心规则

适合抽为 `Domain Service`：

- 规则横跨多个聚合。
- 规则依赖仓储或外部领域抽象。
- 规则属于核心业务，而不是单次用例流程。

不适合抽为 `Domain Service`：

- 只是为了让类更少。
- 只是因为应用服务写胖了。
- 其实更应该抽到 `Logic` 的中间推导能力。

## 4. 方法级设计规则

### 4.1 一个方法只维持一个抽象层级

如果一个方法同时做了以下多件事，就应拆分：

- 编排顺序。
- 构造规则对象。
- 推导中间结论。
- 映射 DTO。
- 记录技术细节。

当前仓库的典型风险点：

- `src/Application/Services/RewriteWorkflowAppService.cs`

这类方法一旦同时出现 orchestration、评估组装、规则构造、DTO 回填，说明抽象层级已经混杂。

当前进展：

- `RewriteWorkflowAppService` 已去掉同模块 `AppService` 串调。
- 工作流阶段输入已收敛为单一 `RewriteWorkflowAssemblyInput`。
- 决策评估解释已进入 `Domain.Decision`，应用层当前主要承担编排和边界回填。

### 4.2 AppService 方法不直接拼复用型规则

局部规则：

- 如果一段构造逻辑可被多个用例重用，抽到 `Logic`。
- 如果一段判断逻辑是在定义业务合法性，抽到 `Domain`。
- 如果只是边界转换，放 Mapper。

不要在 `AppService` 里长期保留：

- 临时 `RuleSet` / `EnabledRule` 构造。
- 风险评分拼装。
- 传播或候选评估算法。

### 4.3 提取方法优先于提取新类型，前提是职责仍清晰

当局部复杂度上升时，优先顺序如下：

1. 先把同层级逻辑提取为私有或受保护方法。
2. 如果该逻辑可复用，再提升为 `Logic` 或 `Domain` 类型。
3. 只有当存在稳定边界时，才增加新接口或新抽象。

不要一遇到 30 行方法就新增三个接口和两个 manager。

## 5. DTO 设计规则

### 5.1 DTO 只表达边界，不表达业务决策

强制规则：

- DTO 不写逻辑。
- DTO 不持有隐式推导规则。
- 输入 DTO 只包含该用例实际需要的字段。

坏味道：

- 一个输入 DTO 带大量“也许以后会用”的字段。
- DTO 直接暴露领域对象集合。
- DTO 为了省事直接承载领域层枚举和值对象，导致边界绑死。

### 5.2 输入 DTO 不共享，输出 DTO 控制层级

结合 ABP 经验：

- 输入 DTO 不要在多个方法间共用。
- 输出 DTO 可以按 basic / detailed 的思路控制粒度。
- 如果一个输出 DTO 只是领域对象镜像，说明契约层没有真正建立边界。

## 6. 仓储设计规则

### 6.1 仓储按聚合根定义

强制规则：

- 仓储接口定义在 `Domain`。
- 一个聚合根一个仓储接口。
- 上层使用具体聚合仓储，不使用通用 IQueryable 风格仓储。

理由：

- 这符合 ABP 的仓储经验。
- 可以避免把查询细节和持久化能力泄露到应用层。

### 6.2 不把查询能力上抛成自由组合 API

禁止倾向：

- 仓储向 `Application` 暴露 `IQueryable`。
- 为了少写仓储方法，把查询表达式直接推给上层。

如果一个查询明显是用例特化查询：

- 优先把它定义成明确的方法。
- 再评估是否需要单独的查询模型，而不是上抛通用查询能力。

## 7. 抽象设计规则

### 7.1 新接口必须有明确收益

只有满足以下至少一项时才新增接口：

- 需要隔离可替换实现。
- 需要表达稳定边界。
- 需要隔离外部依赖。
- 测试或装配确实需要抽象。

否则优先保留具体类。

### 7.2 不为了“看起来可扩展”提前抽象

坏味道：

- 只有一个实现，却提前造接口。
- 接口与实现一一镜像，没有边界收益。
- 新抽象只是把复杂度换到更多文件里。

### 7.3 Builder / Assembler / Evaluator 只在职责稳定时存在

这类类型适合存在于 `Logic`，但必须满足：

- 输入输出稳定。
- 逻辑可复用。
- 名称准确表达职责。

不满足时，先用局部提取或受保护方法收敛。

## 8. 删除优先和复用优先

局部设计优化时优先级如下：

1. 先删重复逻辑。
2. 再复用现有领域对象或 `Logic` 能力。
3. 最后才新增 helper、manager、builder、service。

如果一个新类型只是把旧复杂度平移，没有降低认知成本，就不该新增。

## 9. 局部评审清单

提交局部代码前，至少自检以下问题：

- 这个方法是否混了多个抽象层级。
- 这个规则是否本应进入 `Domain` 或 `Logic`。
- 这个 DTO 是否在泄露领域语义。
- 这个接口是否真的有第二个实现或明确替换点。
- 这个新类型是在消化复杂度，还是在转移复杂度。
- 这个修改是否让 `AppService` 更像 orchestration，而不是更像万能层。

## 10. 当前仓库的局部优化优先级

结合现有代码，近期应优先盯住：

1. `RewriteWorkflowAppService` 是否继续维持编排壳，而不重新吸收阶段构造细节。
2. `Application.Contracts` 是否把内部中间模型重新推回边界。
3. `Logic.Workflow` 中 assembler / event builder 是否继续聚团失控。
4. 领域对象中那些仍然更像阶段事实堆积、而不是清晰行为边界的类型。

局部代码设计的目标不是“每个类都很漂亮”，而是避免局部权宜写法持续反噬项目级边界。

## 11. 2026-04-22 外部资料补充后的局部设计结论

本节补充吸收：

- Martin Fowler `DDD Aggregate`
- ABP `Value Objects`
- Microsoft Learn `Field Design`
- `docs/plans/2026-04-22-DDD架构-模块设计-类设计-资料适配与落地批次.md`

### 11.1 应用层热点类优先做“层内垂直切片”，再考虑新项目

对当前仓库最典型的热点：

- `src/Application/Services/RewriteWorkflowAppService.cs`

下一批优先动作是：

1. 保留 facade 入口；
2. 在 `Application` 层内部继续收口 use case / orchestration 切片；
3. 让热点从“一个越来越胖的 service”变成“一个薄 service + 少量清晰编排对象”。

当前已落地的第一批对应类型：

- `RewriteWorkflowUseCase`
- `RewriteWorkflowStageInputFactory`
- `RewriteWorkflowRunState`
- `RewriteWorkflowRunDtoAssembler`

这组类型的局部设计意义是：

- 把主编排、阶段输入构造、DTO 回填拆开；
- 保持 `RewriteWorkflowAppService` 只做 facade；
- 让局部复杂度停留在 `Application` 层内部受控扩散，而不是重新长回单个大类。

### 11.2 多次成组出现的字段优先提炼明确输入模型

对当前仓库最典型的信号：

- `src/Logic/Workflow/RewriteWorkflowAssemblyInput.cs`

当下面两组字段继续扩张时，优先提炼明确输入模型，而不是继续平铺：

1. `TargetName / DocumentPath / MemberSignature / AnchorText`
2. `SourceCode / ClassName / MethodName / ParameterCount`

当前已落地的最小收口是：

- `RewriteWorkflowTargetDescriptor`
- `RewriteWorkflowExecutionDescriptor`

这样做的价值是：

- 减少万能 transport 倾向；
- 让方法签名更聚焦；
- 让后续值对象或稳定输入模型有自然落点。

### 11.3 事件补齐类优先守边界，再做局部提取

对当前仓库最典型的信号：

- `src/Logic/Workflow/Events/WorkflowEventSequenceBuilder.cs`

这类类的默认优化顺序应是：

1. 先确保它只做事件补齐；
2. 再按事件类别做局部提取；
3. 不把执行、报告、证据装配职责吸回同一个类。

### 11.4 局部类设计的当前执行口径

当前最值钱的局部类设计动作是：

1. 抑制 `Application.Services` 热点类继续变胖；
2. 抑制 `Logic.Workflow` 输入模型继续平铺字段；
3. 抑制事件补齐类继续吸收跨职责逻辑；
4. 把这些判断同步回文档与 `ArchitectureTests`。
