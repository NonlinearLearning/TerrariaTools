# 代码对齐文档约束

## 1. 目的

本文约束 `Isolation` 仓库中“代码变更如何同步驱动文档更新”。

这里的“代码对齐文档”指：

- 代码一旦改变外部行为、架构边界、运行方式、约束规则，相关文档必须同步更新。
- 文档更新要跟代码变更一起进入同一次交付闭环。
- 文档不能长期滞后于代码。

本文是仓库级执行约束，不是经验随笔。

## 3. 外部资料给出的可执行结论

### 3.1 文档要走和代码相同的工作流

Write the Docs 明确把 issue、Git、code review、automated tests 放进 docs-as-code 的核心流程。

对本仓库的直接结论：

- 文档更新要进入版本控制。
- 文档更新要经过评审。
- 文档更新要有自动化检查入口。

### 3.2 功能代码和相关文档应同批交付

GitLab 文档工作流明确建议把文档放进同一个 MR，并允许团队把它提升为 definition of done。

对本仓库的直接结论：

- 影响用户、开发者、架构判断的代码变更，默认必须在同一提交链路补文档。
- 可以拆成单独文档提交的情况，要能说明拆分原因，并在同一任务链路闭环。

### 3.3 文档中的代码应尽量引用真实源码

Microsoft Learn 明确推荐通过源码文件引用代码片段，并优先使用 **named snippets**，因为行号脆弱，源码引用更容易保持同步。

对本仓库的直接结论：

- 文档中的代码示例优先引用真实代码事实。
- 当文档需要贴代码时，优先贴稳定片段，避免大段手写伪同步样例。
- 能用测试、真实类型、真实文件路径表达的内容，优先绑定真实对象。

### 3.4 自动生成的注释和参考文档仍要人工审查

Microsoft 风格指南明确要求：从源码自动生成的参考文档和注释，也要审查质量与适用性，并去掉内部实现细节。

对本仓库的直接结论：

- “代码里有注释”不等于“文档已完成”。
- 自动生成内容可以作为初稿，发布前仍要人工检查术语、边界、受众和内部细节泄露。

### 3.5 公共 API / 公共契约必须有完整注释覆盖

Google 开发者文档明确要求 API 参考通常由源码注释生成，并覆盖所有公开类、方法、参数、返回值和异常。

对本仓库的直接结论：

- 对外契约、公共 DTO、公共接口、公共领域类型，注释覆盖率要高于普通内部实现。
- 弃用项必须写迁移方向或替代方案。

### 3.6 文档同步要持续验证

InfoQ 的 Continuous Documentation 和 Thoughtworks 的 fitness functions 都强调：同步关系需要持续验证，最好进入流水线。

对本仓库的直接结论：

- 文档同步不能只靠作者自觉。
- 重要边界要转成自动检查、回归测试或架构测试。

## 4. 本仓库的代码对齐文档判断规则

### 4.1 哪些代码变更必须同步改文档

满足以下任一条件，就必须同步更新文档：

1. 改变了 `Domain / Logic / Application / Infrastructure` 的边界判断。
2. 改变了公共契约、DTO、输入输出语义、错误语义。
3. 改变了关键聚合不变量、状态迁移规则、默认策略来源。
4. 改变了执行流程、阶段拆分、事件链路、运行步骤。
5. 改变了测试约束、架构约束、交付约束、验收口径。
6. 让现有文档中的文件路径、类型名、方法名、流程描述失效。

### 4.2 文档更新的默认落点

按变更性质落到最接近的文档：

- 架构/分层/DDD 判断：`docs/约束/DDD架构取舍指导.md`
- 项目/模块/目录落位：`docs/约束/项目代码设计取舍指导.md`
- 局部代码风格/方法设计：`docs/约束/局部代码设计取舍指导.md`
- 测试与回归规则：`docs/约束/测试代码与xUnit约束.md`
- 本次新增的代码-文档同步约束：本文

如果现有文档都承载不下，再新增文档。

### 4.3 文档更新的最小要求

代码变更触发文档更新时，至少要同步以下四类信息：

1. **事实变更**
   - 哪个类型、文件、流程、规则变了。
2. **判断变更**
   - 为什么这样改。
3. **边界变更**
   - 影响到哪一层、哪份文档、哪类调用者。
4. **验证变更**
   - 用什么测试、架构检查或手工验证证明文档仍然成立。

## 5. 强制规则

### 5.1 同一任务链路必须包含文档影响评估

每次非纯重命名代码变更，都要显式回答：

- 本次代码是否影响现有文档。
- 影响哪一份文档。
- 文档已更新，或说明本次不需要改文档的理由。

### 5.2 影响边界语义的代码变更，默认同批改文档

以下类型默认同批更新：

- 新增或改变约束规则
- 新增或改变状态机/生命周期
- 新增或改变公共契约
- 新增或改变架构边界

### 5.3 文档中的代码事实优先绑定真实代码

允许：

- 真实文件路径
- 真实类型名
- 真实方法名
- 真实测试名
- 真实日志文件名

优先级：

1. 真实代码
2. 真实测试
3. 精简后的真实片段
4. 伪代码

### 5.4 文档不能保留已失效的硬编码事实

以下内容一旦失效，必须同步更新或删除：

- 文件路径
- 类型名
- 方法名
- 阶段顺序
- 默认值来源
- 已知问题列表
- 验收标准

### 5.5 重要文档结论要有验证锚点

优先锚点包括：

- `tests/ArchitectureTests/Program.cs`
- 对应分析测试
- 对应工作流测试
- 对应计划文件
- 对应日志文件

## 6. 推荐流程

### 6.1 开发前

1. 找出受影响文档。
2. 标注本次代码会让哪些描述失效。

### 6.2 开发中

1. 代码改动落地。
2. 同步修改受影响文档。
3. 把新约束尽量转成测试或架构检查。

### 6.3 交付前

1. 检查文档中的路径、类型、方法、流程名是否仍有效。
2. 检查文档与测试、计划、日志是否互相印证。
3. 在最终汇报里写明哪些文档已更新。

## 7. 自动化收敛方向

优先把以下事项转成自动化：

1. 关键文档引用的路径是否存在。
2. 关键架构规则是否已有 ArchitectureTests。
3. 文档要求的验收标准是否已有测试名称或日志记录。
4. 公共契约、公共 API 是否有文档注释或契约说明。

## 8. Done Definition

一次“代码对齐文档”完成，至少满足：

1. 代码已完成。
2. 受影响文档已同步更新。
3. 文档中的关键事实与当前代码一致。
4. 验证证据已补齐。
5. 最终汇报已点名更新过的文档文件。
