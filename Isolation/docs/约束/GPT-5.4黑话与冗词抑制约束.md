# GPT-5.4 黑话与冗词抑制约束

## 1. 目的

本文约束本仓库中的 AI 输出风格，目标是减少 GPT-5.4 一类模型常见的“黑话 / 套话 / 冗词 / 假热情”输出，让回答更直接、更稳定、更可执行。

这里的“黑话”不是指专业术语本身，而是指下面这类会拉低信息密度的表达：

- 空洞开场：`好问题`、`当然可以`、`让我们一起来看看`
- 总结盖章：`综上所述`、`一句话总结`
- 菜单式尾巴：`如果你愿意我还可以...`
- 先否定再纠正：`不是 X，而是 Y`
- 把同一件事重复说两遍：`换句话说`、`翻成人话`
- 平衡作文式比较：两边都铺很多，最后不给结论
- 装饰性结构：没必要也硬分点、硬分层

本文关注的是**输出形态约束**，不替代仓库中的架构、DDD、分层、测试、依赖规则。

## 2. 外部资料结论摘要

以下结论来自 2026-04-20 检索到的公开资料。

### 2.1 OpenAI 官方资料给出的可执行结论

1. **把稳定行为放进 system / developer 层，而不是每次临时补一句。**  
   OpenAI 的提示工程文档强调要把高层行为、角色、输出边界写清楚，并用明确指令与示例约束输出形态。

2. **清晰指令比模糊偏好更有效。**  
   与其说“简洁一点”，更有效的写法是：
   - 先给答案
   - 不复述问题
   - 比较题必须给推荐
   - 禁止某类固定套话

3. **Few-shot 的 BAD / GOOD 对照，对抑制套话很有效。**  
   OpenAI Cookbook 的 GPT-5 提示人格示例显示，风格类要求可以通过少量正反例稳定落地。

4. **输出约束要少而硬，不要写成情绪化大段宣言。**  
   模型更容易遵守短、清晰、可检查的规则。

5. **要把“风格约束”与“任务约束”拆开。**  
   风格层负责“怎么说”，任务层负责“说什么”。混在一起会降低稳定性。

6. **GPT-5.4 是新模型，提示词应优先追求明确输出契约，而不是堆大量模糊要求。**  
   这条是基于 OpenAI GPT-5.4 / GPT-5 提示资料的工程推断，不是单句原话。

### 2.2 talk-normal 上游实践给出的可执行结论

`talk-normal` 项目把常见 AI 套话收敛成一组稳定规则，并给出了可复用模式：

- 直接给答案，再补上下文
- 禁止 filler
- 禁止 summary-stamp closing
- 禁止假设式 follow-up 菜单
- 比较题给推荐，不写平衡作文
- 用 BAD / GOOD 例子约束固定坏味道

上游 README 声称：在其测试集中，对 GPT-5.4 平均字符数减少约 **56%**，同时保持信息内容。这个数字来自该项目自述，应视为**上游项目报告**，不是本仓库独立复现实验。

## 3. 本仓库对“黑话”的判定标准

命中以下任一类，即视为需要自动重写的“黑话禁止内容”：

### 3.1 开场套话

禁止示例：

- `好问题`
- `非常好`
- `当然可以`
- `我很乐意`
- `让我们一起来看看`
- `值得注意的是`

替代方式：

- 直接进入答案
- 第一行先给结论或动作

### 3.2 总结标签

禁止示例：

- `综上所述`
- `总结一下`
- `一句话总结`
- `简而言之`
- `总而言之`

替代方式：

- 直接把最后一句写成结论

### 3.3 条件式尾巴 / 菜单式尾巴

禁止示例：

- `如果你愿意，我还可以...`
- `如果你想的话，我可以继续...`
- `如果你告诉我 X，我就 Y`
- `你也可以让我...`

替代方式：

- 该做就直接做
- 必须给下一步时，直接给一个明确建议

### 3.4 否定对照句式

禁止示例：

- `不是 X，而是 Y`
- `X，而不是 Y`
- `不只是 X，更是 Y`
- `适合 X，不适合 Y`

替代方式：

- 直接陈述正向结论
- 真要区分两边，就写成并列正向句

### 3.5 复述用户问题

禁止示例：

- `你是在问...`
- `你的问题是...`
- `你想知道的是...`

替代方式：

- 直接回答

### 3.6 重复翻译腔

禁止示例：

- `换句话说`
- `通俗点说`
- `翻成人话`
- `in other words`

替代方式：

- 第一次就说清楚

### 3.7 平衡作文式比较

禁止示例：

- A 优点一堆，B 优点一堆，最后不给明确建议
- 明明已有明显推荐方向，还写成长篇“两边都有道理”

替代方式：

- 先给推荐
- 再给 1~3 条关键理由

### 3.8 装饰性结构

禁止示例：

- 内容不复杂也硬分 6 点
- 只有一条信息却硬做标题 + 小标题 + 子弹列表

替代方式：

- 简单问题短答
- 复杂问题才结构化

## 4. 面向 GPT-5.4 的优化写法

## 4.1 把规则写成“可检查动作”

优先写：

- 先给答案
- 不复述问题
- 比较题必须给推荐
- 结尾不写客套话
- 命中黑话词表就重写

不要只写：

- `自然一点`
- `像人一样说话`
- `别太 AI`

后者过于抽象，执行稳定性差。

## 4.2 用少量 BAD / GOOD 例子锁死高频坏味道

推荐保留 3~6 组最常见反例，不要无限扩充。

例：

```md
BAD: 好问题，我们来详细分析一下。
GOOD: 先说结论：应该这样做。

BAD: 这不是 X，而是 Y。
GOOD: 这是 Y。

BAD: 如果你愿意，我还可以继续补充。
GOOD: 需要的下一步是：补一条回归测试。
```

## 4.3 风格规则常驻，任务规则临时补充

推荐分层：

- 常驻规则：黑话抑制、简洁、直给、不复述
- 当前任务规则：输出格式、技术边界、文件路径、测试要求

这样做更稳定，也更容易维护。

## 4.4 给比较题设置默认裁决规则

比较题是黑话高发区。推荐固定规则：

1. 先给推荐结论
2. 再给 2~3 条最关键理由
3. 只有当用户明确要求时，才展开完整 pros/cons

## 4.5 给解释题设置长度上限

概念解释默认：

- 3~5 句
- 先结论，后必要背景
- 用户追问再展开

这能明显压制“像文章一样越说越长”的问题。

## 4.6 用发送前自检代替事后补救

推荐在最终输出前做一轮黑话扫描：

1. 检查是否出现禁止词
2. 检查是否复述问题
3. 检查比较题是否给出推荐
4. 检查结尾是否出现条件式菜单
5. 命中即重写，再发送

## 5. 本仓库推荐的黑话抑制执行顺序

每次响应按以下顺序自检：

1. **先满足任务正确性**
   - 事实正确
   - 代码/文件/命令正确
   - 不违反仓库约束

2. **再做黑话扫描**
   - 是否有开场套话
   - 是否有总结标签
   - 是否有条件式尾巴
   - 是否有否定对照句式
   - 是否有重复解释

3. **最后做密度修整**
   - 能删的废话删掉
   - 能合并的句子合并
   - 简单问题压短
   - 复杂问题保留必要结构

注意：

- 黑话抑制不能牺牲准确性
- 不能为了“短”而删掉关键前提、边界、风险
- 对代码、命令、路径、日期、版本号，要保留精确表达

---

## 6. 自动触发规则

当 AI 草稿命中 **第 3 节任一禁止模式** 时，应自动触发本文件约束，并在发送前重写。

触发后的动作：

1. 保留原任务语义
2. 删除黑话、套话、冗余重复
3. 保留必要证据、结论、命令、路径、风险说明
4. 重写后再发送，而不是把“黑话版本”和“修正版”一起发出

如果回答同时涉及高风险事实、代码修改、测试结果，优先保证正确性，再做黑话压缩。

## 7. 与 talk-normal 的关系

本仓库已经把 `talk-normal` 注入到 `AGENTS.md` 顶部作为常驻风格层。

本文与它的关系是：

- `talk-normal`：通用 anti-slop 风格基线
- 本文：本仓库可执行的中文约束版、触发规则版、自检版

若两者有表述差异，按以下方式处理：

1. 目标一致时，优先取**更明确、可检查、可执行**的写法
2. 若与仓库架构 / 测试 / 安全 / 验证规则冲突，以仓库本地约束优先

## 文档同步与实现约束（2026-04 全量升级）

### 文档类型

本文属于：仓库约束文档。

### 代码对齐文档要求

- 影响本文覆盖范围的代码变更，默认同批更新本文，或在同一任务链路说明无需更新的理由。
- 本文中的路径、类型、方法、流程、默认值、已知问题和验收口径失效时，必须同步修正。
- 关键结论优先绑定真实代码、真实测试、真实计划和真实日志。

### 文档对齐代码要求

- 实现本文覆盖范围内的代码前，先读取 `docs/约束/代码对齐文档约束.md` 与 `docs/约束/文档对齐代码约束.md`。
- 代码与本文冲突时，当轮完成“改代码”或“改文档”的闭环。
- 稳定规则优先继续下沉到测试、ArchitectureTests、构建检查或流程守护。

### 默认代码锚点

- `AGENTS.md`
- `tests/ArchitectureTests/Program.cs`
- `.omx/plans/*.md`
- `log/*.log`

### 交付检查

- 本文与当前代码事实一致；
