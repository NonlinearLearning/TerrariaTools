# 命名治理执行准则（2026-04-21）

## 1. 目的

本文给 `Isolation` 仓库提供一套**可执行、可分批落地、可由测试守护**的命名治理规则，重点解决以下问题：

- 类名、文件名、DTO 名、规则对象名、变量名语义含混；
- 同一概念在不同层出现“双轨重复命名”，导致边界意图不清；
- 缩写滥用，导致新读者无法快速判断该类型是否属于领域语义、边界契约还是实现细节；
- 命名治理一上来就跨层大改，破坏 `Domain / Logic / Application / Infrastructure` 边界。

本文不是纯风格说明，而是**执行准则 + 批次计划 + 风险收口规则**。

## 3. 好命名 / 差命名的可执行判定标准

### 3.1 好命名必须同时满足四个条件

1. **先说业务语义，再说技术角色**
   - 好：`RewriteDecisionAssessmentBuilder`
   - 差：`DecisionBuilder2`、`RewriteThingHandler`

2. **读名字就能判断边界归属**
   - `Dto` / `Request` / `Response` / `AppService` 应主要停留在 `Application`
   - 值对象、聚合、领域事件、规则语义应主要停留在 `Domain`
   - Builder / Assembler / Stage / Facade / Preset 等实现型角色应主要停留在 `Logic` 或 `Infrastructure`

3. **一个名字只承载一个主语义，不把两个动词或两套概念硬拼在一起**
   - 好：`CreateWorkspaceContextRequest`
   - 高风险：`CreateUpdateBookDto`、`RuleTargetCandidateBuildInput`

4. **文件名、主类型名、目录语义相互对齐**
   - 接口文件优先与接口同名；
   - 主类型文件优先一文件一主类型；
   - 非兼容性原因不要让文件名、类型名、目录边界各说各话。

### 3.2 差命名的四类坏味道

1. **语义含混**
   - `Helper` / `Manager` / `Thing` / `Data` / `Info` / `Handler2`

2. **双轨重复**
   - 同一概念同时靠前缀、后缀、目录三次表达，结果反而更不清晰；
   - 示例形态：`ContractExposureDto` 这类名字需要检查“`Contract` 是边界前缀，还是业务语义的一部分”。

3. **缩写滥用**
   - 新读者无法从名字直接推断含义；
   - 缩写没有被仓库文档、测试或上下文地图承认，却进入公共类型或跨层契约。

4. **阶段性中间名长期固化**
   - `BuildInput`、`Resolution`、`Artifacts`、`Context`、`Descriptor` 若缺少强语义限定，容易变成“什么都能装”的名字。

## 4. 本仓库的层级命名规则

### 4.1 Domain

Domain 命名优先表达：

- 业务对象：`RuleSet`、`RuleCode`、`ChangeCandidate`
- 业务事实：`ContractExposure`、`ExternalCallerPresence`
- 业务决策：`RewriteDecision`、`PlanConflict`

强制规则：

- Domain 不出现 `Dto`、`Request`、`Response`、`AppService`；
- Domain 名称优先是**业务名词**或**业务动词+名词**，而不是技术过程名；
- 若缩写不是仓库核心术语，则不要进入 Domain 公共类型名。

### 4.2 Application.Contracts

Contracts 命名优先表达：

- 对外输入：`CreateWorkspaceContextRequest`
- 对外输出：`WorkspaceContextDto`
- 边界枚举/共享轻量语义：`ContractRunMode`

强制规则：

- `Dto` 只表示边界承载体，不表示领域对象；
- `Request` / `Response` 用于用例输入输出，不要让一个输入模型跨多个不相干用例复用；
- `Contract*` 前缀仅用于**共享边界语义类型**，不要无节制扩散到所有子目录 DTO；
- 当名字已经含有清晰业务主语时，先评估 `Contract` 前缀是否在重复表达边界。

### 4.3 Logic

Logic 命名优先表达：

- 算法角色：`Builder` / `Assembler` / `Preset` / `Stage` / `Factory`
- 工作流阶段：`RewriteWorkflowDecisionStage`
- 技术推导能力：`ImpactPropagator`

强制规则：

- `Input` / `Result` / `Context` / `Resolution` 必须带**足够强的前缀语义**；
- 若一个类型名只剩下阶段词，没有业务核心词，应重命名；
- 接口文件名必须与接口名对齐。

### 4.4 Infrastructure

Infrastructure 命名优先表达：

- 外部依赖适配：`RoslynCodeIsolationGateway`
- 持久化实现：`InMemoryWorkspaceContextRepository`
- 装配扩展：`ServiceCollectionExtensions`

强制规则：

- 名字里必须能看出其依赖或技术落点；
- 不要在 Infrastructure 名字里伪装成 Domain 概念；
- `Facade` / `Gateway` / `Repository` 只能在其真实职责成立时使用。

## 5. 缩写治理规则

### 5.1 默认规则

默认**不用缩写**。只有同时满足以下条件才允许：

1. 缩写在该领域是稳定公共术语；
2. 仓库已有文档、测试或上下文明确承认该术语；
3. 缩写不会让边界归属变模糊。

### 5.2 当前允许保留的核心缩写

- `Dto`：边界对象后缀，允许；
- `Cpg`：本仓库核心分析领域术语，允许保留，但不要再无节制扩散到无关边界；
- `Cfg` / `Cdg`：仅限分析引擎或对齐 Joern 算法语义的局部实现，不应扩散为公共边界默认词。

### 5.3 当前不建议继续新增的缩写模式

- 无文档支撑的自造缩写；
- 同时出现前缀缩写 + 目录语义 + 后缀缩写的三重叠加；
- 仅为缩短长度而引入的 `Cfg` / `Ctx` / `Info` / `Mgr` 类缩写。

## 6. 命名治理执行计划与任务拆解

### Phase 0：先固化规则与风险盘点

交付：

- 本文档；
- 现有 Naming 测试基线梳理；
- 大影响命名清单与风险等级。

### Phase 1：只做低风险、单文件、单层收口

优先处理：

- 文件名与主类型不一致；
- 明显缩写滥用但仅影响内部实现；
- 明显 `Helper` / `Manager` / `Thing` / `Data` 类型噪音。

本仓库已确认候选：

- `Isolation/src/Logic/Analysis/Engine/Passes/ControlFlow/Dominance/CfgAdapter.cs` → 至少应先与 `ICfgAdapter<TNode>` 对齐文件名；
- 继续扫描 `Logic` / `Infrastructure` 内部接口文件名与主类型名不一致问题。

### Phase 2：处理中风险的“中间模型泛名”

优先处理：

- `*BuildInput`
- `*AssemblyInput`
- `*Resolution`
- `*Artifacts`

原则：

- 先按工作流阶段、对象主语做归类；
- 只有当名字不足以区分“请求 / 评估 / 决策 / 执行结果”时才重命名；
- 重命名前先确认测试、映射器、事件序列、目录边界是否同步可收口。

### Phase 3：处理高影响边界命名

高风险对象：

- `Application.Contracts` 下的 `Contract*` 系列共享边界类型；
- `Cpg` 是否进入不该暴露 CPG 语义的边界 DTO；
- `RuleSet` / `RuleCode` / `CandidateReason` 等规则语义在 Domain 与 Contract 之间的映射词是否存在重复表达。

强制要求：

- 先做影响盘点，再决定是否收口；
- 若影响 `Application.Abstractions`、`Application.Contracts`、测试基线、文档样例或外部调用，必须由 leader 统一裁决批次。

### Phase 4：把稳定规则下沉为自动守护

优先下沉：

- `Isolation/tests/Isolation.AnalysisTests/Naming/NamingGovernanceConventionsTests.cs`
- `Isolation/tests/ArchitectureTests/Program.cs`
- 必要时增加目录/文件/多类型文件 allowlist 守护

原则：

- 每完成一批命名治理，就补一批守护；
- 不让治理依赖“人记得”。

## 7. 当前批次的影响盘点与建议优先级

### P0：立即可做（低风险）

- 文件名与主类型不一致；
- 仅在 `Logic` / `Infrastructure` 内部使用的模糊接口名；
- 无跨层公开引用的局部变量名、私有字段名。

### P1：需要同批测试收口（中风险）

- `BuildInput` / `Resolution` / `Artifacts` 这类中间模型；
- Query / Workflow / Passes 内部角色名；
- Domain 内多类型文件是否继续扩散。

### P2：需要 leader 审批后再动（高风险）

- `Application.Contracts` 根目录 `Contract*` 系列；
- 任何会引发对外契约、映射器、ArchitectureTests、文档样例同步修改的重命名；
- 任何会改变 `Domain` / `Logic` / `Application` 边界感知的词汇收口。
