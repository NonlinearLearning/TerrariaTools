# 架构设计

## 1. 目标

本文按 2026-04-21 当前代码事实，定义 `Isolation` 的主架构、层级职责、关键边界、默认验证口径和文档对齐要求。

本文覆盖的交付目标：

- 让代码、规则、技能、设计文档使用同一套四层语言
- 让 `RewriteWorkflow`、`Analysis.Engine`、规则语义、输出证据都有稳定落位
- 让分层边界、默认策略来源和验证入口具备真实代码锚点

## 3. 分层职责

### 3.1 Domain

Domain 承载稳定业务概念、规则语义、不变量和生命周期边界。

当前重点上下文：

- `Workspaces`
- `Analysis`
- `Decision`
- `Execution`
- `Marking`
- `Propagation`
- `Rewrite`
- `Output`
- `Rules`

关键类型锚点：

- `src/Domain/Workspaces/WorkspaceContext.cs`
- `src/Domain/Analysis/AnalysisCpgSnapshot.cs`
- `src/Domain/Analysis/AnalysisCompositeLayerSnapshot.cs`
- `src/Domain/Decision/RewriteDecision.cs`
- `src/Domain/Execution/RewritePlan.cs`
- `src/Domain/Execution/RewriteResult.cs`
- `src/Domain/Propagation/ChangeCandidate.cs`
- `src/Domain/Output/Verification/VerificationEvidence.cs`
- `src/Domain/Output/Audit/RunReport.cs`
- `src/Domain/Rules/RuleSet.cs`

### 3.2 Logic

Logic 承载低等级、可复用、单阶段或跨用例复用的内部能力。

当前重点能力：

- `Logic.Analysis`：快照装配、输入描述、分析结果组合
- `Logic.Analysis.Engine`：查询、切片、语义、语言、Pass、Layer 管线
- `Logic.Marking`：规则命中后的候选装配与事件序列
- `Logic.Propagation`：传播解析与事件发布
- `Logic.Decision`：决策解释与领域策略协作
- `Logic.Rules`：规则目录、规则描述、启用规则工厂
- `Logic.Workflow`：完整工作流的阶段能力与工件装配
- `Logic.Workspaces`：工作区构建与默认规则输入装配

### 3.3 Application

Application 负责用例编排、DTO 边界、事务协调和外部语义翻译。

当前关键事实：

- `Application` 只依赖 `Domain + Logic`
- `WorkspaceContextAppService` 已通过 `WorkspaceContextBuilder` 构建默认规则输入
- `RewriteWorkflowAppService` 负责 request -> stage orchestration -> dto mapping
- 同模块复用能力优先下沉到 `Logic` 或 `Domain`

### 3.4 Infrastructure

Infrastructure 负责外部技术兑现与运行环境接入。

当前重点实现：

- `Infrastructure.Analysis.AnalysisBackedCpgGateway`
- `Infrastructure.Analysis.Engine.Frontend.RoslynCpgFrontend`
- `Infrastructure.Analysis.Engine.Frontend.RoslynAstToCpgBuilder`
- `Infrastructure.Roslyn.RoslynCodeIsolationGateway`
- `Infrastructure.Persistence.InMemory*Repository`

固定依赖如下：

当前约束落点：

## 5. 关键架构判断

### 5.1 Analysis.Engine 的三段式落位

当前 `Analysis.Engine` 已按职责拆开：

- 图核心和稳定分析事实在 `Domain`
- 语言、Pass、Layer、查询执行和语义执行在 `Logic`
- Roslyn 前端和项目加载在 `Infrastructure`

这条边界决定了：

- 领域语言不被 Roslyn 类型污染
- 分析执行可被多个用例复用
- 前端适配可单独替换或扩展

### 5.2 RewriteWorkflow 的薄入口 + 阶段能力

当前主流程由以下对象共同完成：

- `src/Application/Services/RewriteWorkflowAppService.cs`
- `src/Logic/Workflow/RewriteWorkflowPropagationStage.cs`
- `src/Logic/Workflow/RewriteWorkflowDecisionStage.cs`
- `src/Logic/Workflow/RewriteWorkflowArtifactAssembler.cs`
- `src/Logic/Workflow/RewritePlanCompiler.cs`
- `src/Logic/Workflow/RewritePlanExecutor.cs`
- `src/Logic/Workflow/RunReportAssembler.cs`

当前结论：

- AppService 保持薄入口
- 阶段能力留在 `Logic/Workflow`
- 决策、计划、结果、证据、报告的生命周期留在 `Domain`

### 5.3 默认规则来源

当前默认规则语义已经从应用层硬编码转到工作区构建链：

- `WorkspaceContextAppService`
- `Logic.Workspaces.WorkspaceContextBuilder`
- `Logic.Workspaces.WorkspaceRuleDefaultsBuilder`
- `Logic.Rules.RuleCatalog`
- `Logic.Rules.EnabledRuleFactory`
- `Logic.Rules.WorkspaceDefaultRulePreset`
- `Logic.Rules.RewriteWorkflowRulePreset`
- `Domain.Rules.*`

当前已完成的收敛：

- 保持 `Domain.Rules.*` 承载规则语义对象和合法性
- `Logic.Rules` 已承载显式规则目录、规则描述、规则工厂
- `Logic.Rules` 已承载 workspace 默认规则预设
- `Logic.Rules` 已承载 RewriteWorkflow 规则预设
- `WorkspaceRuleDefaultsBuilder` 已退化为薄适配器
- `WorkspaceContextBuilder` 已改为 preset + 显式输入合并入口
- `RuleTargetCandidateBuilder` 已开始复用 `RuleCatalog + EnabledRuleFactory`
- `RewriteWorkflowMarkingPreparer` 已通过 `RewriteWorkflowRulePreset` 获取规则集名称并校验规则归属
- `RewriteWorkflowAppService` 已通过 `RewriteWorkflowRulePreset` 解析 workflow 规则码并规范化保护规则
- `WorkspaceContext.Create(...)` 已收口为最小安全兜底
- `tests/ArchitectureTests/Program.cs` 已守护目录注册与默认值硬编码退出

下一阶段收敛方向：

- 继续把更多 `Logic.Marking / Logic.Workflow` 主链切到统一规则入口
- 继续扩充 `RewriteWorkflowRulePreset` 覆盖更多稳定 workflow 规则与决策保护规则

目标效果：

- 规则默认值来源唯一
- 新规则注册入口唯一
- 规则场景组合显式可测
- 规则边界可继续下沉到 fitness functions
