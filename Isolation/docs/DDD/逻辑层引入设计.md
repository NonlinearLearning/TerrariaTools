# 逻辑层引入设计

## 1. 目的

本文说明 `Logic` 层在当前仓库中的真实职责，而不是历史设想。

## 2. 当前代码事实

当前 `src/Logic` 已经稳定存在以下模块：

- `Analysis`
- `Decision`
- `Marking`
- `Propagation`
- `Rewrite`
- `Workflow`
- `Workspaces`

## 3. Logic 当前承担的职责

### 3.1 单阶段复用能力

例如：

- `RuleTargetBuilder`
- `RuleTargetCandidateBuilder`
- `ImpactPropagator`
- `RewriteDecisionAssessmentBuilder`
- `RewriteDecisionMaker`
- `WorkspaceContextBuilder`

### 3.2 工作流阶段组合

`Logic.Workflow` 当前已经拆成稳定阶段：

- `RewriteWorkflowMarkingPreparer`
- `RewriteWorkflowPropagationStage`
- `RewriteWorkflowDecisionStage`
- `RewriteWorkflowPlanStage`
- `RewriteWorkflowExecutionStage`
- `RewriteWorkflowEvidenceStage`
- `RewriteWorkflowReportStage`
- `RewriteWorkflowEventStage`

当前已经成形的辅助边界：

- `RewriteWorkflowArtifactAssembler`
- `RewriteWorkflowAssemblyInput`
- `RewriteWorkflowStageInputs`
- `RewriteWorkflowStageResults`
- `WorkflowEventSequenceBuilder`
- `IDomainEventRecorder`
- `InMemoryDomainEventRecorder`

### 3.3 Analysis.Engine 的承载层

当前 `Analysis.Engine` 的大量实现位于 `Logic.Analysis.Engine`，Domain 只保留稳定语义和接口。

## 4. Logic 明确不承担的职责

- 不直接暴露 DTO
- 不直接替代 AppService 做对外入口
- 不长期持有领域不变量本身
- 不反向决定 Infrastructure 技术细节
- 不把传播、决策、计划、执行、证据、报告、事件职责重新揉回单个 mega-workflow 类

## 5. 当前结论

1. Logic 已经从“杂物层”收敛为“可复用阶段能力层”。
2. 当前最成功的模式是：Application 编排，Logic 组合，Domain 承载不变量。
3. 当前最需要继续守住的点是：`Logic.Workflow` 不重新长成第二个 Application。
4. 当前最稳定的治理方式是继续把 `Logic.Workflow` 的角色边界下沉到 `tests/ArchitectureTests/Program.cs`。
