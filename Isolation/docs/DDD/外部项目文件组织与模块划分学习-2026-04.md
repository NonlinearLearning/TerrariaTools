## 3. 这些项目的组织方式

## 3.1 ABP：项目边界最强，项目内目录相对克制

ABP 的成熟模块基本采用稳定项目矩阵：

- `Domain.Shared`
- `Domain`
- `Application.Contracts`
- `Application`
- `HttpApi`
- `HttpApi.Client`
- `EntityFrameworkCore`
- `MongoDB`
- 视场景再加 `Web`、`Blazor`、`Installer`

从真实模块代码看：

- `Application.Contracts` 主要放接口、DTO、输入模型、权限常量、远程服务常量
- `Application` 主要放 AppService、Module、少量 Mapper/Base
- `Domain` 承载实体、Manager、Store、Provider、Validator、Repository Interface、Cache Invalidator、Definition Record
- `Domain.Shared` 承载 `Consts`、`ErrorCodes`、`Eto`、资源、少量共享接口
- `HttpApi` 很薄，控制器基本与 AppService 对齐
- EF / Mongo 项目专注技术实现

从项目内目录看，ABP 更依赖：

- 项目边界
- 类型命名
- 契约清晰度

ABP 在层内会用少量目录，如：

- `Localization`
- `Settings`
- `Integration`

但不会在每一层过度切碎目录。

### 对 Isolation 的启发

最值得吸收的是：

- 强边界依赖方向
- AppService 薄壳化
- Contracts 只表达边界输入输出
- 共享稳定语义单独收口
- 技术实现项目保持纯技术职责

## 3.2 CleanArchitecture：层次清晰，UseCases 采用垂直切片

`CleanArchitecture-main` 的典型分层是：

- `Core`
- `UseCases`
- `Infrastructure`
- `Web`

其中最值得看的点是 `UseCases`：

- `UseCases/Contributors/Create`
- `UseCases/Contributors/Get`
- `UseCases/Contributors/List`
- `UseCases/Contributors/Update`
- `UseCases/Contributors/Delete`

每个用例文件夹通常收口：

- Command / Query
- Handler
- DTO / Result

这是一种“层内按 feature 切片”的做法。

`Core` 继续承载：

- 聚合
- 接口
- 服务

`Web` 承载入口与配置。

### 对 Isolation 的启发

最值得吸收的是：

- `Application` 或 `Logic` 内部可以按用例或阶段切片
- Handler / Command / Query 粒度很利于控制复杂度
- 大层之内允许 feature 目录来降低热点文件夹的堆积

## 3.3 modular-monolith-with-ddd：模块化单体最强调模块自治

该项目的顶层结构很清楚：

- `API`
- `BuildingBlocks`
- `Database`
- `Modules`
- `Tests`

每个模块又几乎采用稳定子矩阵：

- `Application`
- `Domain`
- `Infrastructure`
- `IntegrationEvents`
- `Tests`

例如 `Modules/Meetings` 下，`Application` 继续细分：

- `Configuration`
- `Contracts`
- `MeetingGroups`
- `Meetings`
- `Members`
- 其他业务子域目录

这个项目的核心设计点在于：

- 模块是第一边界
- API 很薄
- 模块间通信依靠 Integration Events
- 每个模块有自己的数据和自己的组合根

### 对 Isolation 的启发

最值得吸收的是：

- 模块级自治意识
- 业务上下文优先于技术层切分
- 模块间公开接口和集成事件要单独建边界
- 模块内继续做 Application / Domain / Infrastructure 细分

## 3.4 DotNetCorePlugins：宿主-契约-插件 是很干净的边界模板

该项目主结构非常克制：

- `src/Plugins`
- `src/Plugins.Mvc`
- `samples/*`

README 和源码都非常强调一件事：

- 宿主应用
- 插件实现
- Contracts 契约项目

这是插件体系里最稳定的三段式。

`PluginLoader` 的源码和 README 说明了几个关键技术点：

- 独立插件目录
- `sharedTypes` 显式共享类型
- 基于 `AssemblyLoadContext` 做隔离
- 支持卸载与热重载
- 支持 contextual reflection

### 对 Isolation 的启发

最值得吸收的是：

- 明确“宿主/契约/实现”的边界
- 需要跨边界传递对象时，先定义共享类型和共享契约
- 动态加载、隔离和可卸载能力应该围绕最小边界接口设计

对于 `Isolation` 这类含 Roslyn、分析、重写的系统，这种边界思路很适合迁移到：

- 抽象接口
- 动态扩展点
- 后续规则引擎 / 插件化执行器

## 3.5 OrchardCore：模块化 UI 与后台能力并存，目录以能力职责组织

OrchardCore 的顶层结构清楚表达了产品形态：

- `OrchardCore`
- `OrchardCore.Modules`
- `OrchardCore.Themes`
- `OrchardCore.Cms.Web`
- `OrchardCore.Mvc.Web`
- `docs`

具体模块目录很有代表性，例如 `OrchardCore.OpenId`：

- `Configuration`
- `Controllers`
- `Deployment`
- `Drivers`
- `Handlers`
- `Migrations`
- `Recipes`
- `Services`
- `Settings`
- `Tasks`
- `ViewModels`
- `Views`
- `wwwroot`

这类目录切法表达的不是 DDD 分层，而是：

- 后台入口
- 配置
- UI 展示
- 部署动作
- 内容迁移
- 模块初始化

### 对 Isolation 的启发

最值得吸收的是：

- 当一个模块同时覆盖 UI、配置、迁移、后台任务时，目录按职责能力切分非常有效
- View / Settings / Migrations / Handlers 这些横切能力适合显式分目录
- 文档、资源、部署配方在复杂产品里应被视为一等资产

这类做法适合未来 `Isolation` 若扩展到：

- 可视化前端
- 模板化工作流
- 持久化配置迁移
- 可导入导出的工程配方

## 3.6 Orleans：运行时与能力包拆得很细，核心在编程模型与能力插件化

Orleans 的 `src` 目录展现的是另一种组织方式：

- `Orleans.Core`
- `Orleans.Core.Abstractions`
- `Orleans.Runtime`
- `Orleans.Server`
- `Orleans.Client`
- `Orleans.Serialization.*`
- `Orleans.Streaming.*`
- `Orleans.Transactions`
- `Orleans.Reminders`
- `Orleans.Persistence.Memory`
- 以及各类 Provider / Clustering / Hosting 包

这种结构的核心点不在业务模块，而在能力包：

- 编程模型核心
- 运行时核心
- 通信与序列化
- 流
- 事务
- Reminder
- 持久化
- 测试宿主

### 对 Isolation 的启发

最值得吸收的是：

- 当系统进入平台化阶段后，按“能力包”切项目比按业务目录切项目更合适
- `Abstractions` 与 `Runtime` 分离很利于演进
- 序列化、流、持久化、事务、运行时宿主等横切能力拆开后，替换成本更低

这类结构更像 `Isolation` 的中远期演进参考，而不是当前阶段的直接模板。

## 4. 跨项目共性

综合这些项目，可以提炼出几条高频共性。

### 4.1 顶层目录先表达系统边界

高频顶层边界包括：

- `src`
- `docs`
- `samples`
- `tests` / `test`
- `build`
- `.github`

复杂项目会进一步明确：

- `modules`
- `framework`
- `templates`
- `assets`
- `deploy`

### 4.2 组织方式通常只有两种主轴

主轴一：按层

- Domain
- Application / UseCases
- Infrastructure
- Web / Api

主轴二：按模块 / feature / capability

- Meetings / Payments / Registrations
- Contributors/Create / Update / List
- Streaming / Serialization / Runtime

成熟项目通常会把两条主轴组合使用。

### 4.3 边界越稳定，命名越直接

高质量仓库很少依赖模糊命名。

更常见的是：

- `TenantAppService`
- `PermissionManager`
- `PluginLoader`
- `FeatureManagementStore`
- `IdentityUserManager`

也就是：

- 类型名直接表达职责
- 目录名直接表达能力或上下文

### 4.4 文档、样例、测试都占正式位置

成熟仓库普遍重视：

- 文档目录
- sample 目录
- 测试项目
- CI 工作流

这意味着“架构决策、用法、验证”是同一交付链上的资产。

## 5. 对 Isolation 最值得吸收的设计与技术

## 5.1 设计层

### A. 继续坚持四层主干

当前 `Isolation` 的主线仍应是：

- `Domain`
- `Logic`
- `Application`
- `Infrastructure`

这是本仓库最适配的现实主干。

### B. 在层内引入更清晰的 feature / stage 切片

可吸收自 `CleanArchitecture.UseCases` 与 `modular-monolith-with-ddd`：

- `Logic.Workflow` 下继续按阶段切片
- `Application.Services` 周边边界模型继续按用例切片
- `Rules`、`Propagation`、`Marking`、`Decision`、`Workspaces` 继续成为稳定上下文目录

### C. 把稳定共享语义继续收口

可吸收自 ABP 的 `Domain.Shared`：

- `Consts`
- `ErrorCodes`
- 稳定协议名
- 共享枚举
- 集成边界记录

当前阶段可以先在现有项目中按目录与命名收口，等真正需要独立依赖图时再拆项目。

### D. 模块内共享逻辑继续下沉

可吸收自 ABP Manager / Store / Provider 思路：

- 规则预设
- 规则目录
- 规则工厂
- workflow stage
- 组合装配器

这些能力继续下沉到 `Logic` 或 `Domain`，会让 `Application` 更薄。

### E. 重要边界尽量转成自动化守护

可吸收自 Thoughtworks fitness functions 与成熟开源仓库的 CI 意识：

- ArchitectureTests
- 回归测试
- 核心目录约束检查
- 文档路径有效性检查

## 5.2 技术层

### A. 插件与动态加载

可吸收自 `DotNetCorePlugins`：

- `AssemblyLoadContext`
- shared contract
- unloadable / hot reload
- contextual reflection

这对后续 `Isolation` 的规则执行器、插件式分析器、外部 provider 很有价值。

### B. 模块化产品能力组织

可吸收自 `OrchardCore`：

- 设置、迁移、部署、任务、视图模型、处理器分目录
- 模块文档、配方、资源目录当作正式代码资产

### C. 能力包拆分

可吸收自 `Orleans`：

- `Abstractions`
- `Runtime`
- `Server`
- `Client`
- `Serialization.*`
- `Streaming.*`

这种能力包拆法适合作为平台化演进参考。

## 6. 对 Isolation 当前阶段的适配判断

## 6.1 当前适合立刻吸收的

1. ABP 式边界纪律
2. CleanArchitecture 式层内切片
3. modular-monolith-with-ddd 的模块自治意识
4. DotNetCorePlugins 的宿主/契约/实现边界设计
5. Thoughtworks 风格的 fitness functions 思路

## 6.2 当前适合保守处理的

1. ABP 的大规模多项目矩阵
2. OrchardCore 这类同时覆盖 UI、主题、CMS 的产品结构
3. Orleans 这种平台级能力包爆炸式拆分

它们都很有价值，但更适合中后期。

## 6.3 当前最值钱的落地动作

1. 继续让 `Application` 退出稳定规则构造
2. 继续把 `Logic.Workflow` 做成阶段能力组合
3. 继续把不变量和状态合法性压回 `Domain`
4. 继续在 `ArchitectureTests` 固化边界
5. 等真实独立依赖图出现，再评估 `Application.Contracts` 或 `Domain.Shared` 独立项目化

这批开源项目共同说明：顶层先定边界，层内再按 feature 或 capability 切片，稳定共享语义单独收口，技术实现独立承载，文档与测试一起进入交付链。
