## 3. 批次划分

### Batch A：目录名与文件名对齐（低风险，可先做）

目标：

- 修复目录小写问题；
- 修复文件名与主类型名不一致问题；
- 不改变公开契约语义。

优先对象：

- `src/Logic/Analysis/Engine/Language/**`
- `src/Logic/Analysis/Engine/Passes/ControlFlow/Dominance/CfgAdapter.cs`

验收：

- Naming 测试中的目录命名断言转绿；
- `git diff` 主要是 rename，不引入跨层引用变更。

### Batch B：Logic 中间模型泛名收口（中风险）

目标：

- 评估并收口 `*BuildInput` / `*AssemblyInput` / `*Resolution` / `*Artifacts`；
- 保证每个名字都能一眼区分“阶段输入”“领域结果”“执行结果”“边界 DTO”。

优先对象：

- `src/Logic/Decision/**`
- `src/Logic/Propagation/**`
- `src/Logic/Workflow/**`

验收：

- `ArchitectureTests` 继续通过；
- 若改名，相关 Mapper、事件组装、AppService 同批更新。

### Batch C：Application.Contracts 高影响命名盘点（高风险，先盘点再动）

目标：

- 区分哪些 `Contract*` 前缀是必要边界语义；
- 找出哪些名字已经重复表达边界或与 Domain 概念冲突。

优先对象：

- `src/Application/Contracts/Contract*.cs`
- `src/Application/Contracts/Decision/**`
- `src/Application/Contracts/Workspaces/**`

强制规则：

- 本批先出盘点表，不先改名；
- 若涉及 `Application.Abstractions`、文档样例、测试数据或对外调用，必须 leader 统一裁决。

### Batch D：稳定规则下沉为守护测试（高价值，跟随前面批次）

目标：

- 把已收口的命名规则写进自动测试；
- 防止小写目录、文件名漂移、多类型热点扩散再次回归。

优先对象：

- `tests/Isolation.AnalysisTests/Naming/NamingGovernanceConventionsTests.cs`
- `tests/ArchitectureTests/Program.cs`

## 4. 团队分工建议

- Leader：裁决 Batch C 的高影响命名边界；
- Executor lane 1：做 Batch A 的目录/文件 rename；
- Executor lane 2：做 Batch B 的 Logic 中间模型盘点与小批收口；
- Verifier：每批完成后复跑 `build + ArchitectureTests + Naming tests`。

1. 先把 Batch A 做完，优先清掉 Naming 测试暴露的小写目录；
2. 目录命名转绿后，再处理 `CfgAdapter.cs` 这类文件名与主类型不一致问题；
3. 最后再进入 `Contract*` 的高影响边界盘点。
