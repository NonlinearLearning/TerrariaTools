## 2. 证据清单

### 2.1 外部权威资料（2026-04-21 检索）

1. Microsoft Learn, `Designing a DDD-oriented microservice`
   - 链接：<https://learn.microsoft.com/en-us/dotnet/architecture/microservices/microservice-ddd-cqrs-patterns/ddd-oriented-microservice>
   - 对 `Isolation` 的直接结论：
     - 先守住 bounded context 与层间依赖方向，再细化层内切片。
     - `Application` 必须保持薄壳，只协调任务，不持有领域规则。
     - `Domain` 不能反向依赖 `Infrastructure`。
2. ABP.IO, `Application Services Best Practices & Conventions`
   - 链接：<https://abp.io/docs/latest/framework/architecture/best-practices/application-services>
   - 对 `Isolation` 的直接结论：
     - 同模块内 `Application Service` 不应互调。
     - 需要复用时优先下沉到领域层或抽出共享能力类。
     - 应用层保持独立，不把 Web / UI 细节带入应用服务。
3. Write the Docs, `Docs as Code`
   - 链接：<https://www.writethedocs.org/guide/docs-as-code/>
   - 对 `Isolation` 的直接结论：
     - 文档必须和代码一起走 issue、Git、code review、automated tests。
     - Markdown 规则如果不进入自动化守护，治理强度不够。
4. GitLab Docs, `Workflows`
   - 链接：<https://docs.gitlab.com/development/documentation/workflow/>
   - 对 `Isolation` 的直接结论：
     - 影响功能、流程、API、已文档化工作流的代码改动，应把文档视为 done definition 的组成部分。
     - 文档最好与代码同一任务链路完成，至少同批闭环。
5. Microsoft Learn, `How to include code in docs`
   - 链接：<https://learn.microsoft.com/en-us/contribute/content/code-in-docs>
   - 对 `Isolation` 的直接结论：
     - 文档应优先引用真实源码路径。
     - 优先使用稳定锚点而不是脆弱行号；对本仓库可转译为“路径、类型、测试名、计划文件名必须可验证”。

## 3. 热点盘点

### 热点 1：`src/Application/Services/RewriteWorkflowAppService.cs`

现状：

- 已明显薄于上一轮，但仍是最靠近边界回退的入口。
- 任何新需求最容易从这里重新长回阶段输入拼装、局部规则解析、跨阶段细节。

治理重点：

- 继续把新增细节留在 `Logic.Workflow`。
- 继续禁止应用层回收规则解析与阶段实现职责。

### 热点 2：`src/Logic/Workflow/*`

现状：

- 阶段接口、assembler、event stage、evidence collectors 已成骨架。
- 但目录热点的真实风险变成“设计共识只留在文档，没有持续回归守护”。

治理重点：

- 每出现一个稳定共识，就尽量落到 `tests/ArchitectureTests/Program.cs`。
- 让 `Logic.Workflow` 的阶段职责边界继续被自动化验证，而不是只靠人工记忆。

### 热点 3：docs-as-code 自动化缺口

现状：

- `docs/约束/*.md` 与 `docs/DDD/*.md` 已写入大量架构共识。
- 但“证据清单 / 热点盘点 / 批次计划 / 关键路径存在性”之前还缺一个明确自动化入口。

治理重点：

- 把关键计划文档的必备章节和关键代码锚点落到 `ArchitectureTests`。
- 让“文档对齐代码 / 代码对齐文档”不只停留在规则声明。
