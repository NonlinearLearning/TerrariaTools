## 2. 证据清单

## 2.1 权威网络资料

### A. Microsoft Learn：Use Tactical DDD to Design Microservices

- URL：`https://learn.microsoft.com/en-us/azure/architecture/microservices/model/tactical-domain-driven-design`
- 当前用途：
  - 支撑 `Domain / Logic / Application / Infrastructure` 的边界判断；
  - 支撑“聚合是事务一致性边界、聚合间通过领域事件协作”的取舍；
  - 支撑应用服务保持编排壳、业务逻辑不回流到应用层。

### B. Microsoft Learn：Domain events: Design and implementation

- URL：`https://learn.microsoft.com/en-us/dotnet/architecture/microservices/microservice-ddd-cqrs-patterns/domain-events-design-implementation`
- 当前用途：
  - 支撑 `Logic.Workflow.Events`、`IDomainEventRecorder`、`WorkflowEventSequenceBuilder` 的设计合理性；
  - 支撑“跨聚合协作优先通过事件而不是直接耦合”的判断；
  - 支撑验证证据与事件链一起进入交付链。

### C. Martin Fowler：Bounded Context

- URL：`https://martinfowler.com/bliki/BoundedContext.html`
- 当前用途：
  - 支撑 `Isolation` 继续按边界拆分 `Analysis / Marking / Propagation / Decision / Workflow / Output`；
  - 支撑“上下文边界先于大一统模型”的结论；
  - 支撑目录、命名空间、文档都要显式表达边界。

### D. Martin Fowler：DDD Aggregate

- URL：`https://martinfowler.com/bliki/DDD_Aggregate.html`
- 当前用途：
  - 支撑富领域模型、聚合根、聚合内一致性保护；
  - 支撑领域对象方法封装状态变化，而不是把合法性解释上浮到 `Application`；
  - 支撑事务不跨聚合边界的建模意识。

### E. Write the Docs：Docs as Code

- URL：`https://www.writethedocs.org/guide/docs-as-code.html`
- 当前用途：
  - 支撑把设计文档、约束文档、计划文档、测试锚点一起纳入交付链；
  - 支撑“文档与代码共用版本控制、评审、自动化测试”的执行口径；
  - 支撑每轮结构调整都必须同步更新文档与自动化守护。

### F. Microsoft Learn：How to include code in docs

- URL：`https://learn.microsoft.com/contribute/content/code-in-docs`
- 当前用途：
  - 支撑 docs-as-code 下“文档中的代码/路径/锚点应可验证”的实践；
  - 支撑避免截图式知识沉淀，优先采用可追踪、可校验的代码与路径锚点；
  - 支撑后续继续把设计文档和真实仓库路径绑定。

## 3. 适用于 Isolation 的热点盘点

## 3.1 热点 A：`src/Application/Services`

当前热点原因：

- 容易重新吸收规则解析、阶段输入构造和中间对象拼装；
- 最容易破坏“应用服务只做编排壳”的约束；
- 最需要依赖 `tests/ArchitectureTests/Program.cs` 做持续守护。

当前重点文件：

- `src/Application/Services/RewriteWorkflowAppService.cs`
- `src/Application/Services/WorkspaceContextAppService.cs`
- `src/Application/Services/PropagationAppService.cs`
- `src/Application/Services/RuleTargetAppService.cs`

## 3.2 热点 B：`src/Logic/Workflow`

当前热点原因：

- 这是工作流主链的阶段协作中心；
- 最容易出现阶段职责串味；
- 需要持续守住 `stage / assembler / compiler / executor / evidence / report / events` 的角色边界。

当前重点目录：

- `src/Logic/Workflow`
- `src/Logic/Workflow/Events`

## 3.3 热点 C：`src/Application/Contracts` 与边界 DTO

当前热点原因：

- 边界命名与类型泄露最容易发生在契约层；
- docs-as-code 要求设计文档、约束文档与真实边界 DTO 一致；
- 架构演进不能只改实现而不回写契约说明。

## 3.4 热点 D：文档与自动化守护链

当前热点原因：

- 设计文档、计划文档、约束文档已经很多，但并非全部被自动化测试覆盖；
- 如果没有 `tests/ArchitectureTests/Program.cs` 的文档锚点守护，设计共识容易再次漂移；
- docs-as-code 的关键不只是“有文档”，而是“文档与代码一起被验证”。

当前关键锚点：

- `docs/约束/项目代码设计取舍指导.md`
- `docs/plans/2026-04-21-Isolation目录整改计划与ArchitectureTests补强清单.md`
- `docs/DDD/外部项目文件组织与模块划分学习-2026-04.md`
- `tests/ArchitectureTests/Program.cs`

## 5. 当前执行口径

1. 先看证据，再做结构判断；
2. 先盘点热点，再决定批次；
3. 任何设计改动都同步更新对应设计文档与必要测试锚点；
4. 始终保持 `Domain / Logic / Application / Infrastructure` 的边界纪律；
5. 高影响结构调整先做影响盘点，再收口到最小可验证改动。
