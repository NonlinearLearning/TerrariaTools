## 2. 权威资料证据

### 2.1 Martin Fowler — Bounded Context

- 来源：<https://martinfowler.com/bliki/BoundedContext.html>
- 复核日期：2026-04-22
- 关键事实：
  - Bounded Context 是 DDD 战略设计核心；
  - 大系统不应强求单一统一模型；
  - 上下文之间需要显式边界与映射关系。
- 对 Isolation 的直接适配：
  - `Analysis / Marking / Propagation / Decision / Workflow / Output` 继续按语义边界表达是正确方向；
  - `Domain -> Logic -> Application -> Infrastructure` 主干内的目录、命名空间、文档都应继续显式表达上下文；
  - 不建议为了“看起来更标准”重新混合这些边界。

### 2.2 Martin Fowler — DDD Aggregate

- 来源：<https://martinfowler.com/bliki/DDD_Aggregate.html>
- 复核日期：2026-04-22
- 关键事实：
  - 聚合是可被当作单元处理的一组领域对象；
  - 外部引用应通过聚合根进入；
  - 事务不应跨聚合边界。
- 对 Isolation 的直接适配：
  - 合法性、不变量、状态变迁解释应继续压在 `Domain` 聚合与领域对象上；
  - `Application` 不应重新吸收这些规则；
  - `WorkflowEventSequenceBuilder` 当前保持“补齐事件序列”而不是回收聚合内部规则，是正确方向。

### 2.3 Microsoft Learn — Designing a DDD-oriented microservice

- 来源：<https://learn.microsoft.com/en-us/dotnet/architecture/microservices/microservice-ddd-cqrs-patterns/ddd-oriented-microservice>
- 复核日期：2026-04-22
- 关键事实：
  - 应用层应保持 thin；
  - 应用层只协调任务，不持有业务规则或领域知识；
  - 领域模型应独立于展示层和应用层。
- 对 Isolation 的直接适配：
  - `RewriteWorkflowAppService` 当前已收口为 facade，这与权威建议一致；
  - `RewriteWorkflowUseCase` / `RewriteWorkflowStageInputFactory` / `RewriteWorkflowRunDtoAssembler` 的层内切片方向是对的；
  - 后续不要把规则拼装、决策合法性、事件主链回推回 `Application.Services`。

### 2.4 ABP — Module Architecture Best Practices & Conventions

- 来源：<https://abp.io/docs/latest/framework/architecture/best-practices/module-architecture>
- 复核日期：2026-04-22
- 关键事实：
  - ABP 推荐清晰层包分离；
  - `Application.Contracts` 与 `Application` 分层明确；
  - HTTP 层通过 application service 接口委派，而不是把业务逻辑放在入口层。
- 对 Isolation 的直接适配：
  - 应优先吸收 **边界纪律**、**薄应用层**、**入口委派**，而不是机械照搬全部项目矩阵；
  - 现阶段继续守住现有四层主干比新增 `Domain.Shared` / `HttpApi` / `HttpApi.Client` 更值钱；
  - `tests/ArchitectureTests/Program.cs` 继续承担 fitness functions，能替代一部分尚未项目化的编译期边界。

### 2.5 ABP — Value Objects

- 来源：<https://abp.io/docs/latest/framework/architecture/domain-driven-design/value-objects>
- 复核日期：2026-04-22
- 关键事实：
  - 值对象没有概念标识；
  - 值对象应优先不可变；
  - 组成值对象的属性应构成一个概念整体。
- 对 Isolation 的直接适配：
  - `RewriteWorkflowAssemblyInput` 中重复成组字段已具备值对象/稳定输入模型信号；
  - 若继续增长，优先提炼：
    - `RewriteWorkflowTargetDescriptor`
    - `RewriteWorkflowExecutionDescriptor`
  - 这应先在 `Logic.Workflow` 内部完成，避免把万能 transport 继续做大。

### 2.6 Write the Docs — Docs as Code

- 来源：<https://www.writethedocs.org/guide/docs-as-code/>
- 复核日期：2026-04-22
- 关键事实：
  - 文档应与代码使用同一套工具链；
  - 文档应进入版本控制、代码评审、自动化测试；
  - 新功能缺文档时可以阻断合并。
- 对 Isolation 的直接适配：
  - 设计文档、计划文档、约束文档、ArchitectureTests 锚点继续同批闭环是对的；
  - `tests/ArchitectureTests/Program.cs` 持续校验 docs-as-code 锚点，符合这一路线；
  - 后续新的 DDD/模块边界共识仍应优先回写文档并被架构测试守护。

### 2.7 Microsoft Learn — How to include code in docs

- 来源：<https://learn.microsoft.com/en-us/contribute/content/code-in-docs>
- 复核日期：2026-04-22
- 关键事实：
  - 第一次引用代码元素时可优先给链接；
  - 同一上下文后续引用使用代码格式；
  - 代码块应尽量绑定真实代码文件并支持自动校验。
- 对 Isolation 的直接适配：
  - 设计文档中的类型名、路径名、测试锚点继续显式绑定真实文件是正确方向；
  - `tests/ArchitectureTests/Program.cs` 中对计划文档、代码路径、测试路径的锚点检查应继续保留。

## 3. 与当前热点代码的对照结论

### 3.1 `src/Application/Services/RewriteWorkflowAppService.cs`

结论：**当前方向正确，应继续保持 facade，而不是回胖。**

原因：

- Microsoft Learn 明确要求 application layer 保持 thin；
- ABP 也强调入口层主要做委派；
- 当前文件已经只负责接收请求、委派 use case、回填 DTO，符合权威资料。

建议：

- 不要把规则预设、阶段输入拼装、事件补齐、领域合法性重新搬回这里；
- 若未来继续增长，优先在 `src/Application/Services/RewriteWorkflow/*` 内部细分，而不是加新项目。

### 3.2 `src/Logic/Workflow/RewriteWorkflowAssemblyInput.cs`

结论：**当前可接受，但已出现“重复原语组合”的演化信号。**

原因：

- ABP 值对象最佳实践要求概念整体优先建模；
- 当前已有两组明显成组语义字段：
  - 目标描述：`TargetName / DocumentPath / MemberSignature / AnchorText`
  - 执行描述：`SourceCode / ClassName / MethodName / ParameterCount`

建议：

- 先不急于大拆；
- 但若字段继续扩张，应优先在 `Logic.Workflow` 内部引入描述型输入模型；
- 不建议继续平铺更多裸字段。

### 3.3 `src/Logic/Workflow/Events/WorkflowEventSequenceBuilder.cs`

结论：**当前应守边界，不应上推职责。**

原因：

- Fowler 聚合与 Microsoft Learn 都支持“规则归领域、编排保持边界”；
- 当前类负责主链事件补齐，没有回收执行、证据、报告本体逻辑，边界仍成立。

建议：

- 若继续膨胀，优先按事件类别做内部私有分组或局部提取；
- 不建议把事件主链重新挤回 `Application`。

### 3.4 `tests/ArchitectureTests/Program.cs`

结论：**当前是正确的 docs-as-code 守护点，应继续扩充而非降级。**

原因：

- Write the Docs 明确支持把文档纳入自动化验证链；
- 当前测试已经把计划文档、热点代码、测试路径纳入锚点检查；
- 这正是适合当前 Isolation 的 fitness functions 路线。

建议：

- 后续新增设计文档时，继续优先补这里的锚点守护；
- 若新增 `RewriteWorkflowAssemblyInput` 的 descriptor 共识，可同步新增针对字段膨胀的结构守护。

## 4. 对 leader 的可执行建议

1. **继续保留四层主干**：`Domain -> Logic -> Application -> Infrastructure`。
2. **继续让 `RewriteWorkflowAppService` 做 facade**，不要回收阶段输入构造与领域规则。
3. **把 `RewriteWorkflowAssemblyInput` 当成下一批重点观察点**；一旦字段继续涨，优先建描述型输入模型。
4. **把 `WorkflowEventSequenceBuilder` 的后续优化限定在“事件补齐内部整理”**，不要改成跨层总控。
5. **继续把设计共识写回 docs + ArchitectureTests**，因为这条路线同时符合 Write the Docs 与当前仓库现状。
6. **不要机械照搬 ABP 的完整项目矩阵**；当前先把现有四层边界守干净更划算。
