## 2. 权威网络资料与适配结论

### 2.1 DDD 与分层边界

- Martin Fowler, `Bounded Context`
  - <https://martinfowler.com/bliki/BoundedContext.html>
  - 适配结论：
    - `Isolation` 继续优先按业务语义守边界；
    - `Analysis / Marking / Propagation / Decision / Workflow / Output` 这种上下文切法继续成立；
    - 目录、命名空间、设计文档应继续显式表达上下文边界。

- Martin Fowler, `DDD Aggregate`
  - <https://martinfowler.com/bliki/DDD_Aggregate.html>
  - 适配结论：
    - 聚合内部继续承载状态变化合法性与不变量；
    - 不把合法性解释重新推回 `Application.Services`；
    - 跨聚合协作优先通过领域事件与阶段结果协作。

- Microsoft Learn, `Design a DDD-oriented microservice`
  - <https://learn.microsoft.com/en-us/dotnet/architecture/microservices/microservice-ddd-cqrs-patterns/ddd-oriented-microservice>
  - 适配结论：
    - `Application` 继续保持用例编排壳；
    - `Domain` 继续保持业务语义与不变量真相来源；
    - `Infrastructure` 继续保持技术接入与替换点。

- ABP, `Domain Driven Design`
  - <https://abp.io/docs/latest/framework/architecture/domain-driven-design>
  - 适配结论：
    - `Domain -> Logic -> Application -> Infrastructure` 的主干继续成立；
    - 当前更值钱的动作是继续做干净边界，而不是急着新增更多项目。

### 2.2 模块设计与目录组织

- ABP, `Module Architecture Best Practices & Conventions`
  - <https://abp.io/docs/latest/framework/architecture/best-practices/module-architecture>
  - 适配结论：
    - 继续学习 ABP 的边界纪律、薄应用层、契约层独立意识；
    - 当前不机械照搬 `Domain.Shared / Application.Contracts / HttpApi / Web` 的完整矩阵；
    - 当前先在现有四层内把目录与职责切干净。

- Microsoft Learn, `Common web application architectures`
  - <https://learn.microsoft.com/en-us/dotnet/architecture/modern-web-apps-azure/common-web-application-architectures>
  - 适配结论：
    - 上层优先依赖稳定契约与抽象；
    - 具体技术接线继续留在组合根与基础设施。

- Thoughtworks Radar, `Spring Modulith`
  - <https://www.thoughtworks.com/radar/languages-and-frameworks/spring-modulith>
  - 适配结论：
    - 目录与包结构更适合直接映射业务边界；
    - 这对 `Isolation` 的 `Application / Logic` 层内切片很有参考价值。

### 2.3 类设计与文档同步

- ABP, `Value Objects`
  - <https://abp.io/docs/latest/framework/architecture/domain-driven-design/value-objects>
  - 适配结论：
    - 当一组字段在多个阶段输入中反复成组出现时，优先提炼语义明确的值对象或输入模型；
    - 不再继续传播裸字段组合。

- Microsoft Learn, `Field Design`
  - <https://learn.microsoft.com/en-us/dotnet/standard/design-guidelines/field>
  - 适配结论：
    - 领域对象与核心输入模型优先保持封装；
    - 显式方法优先于公开可写字段与随意状态拼装。

- Write the Docs, `Docs as Code`
  - <https://www.writethedocs.org/guide/docs-as-code/>
  - 适配结论：
    - 设计文档、计划文档、约束文档和测试锚点继续进入同一交付链；
    - 新的设计判断应尽量回流到 `tests/ArchitectureTests/Program.cs`。

### 3.1 ABP

最值得吸收：

1. 层级依赖纪律；
2. `Application Service` 保持薄壳；
3. 共享能力优先下沉到 `Domain`、`Domain Service`、稳定复用层；
4. `Application.Contracts` 只表达边界输入输出。

### 3.2 CleanArchitecture

最值得吸收：

1. 大层内部继续按 feature / use case 切片；
2. `Command / Query / Handler / Result` 这类层内收口方式适合消化热点；
3. 当前最适合 `Isolation` 的不是新增项目，而是在现有层内做垂直切片。

### 3.3 modular-monolith-with-ddd

最值得吸收：

1. 模块自治意识；
2. 业务上下文优先于技术层切分；
3. 模块间边界应单独表达，而不是把所有协调挤进应用服务。

## 4. 当前仓库热点与适配判断

### 4.1 `src/Application/Services/RewriteWorkflowAppService.cs`

当前事实：

- 文件长度在当前 `Application.Services` 中最高；
- 已经明显比早期版本更薄；
- 当前主要承担：
  - 请求入口；
  - `RewriteWorkflowUseCase` 委派；
  - DTO 回填。
- 已下沉到 `Application.Services.RewriteWorkflow/*` 的职责包括：
  - workflow 主编排；
  - stage input 构造；
  - `RewriteWorkflowAssemblyInput` 构造；
  - 规则码解析与保护规则归一。
- `RewriteWorkflowUseCase` 当前已经稳定暴露出五段应用层编排子职责：
  - run correlation 生成与 `domainEventRecorder.Clear(runCorrelationId)` 清理；
  - `GetWorkspaceAsync(...) / GetRuleTargetAsync(...)` 获取工作区与规则目标；
  - `PublishAnalysisSnapshotAsync(...) / PublishMarkingCandidateAsync(...)` 发布 analysis / marking 事实；
  - propagation / decision 串行编排，并统一通过 `RewriteWorkflowStageInputFactory` 构造阶段输入；
  - `rewriteWorkflowArtifactAssembler.Assemble(...)` 收口最终产物。

适配判断：

- 当前不需要新增项目；
- 下一批最值得做的是在 `Application` 层内进一步做 use case / vertical slice 收口；
- `RewriteWorkflowAppService` 继续保留 facade 角色更稳妥。

### 4.2 `src/Logic/Workflow/RewriteWorkflowAssemblyInput.cs`

当前事实：

- 这个类型已经承担 plan / execution / evidence / report / event 五类输入映射；
- 当前可观察到两组稳定字段簇：
  - `TargetName / DocumentPath / MemberSignature / AnchorText`
  - `SourceCode / ClassName / MethodName / ParameterCount`
- 这两组字段目前仍主要通过 `BuildAssembly(...) -> To*StageInput()` 单向投影给下游阶段，还没有出现足够广的跨类型复用面。

适配判断：

- 当前先保持 `RewriteWorkflowAssemblyInput` 作为总装输入模型；
- 如果字段继续扩张，或同一字段簇开始在 `BuildAssembly(...) / ToPlanStageInput() / ToExecutionStageInput() / ToEventStageInput()` 之外重复出现，再优先提炼两个稳定输入模型：
  - `RewriteWorkflowTargetDescriptor`
  - `RewriteWorkflowExecutionDescriptor`
- 目标是阻止 `RewriteWorkflowAssemblyInput` 长成万能 transport。

### 4.3 `src/Logic/Workflow/Events/WorkflowEventSequenceBuilder.cs`

当前事实：

- 当前已经承担主链事件补齐职责；
- 它仍是 `Logic.Workflow` 下最重的文件之一；
- 当前职责边界仍基本清楚：补齐事件，而不执行 plan / result / evidence 逻辑；
- correlation id 回退逻辑应复用 `RewriteWorkflowCorrelationResolver`，避免在事件链侧重复维护。

适配判断：

- 下一批更适合补守护而不是强拆；
- 如果继续增长，优先按事件类别做内部私有分组或局部提取；
- 不建议把事件补齐职责上推回 `Application`。

### 4.4 `src/Logic/Rules`

当前事实：

- `EnabledRuleFactory`、`RewriteWorkflowRulePreset`、`RuleCatalog` 已形成相对稳定的规则入口；
- 这层已经很接近“模块内稳定复用层”。

适配判断：

- 下一批优先守住规则入口集中；
- 不把稳定规则码解析重新散落回 `Application.Services`。

### 4.5 `tests/ArchitectureTests/Program.cs`

当前事实：

- 已经不只是层间引用检查；
- 已经承担 Workflow 边界、docs-as-code 锚点、计划文档路径校验。

适配判断：

- 继续把“新设计文档入口”和“关键热点共识”补进这里；
- 让设计判断继续变成回归守护。
