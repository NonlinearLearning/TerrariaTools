## 1. 设计目标

这份设计稿只解决一件事：

**把一条规则收成一个统一的规则对象模型，并允许在统一模型中定义标记行为、传播行为、决策行为，以及表达复杂行为的 Behavior 接口。**

当前设计结论：

- `RuleDefinition` 是统一规则模型
- 规则模型中包含：
  - 描述
  - 标记规格定义
  - 传播规格定义
  - 决策规格定义
- 难以纯字段表达的复杂行为拆成：
  - `IMarkBehavior`
  - `IPropagationBehavior`
  - `IDecisionBehavior`
- 阶段引擎读取规则模型并执行
- 规则对象自己不直接驱动整个阶段流程

## 2. 总体结构

```text
RuleDefinition
  ├─ RuleMetadata
  ├─ RuleMarkingSpec
  ├─ RulePropagationSpec
  ├─ RuleDecisionSpec
  ├─ IMarkBehavior?
  ├─ IPropagationBehavior?
  └─ IDecisionBehavior?
```

分工如下：

- `RuleDefinition`：统一承载一条规则的知识
- `Rule...Spec`：声明式表达规则能力边界
- `IMarkBehavior / IPropagationBehavior / IDecisionBehavior`：表达难以纯字段描述的复杂行为
- `...Result`：返回复杂判断结果
- 阶段引擎：真正解释并执行规则

### 3.2 `RuleDefinition`

```csharp
namespace NewJoern.Deletion.Rules;

using Microsoft.CodeAnalysis.CSharp;

public sealed class RuleDefinition
{
  public RuleMetadata Metadata { get; }
  public IReadOnlyCollection<SyntaxKind> PotentialBreakageSyntaxKinds { get; }
  public RuleMarkingSpec Marking { get; }
  public RulePropagationSpec Propagation { get; }
  public RuleDecisionSpec Decision { get; }

  public IMarkBehavior? MarkBehavior { get; }
  public IPropagationBehavior? PropagationBehavior { get; }
  public IDecisionBehavior? DecisionBehavior { get; }

  public RuleDefinition(
    RuleMetadata metadata,
    IReadOnlyCollection<SyntaxKind> potentialBreakageSyntaxKinds,
    RuleMarkingSpec marking,
    RulePropagationSpec propagation,
    RuleDecisionSpec decision,
    IMarkBehavior? markBehavior = null,
    IPropagationBehavior? propagationBehavior = null,
    IDecisionBehavior? decisionBehavior = null)
  {
    Metadata = metadata;
    PotentialBreakageSyntaxKinds = potentialBreakageSyntaxKinds;
    Marking = marking;
    Propagation = propagation;
    Decision = decision;
    MarkBehavior = markBehavior;
    PropagationBehavior = propagationBehavior;
    DecisionBehavior = decisionBehavior;
  }
}
```

说明：

- `RuleDefinition` 是统一规则模型
- 负责承载一条规则的全部静态知识与复杂行为入口
- 自己不直接驱动完整阶段执行
- 它自己不是值对象，因为内部还持有行为对象引用

字段解释：

- `Metadata`
  - 规则身份卡。
  - 用来表达这条规则是谁、叫什么、版本是什么、默认是否启用。
  - 没有它，规则注册、审计、日志、规则清单都会缺统一锚点。
- `PotentialBreakageSyntaxKinds`
  - 规则预声明的语法结构风险种类清单。
  - 用来表达“这条规则一旦命中，可能会破坏哪些 Roslyn `SyntaxKind` 对应的语法节点结构”。
  - 这是一份静态风险边界，不是某次运行后的事实结果。
- `Marking`
  - 标记阶段的静态规格。
  - 用来定义这条规则理论上可以命中什么目标，以及运行前依赖哪些图能力。
- `Propagation`
  - 传播阶段的静态规格。
  - 用来定义这条规则是否允许传播、传播边界在哪里。
- `Decision`
  - 决策阶段的静态规格。
  - 用来定义这条规则在决策时的默认倾向和升级边界。
- `MarkBehavior`
  - 标记阶段的动态复杂行为入口。
  - 用来补足静态字段难以表达的命中判断、结构残缺判断、派生目标判断。
- `PropagationBehavior`
  - 传播阶段的动态复杂行为入口。
  - 用来补足唯一来源传播、跨文件传播、复杂派生传播这类运行时判断。
- `DecisionBehavior`
  - 决策阶段的动态复杂行为入口。
  - 用来补足候选动作是否成立、是否升级、是否跳过这类运行时判断。

### 3.3 `RuleMetadata`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed record RuleMetadata(
  string RuleId,
  string Name,
  string Description,
  string Version,
  bool IsEnabledByDefault);
```

说明：

- 只承载规则元数据
- 不承载阶段行为逻辑
- 适合作为不可变值对象

字段解释：

- `RuleId`
  - 规则唯一标识。
  - 用来做注册、去重、审计引用、日志定位、规则命中统计。
- `Name`
  - 人类可读名称。
  - 主要服务于文档、日志、调试输出，不承担逻辑判断职责。
- `Description`
  - 规则说明文字。
  - 用来解释规则意图、适用场景、设计意图。
- `Version`
  - 规则版本。
  - 用于规则升级、兼容审计和规则包治理。
- `IsEnabledByDefault`
  - 默认启用状态。
  - 用于宿主加载规则时决定默认是否参与运行。

当前结论：

- `RuleMetadata` 不再保留 `RuleCategory`
- 规则分类不进入核心对象模型
- 如果后续确实需要做规则清单筛选、展示分组、规则市场归类，应降级为宿主侧标签字段或元数据字典，而不是重新回到正式领域对象里

### 3.4 贴近 Roslyn 的轻量语法基础模型

这一层当前不再单独建 `*NodeModel` 语法骨架对象。

前期正式模型直接依赖 Roslyn 事实：

1. `SyntaxKind`
2. `TextSpan`
3. `SyntaxAnnotation`
4. 必要时配合 `Code`

这样可以减少重复抽象，避免同时维护一套自定义轻量语法模型和一套 Roslyn 语法事实。

当前结论：

1. 结构种类统一以 Roslyn `SyntaxKind` 为准。
2. 源码定位统一以 Roslyn `TextSpan` 为准。
3. 语法树重写跟踪统一以 `SyntaxAnnotation` 为准。
4. 前期不额外设计 `BlockNodeModel`、`IfNodeModel`、`LoopNodeModel` 一类包装对象。

### 3.5 `PotentialBreakageSyntaxKinds`

```csharp
namespace NewJoern.Deletion.Rules;

using Microsoft.CodeAnalysis.CSharp;

public sealed class RuleDefinition
{
  public IReadOnlyCollection<SyntaxKind> PotentialBreakageSyntaxKinds { get; }
}
```

说明：

- 定义规则级别的潜在语法结构风险种类清单。
- 当前前期阶段先不用 `SyntaxBreakage` 对象壳，而是直接用 `SyntaxKind` 集合占位。
- 它是 syntax-first 模型：主语义是“源码结构会不会坏”，不是“symbol 会不会失效”，也不是“operation 会不会断”。
- 这里的 `SyntaxKind` 指 `Microsoft.CodeAnalysis.CSharp.SyntaxKind`。
- 当前约束只允许放 `SyntaxNode` 对应的 kind，不放 token kind，也不放 trivia kind。

字段解释：

- `PotentialBreakageSyntaxKinds`
  - 规则级静态风险声明。
  - 例如 `SyntaxKind.IfStatement`、`SyntaxKind.InvocationExpression`、`SyntaxKind.MethodDeclaration`、`SyntaxKind.ParameterList`。
  - 它只回答“可能影响哪些 Roslyn 语法种类”，不记录运行期具体节点实例，也不承载破坏原因明细。

### 3.6 `RuleMarkingSpec`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed record RuleMarkingSpec(
  IReadOnlyCollection<DeletionTargetKind> SupportedTargetKinds,
  IReadOnlyCollection<string> RequiredGraphCapabilities);
```

说明：

- 定义规则的标记能力边界
- 说明命中什么粒度、需要哪些图能力
- 当前不把“标记强度”作为正式模型字段，相关判断由 `IMarkBehavior` 负责
- 适合作为不可变值对象

字段解释：

- `SupportedTargetKinds`
  - 这条规则在标记阶段允许命中的删除目标粒度。
  - 例如只允许命中 `Expression`，就不该去碰 `Statement` 或 `Definition`。
  - 这个字段的存在是为了先把规则的静态适用范围框住。
- `RequiredGraphCapabilities`
  - 这条规则成立前需要依赖的图能力。
  - 例如 AST、源码位置、调用图、到达定义。
  - 这个字段的作用是让宿主或阶段引擎知道缺哪些能力时不应运行这条规则。

### 3.7 `RulePropagationSpec`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed record `RulePropagationSpec`(
  bool IsEnabled,
  IReadOnlyCollection<DeletionTargetKind> SourceKinds,
  IReadOnlyCollection<DeletionTargetKind> ResultKinds,
  int MaxPropagationDepth);
```

说明：

- 定义规则是否允许传播
- 定义传播源粒度、结果粒度和传播深度
- 适合作为不可变值对象

字段解释：

- `IsEnabled`
  - 这条规则在传播阶段是否启用。
  - 有些规则只负责原始标记，不应自动扩散，所以需要这个总开关。
- `SourceKinds`
  - 允许作为传播起点的目标粒度集合。
  - 例如某规则只允许从 `Expression` 传播，不允许从 `Definition` 传播。
- `ResultKinds`
  - 允许传播生成的新目标粒度集合。
  - 例如允许传播成 `Statement`，但不允许直接传播成 `Scope`。
- `MaxPropagationDepth`
  - 最大传播深度。
  - 它存在是为了防止传播链无边界扩张，避免过度传播或误删扩大化。

### 3.8 `RuleDecisionSpec`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed record RuleDecisionSpec(
  DecisionActionKind PreferredAction,
  DeletionTargetKind PreferredDeletionKind,
  bool MayBreakParentStructure,
  bool AllowEscalation,
  IReadOnlyCollection<DeletionTargetKind> EscalationTargets);
```

说明：

- 定义规则在决策阶段偏向什么动作
- 说明是否可能破坏结构、是否允许升级
- 适合作为不可变值对象

字段解释：

- `PreferredAction`
  - 规则进入决策阶段时默认偏向的动作。
  - 前期只区分 `Delete` 或 `Skip`。
  - 它存在是为了让决策阶段先知道这条规则的默认意图。
- `PreferredDeletionKind`
  - 规则默认偏向的删除目标粒度。
  - 它的作用是把“偏向删什么大小的东西”显式化。
- `MayBreakParentStructure`
  - 规则如果按首选方式删除，是否可能导致父结构残缺。
  - 这个字段存在是为了提醒后续要关注升级或保守跳过。
- `AllowEscalation`
  - 是否允许从当前粒度升级到更大粒度。
  - 没有这个字段，升级逻辑就会退化成引擎里的硬编码默认行为。
- `EscalationTargets`
  - 允许升级到的目标粒度集合。
  - 这个字段的作用是限定升级边界，防止规则无限扩大删除范围。

## 4. 三个 Behavior 接口

### 4.1 `IMarkBehavior`

```csharp
namespace NewJoern.Deletion.Rules;

public interface IMarkBehavior
{
  MarkMatchResult Evaluate(
    RuleExecutionContext context,
    RuleDefinition rule,
    DeletionTarget target);
}
```

作用：

- 表达难以纯字段建模的复杂标记行为
- 例如参数位置命中、结构残缺发现、派生更大粒度目标

参数解释：

- `context`
  - 当前规则执行上下文。
  - 提供图、配置、分析轮次等运行期事实。
- `rule`
  - 当前正在执行的规则定义。
  - 让 `Behavior` 能读取自己的静态 `Spec`。
- `target`
  - 当前待判断的删除语义目标。
  - 表示这次标记行为正在尝试命中谁。

### 4.2 `IPropagationBehavior`

```csharp
namespace NewJoern.Deletion.Rules;

public interface IPropagationBehavior
{
  PropagationCheckResult Evaluate(
    RuleExecutionContext context,
    RuleDefinition rule,
    Mark sourceMark,
    DeletionTarget sourceTarget);
}
```

作用：

- 表达复杂传播行为
- 例如唯一来源判断、跨文件传播判断、派生新目标

参数解释：

- `context`
  - 当前规则执行上下文。
- `rule`
  - 当前规则定义。
- `sourceMark`
  - 当前传播尝试的来源标记。
  - 表示“哪一条标记要继续扩散”。
- `sourceTarget`
  - 来源标记所依附的删除目标。
  - 表示“这次传播从哪个语义目标出发”。

### 4.3 `IDecisionBehavior`

```csharp
namespace NewJoern.Deletion.Rules;

public interface IDecisionBehavior
{
  DecisionCheckResult Evaluate(
    RuleExecutionContext context,
    RuleDefinition rule,
    DecisionCandidate candidate);

```

作用：

- 表达复杂决策行为
- 例如动作升级、动作拒绝、跳过判断、最终候选动作建议

参数解释：

- `context`
  - 当前规则执行上下文。
- `rule`
  - 当前规则定义。
- `candidate`
  - 当前进入决策阶段的候选聚合对象。
  - 它包含该目标上的 marks、候选动作和当前阶段信息。

## 5. Result 对象

### 5.1 `MarkMatchResult`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed class MarkMatchResult
{
  public bool IsMatch { get; }
  public string Reason { get; }
  public MarkTarget? MarkTarget { get; }
  public bool StructuralBreakDetected { get; }
  public DeletionTarget? DerivedTarget { get; }

  public MarkMatchResult(
    bool isMatch,
    string reason,
    MarkTarget? markTarget,
    bool structuralBreakDetected,
    DeletionTarget? derivedTarget)
  {
    IsMatch = isMatch;
    Reason = reason;
    MarkTarget = markTarget;
    StructuralBreakDetected = structuralBreakDetected;
    DerivedTarget = derivedTarget;
  }
}
```

说明：

- 标记复杂判断结果
- 除了命中与否，还可表达结构破损与派生目标

字段解释：

- `IsMatch`
  - 这次标记行为是否成立。
  - 是标记阶段最基本的布尔结论。
- `Reason`
  - 这次判断的解释文本。
  - 用于日志、调试、审计和后续报告。
- `MarkTarget`
  - 如果命中成立，本次命中的具体视图对象。
  - 它回答“这条 mark 具体打到了哪里”。
- `StructuralBreakDetected`
  - 当前命中后是否已发现父结构可能残缺。
  - 它存在是为了让后续阶段知道是否需要升级或特殊处理。
- `DerivedTarget`
  - 这次标记判断是否已经派生出新的删除目标。
  - 例如从表达式判断中直接派生出父语句目标。

### 5.2 `PropagationCheckResult`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed class PropagationCheckResult
{
  public bool CanPropagate { get; }
  public string Reason { get; }
  public DeletionTarget? DerivedTarget { get; }
  public MarkTarget? DerivedMarkTarget { get; }

  public PropagationCheckResult(
    bool canPropagate,
    string reason,
    DeletionTarget? derivedTarget,
    MarkTarget? derivedMarkTarget)
  {
    CanPropagate = canPropagate;
    Reason = reason;
    DerivedTarget = derivedTarget;
    DerivedMarkTarget = derivedMarkTarget;
  }
}
```

说明：

- 传播复杂判断结果
- 可表达是否传播、传播理由、传播后新目标和新命中视图

字段解释：

- `CanPropagate`
  - 这次传播是否成立。
- `Reason`
  - 为什么能传播，或为什么不能传播。
- `DerivedTarget`
  - 传播阶段是否派生出新的删除目标。
  - 它是传播阶段的重要结果，因为传播往往意味着新目标诞生。
- `DerivedMarkTarget`
  - 如果传播成立，新标记命中的具体视图对象。
  - 它回答“传播后的 mark 具体打到了哪里”。

### 5.3 `DecisionCheckResult`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed class DecisionCheckResult
{
  public bool IsApplicable { get; }
  public string Reason { get; }
  public DecisionActionKind? SuggestedAction { get; }
  public DeletionTarget? DerivedTarget { get; }

  public DecisionCheckResult(
    bool isApplicable,
    string reason,
    DecisionActionKind? suggestedAction,
    DeletionTarget? derivedTarget)
  {
    IsApplicable = isApplicable;
    Reason = reason;
    SuggestedAction = suggestedAction;
    DerivedTarget = derivedTarget;
  }
}
```

说明：

- 决策复杂判断结果
- 可表达建议动作与升级派生目标

字段解释：

- `IsApplicable`
  - 这条规则在当前候选对象上是否适用。
- `Reason`
  - 适用或不适用的解释文本。
- `SuggestedAction`
  - 这条规则对当前候选目标建议的动作。
  - 它存在是为了让决策阶段能聚合多个规则的建议。
- `DerivedTarget`
  - 决策阶段是否因为升级而派生出新的删除目标。
  - 例如表达式目标因结构残缺升级成语句目标。

## 6. 运行上下文

### 6.1 `RuleExecutionContext`

```csharp
namespace NewJoern.Deletion.Rules;

public sealed class RuleExecutionContext
{
  public string AnalysisId { get; }
  public NewJoernGraph Graph { get; }
  public IReadOnlyDictionary<string, string> Options { get; }

  public RuleExecutionContext(
    string analysisId,
    NewJoernGraph graph,
    IReadOnlyDictionary<string, string> options)
  {
    AnalysisId = analysisId;
    Graph = graph;
    Options = options;
  }
}
```

说明：

- 这份设计稿默认阶段执行时有统一上下文
- 条件接口只在当前上下文中做判断，不负责跨阶段总调度

字段解释：

- `AnalysisId`
  - 当前分析轮次标识。
  - 用于让一次规则执行具备统一上下文身份。
- `Graph`
  - 当前使用的 `NewJoern` 图对象。
  - 提供所有图事实来源。
- `Options`
  - 当前运行选项集合。
  - 用于把宿主配置、阶段开关、实验选项等传入行为实现。

## 7. 依赖的已有对象

这份设计稿默认下面这些对象已经在删除语义模型中存在：

- `DeletionTarget`
- `Mark`
- `MarkTarget`
- `DecisionCandidate`

当前默认的前期版 `DeletionTarget` 设计要点是：

1. 保留 `TargetId` 和 `DeletionTargetKind` 作为删除语义身份与粒度。
2. 直接使用 Roslyn 的 `SyntaxKind` 和 `TextSpan` 表达结构事实。
3. 直接携带 `Code` 片段快照，便于解释和调试。
4. 直接携带 `SyntaxAnnotation TrackingAnnotation`，便于在不可变语法树重写时稳定追踪目标。
5. 不再额外拆独立 `TargetSnapshot`，也不额外引入 `ResolvedDeletionTarget`。

本稿只关注规则对象模型，不重复展开这些对象的完整定义。

## 8. 规则实例

### 8.1 `DEL-SOBJ-001`

```csharp
namespace NewJoern.Deletion.Rules;

public static class BuiltInRules
{
  public static RuleDefinition DeleteSObjectExpressionRule() =>
    new(
```

## 10. 基于外部资料的优化结论

这次收口参考了两类外部依据：

- Microsoft / Fowler 关于领域模型、值对象、反贫血领域模型的建议
- NRules 关于规则元数据、优先级、标签等宿主治理信息的分离方式

落到本稿后的优化结论如下：

1. `RuleMetadata` 与阶段 `Behavior` 分离
   - 元数据只回答“这条规则是谁”
   - `Behavior` 才回答“这条规则怎么判断”
   - 这样可以避免把说明字段和执行字段混在一起

2. `RuleMarkingSpec / RulePropagationSpec / RuleDecisionSpec` 明确按值对象对待
   - 它们都是声明式边界
   - 不保存运行期状态
   - 应该保持不可变，便于比较、缓存、审计、测试

3. 分类、标签、优先级这类宿主治理信息不回灌到核心对象模型
   - `RuleCategory` 已删除
   - 如果未来需要标签、展示分组、加载筛选、调度优先级，优先放到 `RuleHost` / `RuleManifest`
   - 不把这些治理字段重新塞回 `DeletionDecision` 的核心规则语义对象

4. 阶段引擎保持薄，规则判断放在 `Behavior`
   - 阶段引擎只负责组织流程、筛选规则、聚合结果
   - 复杂命中、复杂传播、复杂裁决仍由规则自身的 `Behavior` 完成
   - 这样可以避免阶段引擎膨胀成新的业务中心

5. `Behavior` 可以依赖查询服务，但不应直接吞掉整个框架
   - 真实实现时允许通过构造函数注入图查询服务、辅助分析服务、配置读取服务
   - 但 `Behavior` 的输入输出仍应收口到当前规则模型定义的参数与结果对象
   - 不应让 `Behavior` 直接变成“拿到容器后到处调服务”的脚本对象

## 11. 设计总结

这份设计稿的关键取舍如下：

1. 保留统一规则模型
2. 保留阶段边界
3. 允许复杂行为策略
4. 避免规则对象自己变成胖执行器

当前不把 `RuleSafetySpec` 作为正式对象模型的一部分。

原因：

- 当前先收紧最小规则对象模型
- 安全策略可以先作为设计说明、阶段约束或引擎侧默认规则存在
- 等规则治理和风险控制需求进一步稳定后，再决定是否单独提升为正式 `Spec`

一句话收口：

**`RuleDefinition` 统一承载规则知识，三个 Behavior 接口承载复杂阶段行为，Result 对象承载阶段判断结果，各阶段引擎负责真正执行。**
