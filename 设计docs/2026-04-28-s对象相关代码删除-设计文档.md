# `s` 对象相关代码删除设计

## 1. 这份文档负责什么

这份文档只讨论一个专项场景：基于 `NewJoern` CPG 删除实例对象 `s` 及其直接相关代码。

它负责：

1. 说明这个场景依赖哪些图事实。
2. 说明这个场景如何命中、扩展、裁决和回写。
3. 固定这个场景的风险边界和验证方式。

共享规则主定义见 `2026-04-29-代码删除规则设计.md`，内部正式模型见 `2026-04-29-内部模型总览.md`。

## 2. 场景定义

当前目标是删除实例对象 `s` 以及所有直接依赖 `s` 的代码。

第一期只处理直接依赖，不处理跨过程长链数据传播。当前重点命中下面这些结构：

- `s`
- `s.c`
- `s.F()`
- `s.A.B.C`
- `s.GetValue().ToString()`
- `Foo(a, s.c, b)` 里作为实参出现的 `s` 相关表达式

这里的核心判断不是“文本里有没有 `s`”，而是“图上能不能稳定证明这段结构以 `s` 为根”。

## 3. 目标、非目标与约束

### 3.1 目标

1. 基于 CPG 图准确识别所有以 `s` 为根的直接相关表达式。
2. 把图上的命中节点映射回源码位置并执行稳定删除。
3. 删除后尽量保留仍然语法合法且有意义的剩余表达式。
4. 无法稳定保留时，退化为删除整条语句。

### 3.2 非目标

1. 不处理“某个局部变量最初来自 `s`，再被传给多层函数”的跨过程长链传播。
2. 不处理自动常量折叠、整棵表达式树语义优化。
3. 不在第一期直接修改 CPG 图并重新生成源码。

### 3.3 约束

1. 必须先基于 `NewJoern` CPG 图分析，不允许只靠字符串替换。
2. 删除后输出代码必须保持语法合法。
3. 高风险结构位置无法稳定保留时，优先删整条语句。
4. 同一目标若承受多个标记，必须进入决策阶段聚合。

## 4. 图证据前提

从当前仓库代码可以确认以下事实：

- `NewJoern\Frontend\Roslyn\RoslynCSharpFrontend.Expressions.cs` 会为调用创建 `CALL` 节点，并写入 `METHOD_FULL_NAME`。
- 同一前端文件也会为方法引用创建 `METHOD_REF` 节点。
- `NewJoern\Graph\NodePropertyNames.cs` 已定义 `CODE`、`METHOD_FULL_NAME`、`LINE_NUMBER`、`COLUMN_NUMBER` 等属性，可作为命中与回写定位依据。
- 当前默认图会叠加 `callgraph`、`controlflow`、`reachingdef` 等 overlays。

这说明当前仓库并不缺结构化中间表示，缺的是一套基于这些图事实做 `s` 对象删除的场景规则。

## 5. 场景流程

```text
源码 -> NewJoern 产图 -> 图上识别 s 相关节点 -> 初始标记 -> 传播扩展 -> 决策裁决 -> 源码回写
```

各步骤职责如下：

| 步骤 | 处理 | 输出 |
| --- | --- | --- |
| 图命中 | 查找名字为 `s` 的标识及其相关访问链 | 初始表达式标记 |
| 传播扩展 | 扩到语句、分支、作用域、跨文件依赖 | 收敛后的标记集合 |
| 决策裁决 | 聚合同一目标上的多个标记 | 最终删除动作 |
| 源码回写 | 把最终动作映射回源码位置并重写 | 新源码 |

## 6. 场景步骤职责

### 6.1 图命中

作用：在 CPG 图上识别所有以 `s` 为根的代码片段。

输入：CPG 图、目标对象名 `s`。

输出：命中节点集合。

命中规则：

1. 节点源码片段是 `s` 本身，命中。
2. `CALL`、成员访问或相关表达式的接收者是 `s`，或以 `s` 开头的访问链，命中。
3. 参数节点的实参表达式根节点命中 `s`，则参数位置命中。

这一阶段只负责产生 `DEL-SOBJ-001` 的初始表达式标记，不直接删除，不直接扩展到父语句。

### 6.2 父结构升级

作用：把命中的子节点扩展到删除时真正需要处理的父结构。

重点边界：

- 二元表达式
- 赋值右值
- 参数列表
- `if` 条件
- `for` 初始化项

例如：

```csharp
z = s.c + 1;
```

图上先命中 `s` 与 `s.c`，扩展后命中整个右值中的左操作数。

```csharp
for (int i = s.c; ; )
{
  DoWork();
}
```

图上命中 `s.c` 后，扩展到 `for` 初始化项，再由删除策略判定整句删除。

### 6.3 删除裁决

作用：在决策阶段决定删子表达式、删整条语句、删更大结构，还是跳过。

决策原则：

1. 原始命中的 `s` 表达式标记通常强于纯传播标记，但不自动覆盖一切。
2. 语句级动作不因为优先级高就天然成立，必须先判断表达式级保留是否已经失效。
3. 若多个动作都成立，先比较删除语义和风险，再比较优先级。
4. 无法稳定裁决时，跳过优先于激进删除。

### 6.4 源码回写

这一步负责把图节点映射成源码区间，并执行最小安全重写。

这里固定两条约束：

1. 分析和回写必须基于同一份源码与图快照。
2. 同一代码重复执行两次，第二次不应再产出新增删除。

## 7. 本场景对共享规则的绑定

| 规则 | 在本场景里的职责 |
| --- | --- |
| `DEL-SOBJ-001` | 标记以 `s` 为根的表达式 |
| `DEL-SOBJ-002` | 表达式删除后若父结构残缺，则升级为整句标记 |
| `DEL-PROP-001` | 若被标记表达式定义出的结果被后续语句唯一使用，则继续传播 |
| `DEL-PROP-002` | 若某个分支或 block 只剩待删语句，则提升为更大结构标记 |
| `DEL-PROP-003` | 若标记对象跨文件被唯一依赖，则在目标文件形成新标记 |
| `DEL-GENERIC-001` | 无法稳定回写时保守跳过 |

这份文档不重复定义这些规则的共享语义，只说明它们在本场景里的落点。

## 9. 风险与保守边界

| 风险 | 触发条件 | 应对 |
| --- | --- | --- |
| 局部化简不符合用户直觉 | 某些复杂表达式存在多种保留方式 | 先固化规则，再补测试 |
| 图命中不全 | 成员访问链未完全映射到可识别节点 | 先做直接场景，逐步扩展 |
| 间接依赖未删 | `s` 先赋值给别名再使用 | 第一期开成非目标 |
| 回写后结构残缺 | 位置在条件头、初始化项或返回位置 | 升级为删整句或保守跳过 |

当前开放问题：

1. 是否需要支持 `s` 之外的任意目标对象名。
2. 是否继续扩展到 `ReachingDef` 驱动的间接依赖删除。
3. `return s.c;` 这类语句是统一删整句，还是允许替换为默认值。
4. 跨文件传播是否默认关闭，只在显式开关下启用。

## 10. 验证矩阵

| 设计点 | 验证方式 | 成功标准 |
| --- | --- | --- |
| 删除成员访问 | 单元测试 | `z = s.c + 1;` 改写为 `z = 1;` |
| 删除方法调用 | 单元测试 | `x = s.GetValue() + 2;` 改写为 `x = 2;` |
| 删除参数 | 单元测试 | `Foo(a, s.c, b);` 改写为 `Foo(a, b);` |
| 删除条件语句 | 单元测试 | `if (s.IsReady) { Run(); }` 整句删除 |
| 删除控制语句初始化项 | 单元测试 | `for (int i = s.c; ; )` 整句删除 |
| 标记传播到后续语句 | 单元测试 | 若后续语句唯一依赖被删结果，则形成新标记 |
| 标记传播到分支 / block | 单元测试 | 一个分支只剩待删语句时，分支整体删除 |
| 多标记决策裁决 | 单元测试 | 同一目标同时命中原始标记和传播标记时，仍只产出一个最终动作 |

## 11. 当前结论

这个专项的核心不是“删掉所有出现 `s` 的文本”，而是“基于图证据识别以 `s` 为根的结构，再做最小安全回写”。

表达式级保留优先，结构无法稳定保留时升级为语句级删除，仍无法稳定时保守跳过。
