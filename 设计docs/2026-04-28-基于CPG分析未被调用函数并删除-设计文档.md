# 不可达函数链删除设计

## 1. 这份文档负责什么

这份文档只讨论一个专项场景：基于 `NewJoern` CPG 调用图识别入口不可达的函数链，并按完整函数定义删除。

它负责：

1. 说明这个场景依赖哪些调用图事实。
2. 说明入口识别、可达性分析、链路归并、决策和回写如何组织。
3. 固定这个场景的风险边界和验证方式。

共享规则主定义见 `2026-04-29-代码删除规则设计.md`，内部正式模型见 `2026-04-29-内部模型总览.md`。

## 2. 场景定义

当前目标是找出“从有效入口不可达”的函数，并继续归并那些只服务于不可达链的下游函数，最终按完整函数定义删除。

这里的核心判断不是“函数名有没有被文本引用”，而是“能不能从可信入口沿调用图走到这个函数”。

## 3. 目标、非目标与约束

### 3.1 目标

1. 基于 CPG 调用图识别项目中的不可达函数。
2. 递归找出只服务于不可达链的下游函数，并合并成待删除函数集合。
3. 至少能识别并删除两条不可达函数链。
4. 回写源码时按完整函数定义删除，避免只删函数体片段。

### 3.2 非目标

1. 不解决反射、字符串注册、外部配置回调这类静态调用图不可见的动态调用。
2. 不在第一期处理字段、属性、类型、局部函数、lambda 的死代码裁剪。
3. 不以“代码从未执行过”作为标准，第一期只做“从入口不可达”的静态分析。

### 3.3 约束

1. 必须基于 `NewJoern` 生成的 CPG 图分析，不允许只靠文本搜索函数名。
2. 删除前必须给出待删除函数列表及其不可达理由。
3. 目标函数若可能被外部入口、框架回调、反射或导出 API 使用，默认不删。
4. 同一函数、语句或结构若叠加多个标记，必须进入决策阶段。

## 4. 图证据前提

从当前仓库代码可以确认以下事实：

- `NewJoern\Frontend\Roslyn\RoslynCSharpFrontend.Expressions.cs` 在创建 `CALL` 节点时会写入 `METHOD_FULL_NAME`。
- `NewJoern\Overlays\CallGraphOverlay.cs` 会把 `CALL` 节点解析并连到目标 `METHOD` 节点，形成 `CALL -> METHOD` 边。
- `NewJoern\Validation\NewJoernGraphValidator.cs` 已对 `CALL -> METHOD` 的边形状和 `METHOD_FULL_NAME` 对齐做了验证。

这说明当前仓库已经有可用的调用图基础，不需要重新发明“谁调用了谁”的底层结构。

## 5. 场景流程

```text
源码 -> NewJoern 前端产图 -> CallGraphOverlay 连边 -> 入口识别 -> METHOD 可达性遍历 -> 初始死定义标记 -> 传播扩展 -> 决策裁决 -> 源码删除
```

各步骤职责如下：

| 步骤 | 处理 | 输出 |
| --- | --- | --- |
| 入口识别 | 选出可信分析根节点 | 入口方法集合 |
| 可达性遍历 | 沿调用图遍历所有可达方法 | 可达方法集合 |
| 候选归并 | 求出不可达方法并归并其下游链 | 待删除方法集合与链路说明 |
| 传播扩展 | 扩到辅助语句、结构、跨文件依赖 | 收敛后的标记集合 |
| 决策裁决 | 处理共享下游函数和多标记冲突 | 最终删除动作 |
| 源码删除 | 把函数节点映射回源码并按整定义删除 | 新源码 |

## 6. 场景步骤职责

### 6.1 入口识别

作用：给调用图分析提供可信根节点。

第一期默认入口来源：

- 编译入口点
- 公共根 API 白名单
- 用户手工补充入口集合

这一步的关键不是把入口找得越多越好，而是把“确定应保留的入口”找稳。入口不足时，系统应先报告，不要直接删源码。

### 6.2 可达性遍历

作用：从入口出发遍历调用图，标记所有可达函数。

可达判定规则：

1. 方法本身是入口，则可达。
2. 如果可达方法内部存在 `CALL`，且 `CALL` 边连到目标 `METHOD`，则目标 `METHOD` 可达。
3. 重复直到没有新增方法。

需要处理的边界：

- 递归调用
- 环调用
- 外部方法

### 6.3 不可达链归并

作用：把“单个不可达函数”提升成“不可达函数链”。

归并规则：

1. 某方法不可达，则进入候选集合。
2. 候选方法调用的下游函数，如果没有任何来自可达集合的调用边，也进入候选集合。
3. 若某下游函数同时被可达方法调用，则从候选集合中排除。
4. 最终源码删除阶段由 `DEL-DEAD-001` 执行完整函数定义删除。

这一步的关键价值是识别共享下游函数。某个函数虽然被不可达链引用，只要它仍被可达区调用，就必须保留。

### 6.4 源码删除

这一步负责把方法节点映射回源码位置，并执行完整函数定义删除。

这里固定两条约束：

1. 同一轮分析必须基于同一份图快照。
2. 同一份源码重复执行两次，第二次不应再发现已删除的方法。

## 7. 本场景对共享规则的绑定

| 规则 | 在本场景里的职责 |
| --- | --- |
| `DEL-DEAD-001` | 标记从入口不可达的函数定义 |
| `DEL-PROP-001` | 若某语句或定义唯一依赖已标记死定义，则继续形成新标记 |
| `DEL-PROP-002` | 若某个局部结构只服务于死函数相关逻辑，则提升为更大结构标记 |
| `DEL-PROP-003` | 若死定义跨文件被唯一依赖，则在目标文件形成新标记 |
| `DEL-GENERIC-001` | 图证据不足或回写不安全时保守跳过 |

这份文档不重复定义这些规则的共享语义，只说明它们在本场景里的落点。

## 8. 关键样例

### 8.1 两条不可达函数链

```csharp
void MainEntry()
{
  Run();
}

void Run()
{
}

void F()
{
  G();
}

void G()
{
  H();
}

void H()
{
}

void X()
{
  Y();
}

void Y()
{
}
```

分析结果：

- 可达链：`MainEntry -> Run`
- 不可达链 1：`F -> G -> H`
- 不可达链 2：`X -> Y`

删除后源码只保留可达链。

### 8.2 共享下游函数必须保留

```csharp
void MainEntry()
{
  Live();
}

void Live()
{
  Shared();
}

void DeadA()
{
  Shared();
}

void Shared()
{
}
```

分析结果：

- `DeadA` 不可达，可删除。
- `Shared` 虽被 `DeadA` 调用，但同时被 `Live` 调用，因此必须保留。

## 9. 风险与保守边界

| 风险 | 触发条件 | 应对 |
| --- | --- | --- |
| 动态调用漏标活代码 | 反射、事件、配置驱动调用 | 默认白名单保守保留 |
| 入口识别不全 | 框架入口未录入 | 先报告，后删除 |
| 图解析不完整 | unresolved call 较多 | 对 unresolved 方法降级保留 |
| 共享下游误删 | 公共工具函数同时被活代码和死链调用 | 决策阶段默认保留 |

当前开放问题：

1. 第一批入口白名单如何定义。
2. 是否把 `public` 方法默认视为保留，还是只对白名单保留。
3. 后续是否扩展到属性访问器、局部函数和 lambda 对应的 `METHOD` 节点。
4. `DEL-PROP-001` 在这个场景里是否需要区分“调用依赖”和“返回值依赖”。
5. 跨文件传播是否只允许在同一项目内发生。

## 10. 验证矩阵

| 设计点 | 验证方式 | 成功标准 |
| --- | --- | --- |
| 入口可达函数保留 | 单元测试 | 入口链上的函数全部保留 |
| 单条不可达链识别 | 单元测试 | 能识别 `F -> G -> H` 全链不可达 |
| 共享工具函数保留 | 单元测试 | 若 `H` 同时被活代码调用，则不删除 `H` |
| 两条死函数链输出 | 单元测试 | 至少输出两条独立不可达链 |
| 源码整函数删除 | 集成测试 | 删除的是完整函数定义，不留半截代码 |
| 死定义标记传播 | 单元测试 | 若某语句或定义仅依赖死函数结果，则形成新标记 |
| 跨文件死定义传播 | 集成测试 | 若唯一依赖发生在别的文件，目标文件也能形成正确标记 |
| 多标记决策裁决 | 单元测试 | 同一共享下游函数同时承受多个标记时，仍只产出一个最终动作 |

## 11. 当前结论

这个专项的核心不是“找出没被文本引用的函数”，而是“基于可信入口和调用图证明某个函数不可达，再按整定义安全删除”。

共享下游函数保留、动态调用保守处理、入口不足先报告，是这个场景避免误删的三条底线。
