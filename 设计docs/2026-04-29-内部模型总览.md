# 内部模型总览

## 1. 这份文档负责什么

这份文档是删除框架内部正式模型的主入口，只负责三组对象：

1. 删除目标语义模型。
2. 规则定义模型。
3. 阶段运行结果模型。

这份文档不负责外部规则包契约，也不负责实现级 C# 草图。外部契约见 `2026-04-29-外部规则接口-CSharp-设计稿.md`，内部代码草图见 `2026-04-29-规则对象模型-CSharp-设计稿.md`。

## 2. 为什么要收成一份总文档

原先三份对象文档分别讨论“删谁”“规则怎么定义”“阶段产出什么”，边界本身没有问题，但它们高度互相引用：

1. `Mark` 必须引用 `DeletionTarget`。
2. `RuleDefinition` 必须约束 `Mark`、`DecisionOutcome` 这类阶段对象的来源。
3. `DecisionOutcome` 和 `DeletionPlan` 又必须回指前两组对象。

如果继续拆成三份并列主文档，读者会在正式语义、运行结果和规则模型之间来回跳转。现在把三组对象收成一份总览，主线会更稳定。

## 3. 三组模型的关系

可以把内部正式模型理解成下面这条链：

```text
DeletionTarget
  -> RuleDefinition
    -> Mark / PropagationTrace / DecisionOutcome / DeletionPlan
```

它们分别回答三个问题：

| 组别 | 回答的问题 | 代表对象 |
| --- | --- | --- |
| 删除目标模型 | 当前正在讨论删谁 | `DeletionTarget` |
| 规则定义模型 | 规则如何参与标记、传播、决策 | `RuleDefinition` |
| 阶段结果模型 | 规则运行后产出什么 | `Mark`、`DecisionOutcome`、`DeletionPlan` |

## 4. 删除目标语义模型

### 4.1 角色

删除目标语义模型只表达“目标是谁”，不表达“规则打算怎么处理它”。

当前结论：

- `DeletionTarget` 是删除语义讨论的基础对象。
- `TargetRelation` 表达目标之间的语义关系。

### 4.2 对象清单

```text
DeletionTarget
  └─ TargetRelation[]
```

### 4.3 `DeletionTarget`

```csharp
using Microsoft.CodeAnalysis;
using Microsoft.CodeAnalysis.CSharp;
using Microsoft.CodeAnalysis.Text;

public sealed record DeletionTarget(
  string TargetId,
  DeletionTargetKind Kind,
  string FilePath,
  SyntaxKind SyntaxKind,
  TextSpan Span,
  string Code,
  SyntaxAnnotation TrackingAnnotation,
  IReadOnlyCollection<TargetRelation> Relations);
```

作用：

- 表示删除语义世界里的一个正式目标。
- 它是标记、传播、决策、计划都会引用的基础对象。

字段要点：

- `TargetId`：删除语义里的稳定身份，不能直接拿图节点 id 替代。
- `Kind`：当前目标粒度，例如 `Expression`、`Statement`、`Definition`。
- `FilePath`：所在源码文件。
- `SyntaxKind`：目标对应的 Roslyn 语法种类。
- `Span`：目标在源码中的 Roslyn 文本区间。
- `Code`：目标被抽出时的源码片段快照。
- `TrackingAnnotation`：用于在不可变语法树重写过程中稳定追踪当前目标。
- `Relations`：目标和其他目标之间的关系。

### 4.4 `TargetRelation`

```csharp
public sealed record TargetRelation(
  string RelationId,
  string SourceTargetId,
  string DerivedTargetId);
```

作用：

- 表达一个删除目标派生到另一个删除目标。
- 前期不再维护独立关系枚举，只保留最小派生信息。

### 4.5 典型例子

源码：

```csharp
z = s.c + 1;
```

这里先讨论的是表达式目标本身：

- `DeletionTarget.Kind = Expression`
- `DeletionTarget.SyntaxKind = SyntaxKind.SimpleMemberAccessExpression`
- `DeletionTarget.Code = "s.c"`

如果后续发现只删表达式会破坏父结构，则创建新的 `Statement` 级目标，并通过 `DerivedTargetId` 把两者关联起来，而不是原地改旧目标粒度。

当前约束：前期不额外建父子、包含或控制拥有关系，避免删除目标语义层过早膨胀成第二套结构图。

### 4.6 这组对象不负责什么

下面这些内容不进入删除目标语义模型：

- 规则身份
- 规则命中结果
- 传播链证据
- 最终删除动作
- 技术重写策略

## 5. 规则定义模型

### 5.1 角色

规则定义模型只讨论“规则自身如何被定义”，不直接承载运行结果。

当前结论：

- `RuleDefinition` 是统一规则定义对象。
- `RuleMetadata` 只承载规则身份与说明。
- `RuleMarkingSpec / RulePropagationSpec / RuleDecisionSpec` 承载阶段静态边界。
- `IMarkBehavior / IPropagationBehavior / IDecisionBehavior` 只承载复杂行为入口。

### 5.2 对象清单

```text
RuleDefinition
  ├─ RuleMetadata
  ├─ SyntaxBreakage[]
  ├─ RuleMarkingSpec
  ├─ RulePropagationSpec
  ├─ RuleDecisionSpec
  ├─ IMarkBehavior?
  ├─ IPropagationBehavior?
  └─ IDecisionBehavior?
```

### 5.3 `RuleDefinition`

```csharp
using Microsoft.CodeAnalysis.CSharp;

public sealed class RuleDefinition
{
  public RuleMetadata Metadata { get; }
  public IReadOnlyCollection<SyntaxKind> PotentialBreakageSyntaxKinds { get; }
  public RuleMarkingSpec Marking { get; }
  public RulePropagationSpec Propagation { get; }
  public RuleDecisionSpec Decision { get; }

  public IMarkBehavior? MarkBehavior { get; }
  public IPropagationBehavior? PropagationBehavior { get; }
  public IDecisionBehavior? DecisionBehavior { get; }
}
```

作用：

- 统一承载一条规则的正式定义。
- 作为阶段引擎读取规则的唯一入口。
- 显式声明这条规则命中后可能破坏哪些语法结构。

当前约束：

- `RuleDefinition` 自己不直接执行完整流程。
- 它不是值对象，因为里面含有行为对象引用。
- `PotentialBreakageSyntaxKinds` 只表达规则级静态结构风险种类，不直接记录某次运行里的实际命中节点。

字段要点：

- `Metadata`：规则身份与说明。
- `PotentialBreakageSyntaxKinds`：当前规则命中后可能破坏的 Roslyn 语法种类清单。
- `Marking / Propagation / Decision`：规则在三个阶段的静态边界。
- `...Behavior`：难以纯字段表达的复杂行为入口。

### 5.4 `PotentialBreakageSyntaxKinds`

```csharp
using Microsoft.CodeAnalysis.CSharp;

public IReadOnlyCollection<SyntaxKind> PotentialBreakageSyntaxKinds { get; }
```

作用：

- 描述某条规则命中后可能影响的 Roslyn 语法种类。
- 这是一个前期轻量占位字段，不引入额外 breakage 对象壳。
- 把“删完后可能打到哪些语法种类”前置到规则定义层，而不是只散落在专项说明和运行期备注里。

字段要点：

- `SyntaxKind` 指 `Microsoft.CodeAnalysis.CSharp.SyntaxKind`。
- 当前只允许放 `SyntaxNode` 对应的 kind。
- 不放 token kind。
- 不放 trivia kind。
- 它只表达规则级静态风险边界，不表达运行期具体节点实例。

### 5.5 粗粒度语法基础模型的边界

`PotentialBreakageSyntaxKinds` 只回答“可能影响哪些语法种类”。

如果后续要把语法结构本体也单独建模，建议继续沿 Roslyn 语法家族收口，但只保留轻量骨架：

- `StatementSyntax`
- `BlockSyntax`
- `IfStatementSyntax`
- `For/ForEach/While/DoStatementSyntax`
- `ReturnStatementSyntax`
- `InvocationExpressionSyntax`
- `MemberDeclarationSyntax`
- `TypeDeclarationSyntax`
- `ParameterListSyntax`
- `ArgumentListSyntax`

这里的关键边界是：

1. 基础语法模型负责表达“结构是什么”。
2. `PotentialBreakageSyntaxKinds` 负责表达“风险会落到哪些 Roslyn 语法种类上”。
3. 二者都不追求 Roslyn 的 token / trivia / full-fidelity 级细节。
4. 语义和 `IOperation` 可以作为运行期证据来源，但不进入这个静态风险字段。

### 5.6 推荐样例

```csharp
new[]
{
  SyntaxKind.IfStatement,
  SyntaxKind.InvocationExpression,
  SyntaxKind.MethodDeclaration,
  SyntaxKind.ParameterList
}
```

### 5.7 `RuleMetadata`

```csharp
public sealed record RuleMetadata(
  string RuleId,
  string Name,
  string Description,
  string Version,
  bool IsEnabledByDefault);
```

作用：

- 规则身份卡。
- 只表达这条规则是谁，不表达规则怎么判断。

当前结论：

- 不再保留 `RuleCategory`。
- 分类、展示分组、宿主筛选不进入内部正式规则对象。

### 5.8 三个 `Spec`

#### `RuleMarkingSpec`

```csharp
public sealed record RuleMarkingSpec(
  IReadOnlyCollection<DeletionTargetKind> SupportedTargetKinds,
  IReadOnlyCollection<string> RequiredGraphCapabilities);
```

作用：定义标记阶段的静态边界与依赖能力。

#### `RulePropagationSpec`

```csharp
public sealed record RulePropagationSpec(
  bool IsEnabled,
  IReadOnlyCollection<DeletionTargetKind> SourceKinds,
  IReadOnlyCollection<DeletionTargetKind> ResultKinds,
  int MaxPropagationDepth);
```

作用：定义传播阶段的静态边界。

#### `RuleDecisionSpec`

```csharp
public sealed record RuleDecisionSpec(
  DecisionActionKind PreferredAction,
  bool AllowEscalation,
  IReadOnlyCollection<DeletionTargetKind> EscalationTargets,
  bool MayBreakParentStructure);
```

作用：定义决策阶段的静态边界与默认倾向。

当前结论：

- 技术重写策略应放到 `Planning / SourceRewrite`，不回灌到内部正式规则对象。

### 5.9 三个 `Behavior` 接口

```csharp
public interface IMarkBehavior
{
  MarkMatchResult Evaluate(
    RuleExecutionContext context,
    RuleDefinition rule,
    DeletionTarget target);
}

public interface IPropagationBehavior
{
  PropagationCheckResult Evaluate(
    RuleExecutionContext context,
    RuleDefinition rule,
    Mark sourceMark,
    DeletionTarget sourceTarget);
}

public interface IDecisionBehavior
{
  DecisionCheckResult Evaluate(
    RuleExecutionContext context,
    RuleDefinition rule,
    DecisionCandidate candidate);
}
```

统一说明：

- `Spec` 是静态边界。
- `Behavior` 是复杂行为入口。
- 阶段引擎负责组织流程。
- 规则对象自己不承担流程编排。

### 5.10 这组对象不负责什么

下面这些内容不属于规则定义模型：

- 标记实例结果
- 传播链实例结果
- 决策结果
- 技术删除计划
- 重写动作
- 某次运行后真实发生的结构破坏实例

## 6. 阶段运行结果模型

### 6.1 角色

阶段运行结果模型只讨论“规则在各阶段运行后真正产出的结果对象”。

当前结论：

- `Mark` 是标记阶段正式结果。
- `PropagationTrace` 是传播阶段证据对象。
- `DecisionCandidate` 是决策阶段输入聚合对象。
- `DecisionOutcome` 是决策阶段正式输出。
- `DeletionPlan` 是计划阶段正式输出。

### 6.2 对象清单

```text
Mark
PropagationTrace
DecisionCandidate
DecisionOutcome
DeletionPlan
  └─ RewriteAction[]
```

### 6.3 `Mark`

```csharp
public sealed record Mark(
  string MarkId,
  string RuleId,
  string TargetId,
  string Reason,
  int PropagationDepth);
```

作用：表达“某条规则命中了某个目标”。

当前约束：

- `Mark` 只表达命中事实。
- 不在 `Mark` 里直接放最终删除动作。
- 不再单独区分命中视角枚举或来源枚举；前期这些差异直接体现在 `Reason` 和 `PropagationDepth` 上。

### 6.4 `PropagationTrace`

```csharp
public sealed record PropagationTrace(
  string TraceId,
  string SourceMarkId,
  string SourceTargetId,
  string ProducedMarkId,
  string ProducedTargetId,
  string Reason,
  int Depth);
```

作用：

- 保存传播链证据。
- 让传播不是静默地产生新标记。

### 6.5 `DecisionCandidate`

```csharp
public sealed record DecisionCandidate(
  string TargetId,
  DeletionTarget Target,
  IReadOnlyCollection<Mark> Marks,
  IReadOnlyCollection<PropagationTrace> Traces,
  bool StructuralBreakDetected);
```

作用：

- 围绕同一个目标聚合全部证据。
- 作为决策阶段的正式输入对象。

### 6.6 `DecisionOutcome`

```csharp
public sealed record DecisionOutcome(
  string TargetId,
  DecisionActionKind Action,
  string Reason,
  string? WinningRuleId,
  DeletionTarget? FinalTarget);
```

作用：决策阶段正式输出。

### 6.7 `DeletionPlan`

```csharp
public sealed record DeletionPlan(
  string PlanId,
  string TargetId,
  DecisionActionKind Action,
  string FilePath,
  IReadOnlyCollection<RewriteAction> RewriteActions,
  string Reason);
```

作用：表达技术可执行删除计划。

### 6.8 `RewriteAction`

```csharp
public sealed record RewriteAction(
  string ActionId,
  RewriteActionKind Kind,
  string FilePath,
  int StartLine,
  int StartColumn,
  int EndLine,
  int EndColumn,
  string? ReplacementText);
```

### 6.9 阶段内部结果对象

运行期还会出现一些阶段内部结果对象：

- `MarkMatchResult`
- `PropagationCheckResult`
- `DecisionCheckResult`

它们用于 `Behavior` 返回值和阶段内部判断，但不替代正式的阶段出口对象。

## 7. 三组模型之间的边界

为了避免后续文档漂移，这里固定三条边界：

1. 删除目标模型只回答“目标是谁”。
2. 规则定义模型只回答“规则如何参与各阶段”。
3. 阶段结果模型只回答“规则运行后产出什么”。

任何一组对象都不应该吞掉另外两组的职责。

## 8. 内部模型与外部契约的边界

内部正式模型包括：

- `DeletionTarget`
- `RuleDefinition`
- `Mark`
- `PropagationTrace`
- `DecisionCandidate`
- `DecisionOutcome`
- `DeletionPlan`

外部规则包契约包括：

- `RuleManifest`
- `IMarkRule`
- `IPropagateRule`
- `IDecisionPolicy`
- `MarkTarget`
- `RuleTargetKind`

如果某类对象的主要目的是给宿主和规则包之间做稳定边界，它应属于外部契约层，而不是内部正式模型层。

## 9. 审核重点

这份总文档建议优先审这 6 个问题：

1. `DeletionTarget` 是否继续作为删除语义基础对象。
2. “目标升级”是否坚持采用派生新目标，而不是原地改粒度。
3. `RuleDefinition` 是否继续保留为统一规则入口。
4. 三个 `Spec` 是否已经压到足够小。
5. `DecisionOutcome` 是否只保留决策结果，不带重写细节。
6. `DeletionPlan` 是否应继续作为计划阶段正式出口。
