# 从 joern-master 收缩到简化 CPG 实现指南

## 1. 这份文档负责什么

这份文档只回答一件事：

```text
如果目标不是复刻完整 Joern，而是只保留 CPG 图的依赖建图能力，
并改用 Roslyn Syntax / Semantic / Operation 三层作为事实来源，
应该怎么从 joern-master 收缩出一套更小、更稳、更适合当前 NL 的实现。
```

这份文档负责：

1. 明确简化版 CPG 的目标范围。
2. 说明 joern-master 里哪些代码是核心抓手，哪些属于外围设施。
3. 说明 Roslyn 三层各自提供什么事实，如何投影成统一 CPG。
4. 给出建议的模块拆分、建图步骤、最小节点/边集合、实现顺序。

这份文档不负责：

1. CLI 形状。
2. 导出格式。
3. 扫描器、规则库、SARIF、控制台交互。
4. 删除框架的规则语义与执行策略。

## 2. 总体结论

当前推荐方向是：

```text
抄 Joern 的建图思路，不抄它的产品壳；
抄 Roslyn 的事实来源，不抄完整编译器外观；
最终只保留一套 canonical CPG 作为分析主模型。
```

进一步压缩成一句话：

```text
Roslyn 三层提供真相，简化版 CPG 提供统一分析图。
```

这意味着：

1. 不直接把 Roslyn 对象暴露给上层分析规则。
2. 不把 Syntax、Semantic、Operation 做成三套并列主模型。
3. 不复制 Joern 的 CLI、导出、扫描、产品化外围设施。
4. 只保留前端 lowering、图节点/边、少量 overlay/layer、统一查询入口。

## 3. 简化版 CPG 的目标范围

### 3.1 要保留的能力

简化版 CPG 只保留下面这些能力：

1. 文件、命名空间、类型、方法、参数、局部变量的建图。
2. 表达式与语句的 AST 建图。
3. 控制结构与条件的结构化建图。
4. 调用、字段访问、索引访问、赋值、比较、逻辑运算的统一 operator/call 建模。
5. 声明与引用之间的绑定。
6. 基础 CFG。
7. 基础 data-flow / reaching-def 输入图。
8. 面向后续规则系统的统一查询入口。

### 3.2 明确不保留的能力

这版实现明确排除：

1. CLI。
2. 图导出器。
3. 扫描器、规则数据库、批处理外壳。
4. 控制台、脚本壳、交互命令。
5. SARIF、dot 导出、可视化、vectors、scan/reporting。
6. 与当前 NL 无关的多语言前端。

## 4. 从 joern-master 抄什么，不抄什么

## 4.1 必须抄的部分

从 `joern-master` 必须抄的是“前端 lowering 思路”和“CPG 统一形状”，不是产品壳。

重点抓手有三类。

### A. x2cpg 通用建图骨架

核心文件：

- `joern-cli/frontends/x2cpg/src/main/scala/io/joern/x2cpg/AstNodeBuilder.scala`
- `joern-cli/frontends/x2cpg/src/main/scala/io/joern/x2cpg/AstCreatorBase.scala`

必须抄的思想：

1. `CALL` 是统一表达动作节点。
2. operator 也建成 `CALL(<operator>.*)`。
3. `CONTROL_STRUCTURE` 不只存 `code`，还显式挂 `condition`。
4. `AST` child 顺序是正式契约。
5. `ARGUMENT`、`CONDITION`、`FOR_INIT`、`FOR_UPDATE`、`TRY_BODY` 这类边由建图骨架统一补。

这两个文件已经说明：

- `operatorCallNode(...)` 是一等基础设施。
- `controlStructureAst(...)` 会统一补 `condition edges`。
- `forAst(...)` / `tryCatchAst(...)` 这类 helper 会给结构节点补特定边。

### B. csharpsrc2cpg 的 C# lowering 逻辑

核心文件：

- `.../csharpsrc2cpg/astcreation/AstForExpressionsCreator.scala`
- `.../csharpsrc2cpg/astcreation/AstForStatementsCreator.scala`
- `.../csharpsrc2cpg/astcreation/AstForDeclarationsCreator.scala`

必须抄的思想：

1. `BinaryExpression` 不扁平化，直接 lower 成 operator call。
2. `If / While / For / Switch` 的 condition 单独成 AST，并显式挂 condition edge。
3. `foreach`、`using`、属性 getter/setter 这种语法糖要在前端 lower 成更规则的 CPG 形状。
4. 表达式与语句不是原样复制 parser 输出，而是做规范化投影。

### C. semanticcpg 的统一查询与 operator 分类

核心文件：

- `semanticcpg/.../language/nodemethods/AstNodeMethods.scala`
- `semanticcpg/.../language/operatorextension/package.scala`
- `semanticcpg/.../language/operatorextension/OpNodes.scala`

必须抄的思想：

1. 上层分析依赖统一的 AST/query API，而不是前端内部对象。
2. operator 按 assignment/arithmetic/fieldAccess/arrayAccess 等语义类别再分层。
3. 查询层以 canonical CPG 为中心，不暴露编译器对象。

## 4.2 不要抄的部分

从 `joern-master` 不建议带进来的部分：

1. `joern`、`joern-cli` 顶层产品壳。
2. `joern-parse`、`joern-flow`、`joern-slice`、`joern-scan` 等脚本入口。
3. `joern-export`、`dotgenerator`、`sarif`、`querydb`、`console`。
4. 多语言前端。
5. 与产品交付有关的 distro/test harness。

这些都不是“建图依赖能力”的核心。

## 5. 为什么要依赖 Roslyn 三层

## 5.1 Syntax 层解决什么

Syntax 层负责提供：

1. 源码结构长什么样。
2. 条件、参数、成员访问、声明、列表等结构角色。
3. AST child 顺序与源码位置。

对简化版 CPG 来说，Syntax 层最适合产出：

- `AST`
- `ORDER`
- `CONTROL_STRUCTURE` 的结构外壳
- `BLOCK`
- `METHOD` / `TYPE_DECL` / `LOCAL` / `PARAMETER`
- 各种表达式/语句的 source span

如果没有 Syntax 层，`if (x > i)`、`Foo(a, b)`、`a[i]`、`obj.Prop` 这类结构都很容易被压扁。

## 5.2 Semantic 层解决什么

Semantic 层负责提供：

1. 名字到底绑定到谁。
2. 表达式类型是什么。
3. 调用到底选中了哪个重载。
4. 字段、属性、事件、局部、参数、类型之间的引用身份。
5. 跨文件/跨 project 的声明与引用关系。
6. `AnalyzeDataFlow`、`AnalyzeControlFlow` 这类编译器级分析入口。

对简化版 CPG 来说，Semantic 层最适合产出：

- `REF`
- `TYPE`
- `TYPE_REF`
- `METHOD_REF`
- `CALL.methodFullName`
- `CALL.signature`
- `Expression.typeFullName`
- 声明与引用的唯一身份

如果没有 Semantic 层，图里就只能看到“长得像调用”“长得像标识符”，看不到它真实引用什么。

## 5.3 Operation 层解决什么

Operation 层负责提供：

1. 已经做过语义归一化的操作树。
2. 不同语法写法背后的统一语义动作。
3. operator、assignment、conversion、conditional、property/index access 等动作的统一表示。

对简化版 CPG 来说，Operation 层最适合产出：

- `<operator>.greaterThan`
- `<operator>.equals`
- `<operator>.logicalAnd`
- `<operator>.assignment`
- conversion call
- normalized invocation/property/index access semantics

如果没有 Operation 层，你仍然可以只靠 Syntax + Semantic 自己做 lowering，但表达式特判会越来越多。

## 6. 最推荐的实现分层

推荐的建图架构是：

```text
Roslyn Syntax
  -> 结构骨架 AST

Roslyn Semantic
  -> symbol/type/binding facts

Roslyn Operation
  -> normalized semantic actions

以上三层统一 lowering 到:
  -> Canonical CPG
```

这里的关键点是：

1. Roslyn 三层只是事实来源。
2. 最终对上层暴露的只能是一套 canonical CPG。
3. 上层规则、传播、切片、决策都只面向 CPG API。

## 7. 简化版 CPG 的最小节点集合

建议第一版至少保留下面这些节点类型。

### 7.1 结构节点

- `MetaData`
- `File`
- `NamespaceBlock`
- `TypeDecl`
- `Method`
- `MethodParameterIn`
- `MethodReturn`
- `Block`
- `ControlStructure`
- `JumpTarget`

### 7.2 表达式/引用节点

- `Call`
- `Identifier`
- `Literal`
- `FieldIdentifier`
- `Local`
- `Member`
- `MethodRef`
- `TypeRef`
- `Unknown`

### 7.3 不建议第一版就细分成大量专用节点

Joern 的真实做法已经说明：

1. operator 优先统一成 `Call`。
2. 很多语法动作优先投影到 `Call + children`。
3. 不需要先造一堆 `BinaryExprNode`、`CompareNode`、`ConditionNode` 自定义类型。

## 8. 简化版 CPG 的最小边集合

建议第一版至少保留：

- `AST`
- `ARGUMENT`
- `REF`
- `CONDITION`
- `CONTAINS`
- `CFG`

如果第一版就要支撑 `for` / `try-catch-finally` 的更完整控制依赖输入，可继续保留：

- `FOR_INIT`
- `FOR_UPDATE`
- `FOR_BODY`
- `TRY_BODY`
- `CATCH_BODY`
- `FINALLY_BODY`

是否第一版就保留这些专门边，取决于你是否想像 Joern 一样把 CFG 辅助边前置到 lowering 骨架里。

## 9. 表达式与条件的建议 lowering 规则

## 9.1 比较与逻辑表达式

不要保留为扁平叶子集合。

应该 lower 成：

```text
CALL(<operator>.greaterThan)
CALL(<operator>.lessThan)
CALL(<operator>.equals)
CALL(<operator>.logicalAnd)
CALL(<operator>.logicalOr)
CALL(<operator>.logicalNot)
```

左右操作数或子条件通过 `ARGUMENT` + `AST` 连接。

## 9.2 条件结构

`if` / `while` / `for` / `switch` 统一规则：

1. `CONTROL_STRUCTURE` 仍保留结构外壳。
2. 条件根表达式必须单独成 AST。
3. 通过 `CONDITION` 边把控制结构和条件根显式连起来。

例如：

```text
CONTROL_STRUCTURE(IF)
  ├─ CONDITION -> CALL(<operator>.greaterThan)
  ├─ AST child 1: condition root
  ├─ AST child 2: then block
  └─ AST child 3: else block
```

## 9.3 赋值、字段访问、索引访问

建议统一成 operator/call：

- `a = b` -> `<operator>.assignment`
- `obj.x` -> `<operator>.fieldAccess`
- `arr[i]` -> `<operator>.indexAccess`

这是 Joern 的现成模式，后续查询、传播、切片会简单得多。

## 10. 只保留“建图依赖能力”时，第一版可以不做什么

为了把范围卡住，第一版可以明确不做：

1. 全量 schema 覆盖。
2. 多语言兼容抽象。
3. 导出层。
4. CLI 和参数编排。
5. querydb / scan / reporting。
6. 大而全的 language extension。
7. 所有高阶数据流、污点、切片产品壳。

第一版只需要保证：

1. 图能正确建出来。
2. 条件、operator、调用、引用、控制结构不丢语义。
3. 后续分析层可以稳定沿图查询。

## 11. 推荐模块拆分

建议拆成下面 5 个模块，不要更多：

### 11.1 `Frontend`

负责：

- Roslyn parse
- Roslyn semantic binding
- Roslyn operation lowering
- 生成 canonical CPG diff / graph

### 11.2 `Graph`

负责：

- 节点/边定义
- node property names
- 图容器
- AST child order / edge dedup 规则

### 11.3 `Lowering`

负责：

- expression lowering
- statement lowering
- declaration lowering
- operator mapping
- control-structure mapping

### 11.4 `Semantics`

负责：

- Roslyn `SemanticModel` 查询封装
- symbol identity
- type resolution
- declaration/reference resolution
- operation-to-call normalization

### 11.5 `AnalysisCore`

负责：

- 最小 CFG
- 最小 reaching-def 输入准备
- 上层查询入口

第一版里，这个模块不需要做成 Joern 那样的大平台，只要足够支撑依赖建图能力。

## 12. 推荐实现顺序

### 第 1 步：先做 canonical graph，不做外围壳

只保留：

- 节点
- 边
- graph builder
- source location

### 第 2 步：先打通 expression/control structure lowering

优先处理：

- `Identifier`
- `Literal`
- `Call`
- `Assignment`
- `Binary operators`
- `If`
- `While`
- `For`
- `Return`
- `FieldAccess`
- `IndexAccess`

### 第 3 步：补 declaration/reference/type binding

优先处理：

- `Method`
- `Parameter`
- `Local`
- `Member`
- `TypeDecl`
- `REF`
- `METHOD_REF`
- `TYPE_REF`

### 第 4 步：补最小 CFG / reaching-def 输入

这一阶段只做最小能跑的：

- `if`
- `loop`
- `return`
- `try/catch/finally`

### 第 5 步：最后才补 query ergonomics

也就是：

- `condition`
- `whenTrue`
- `whenFalse`
- operator 分类查询
- `astChildren` / `argument`

## 13. 对 NL 当前问题的直接意义

如果按这套路线做，`if (x > i)` 这类问题的解法会天然变成：

1. Syntax 保留 `if` 与 `x > i` 的结构。
2. Semantic 绑定 `x`、`i` 和任何成员/调用引用。
3. Operation 或 lowering 把 `x > i` 统一成 `<operator>.greaterThan`。
4. CPG 里再用 `condition` 边把 `IF` 和这个 operator call 连起来。

这时：

- 查询能知道条件内部结构。
- 传播能知道 guard 是什么。
- 改写前的 plan 层也能知道应该回到哪个语法根去处理。

## 14. 最终建议

如果目标只是“新的 CPG，也就是简化版 joern-master”，当前最合理的路线是：

```text
抄 Joern 的 canonical CPG 形状，
抄 Roslyn 三层的事实来源能力，
删掉 Joern 的产品壳和外围设施。
```

这条路线的价值在于：

1. 不重复发明 AST / symbol / operator 语义。
2. 不再把 `if (x > i)` 压扁成只剩 code 文本。
3. 不把上层规则系统暴露给 Roslyn 细节。
4. 外部只维护一套 CPG 主模型，认知负担最低。

一句话收口：

```text
Joern 负责告诉你 CPG 应该长什么样，
Roslyn 负责告诉你 C# 代码到底是什么意思，
新的简化版 CPG 负责把这两者收成一套可分析、可继续演进的统一图。
```
