# 代码删除规则体系设计

## 1. 这份文档的角色

这份文档只负责共享删除规则体系。

它回答的是：

1. 删除规则如何分阶段组织
2. 规则编号如何定义
3. 标记、传播、决策之间如何配合
4. 什么情况下升级删除单位，什么情况下保守跳过

它不负责：

- 总架构和模块边界
- 某个专项场景的完整流程
- 某组内部对象的全部字段定义
- 外部规则包的完整 C# 接口

总架构归 `初步设计.md`，专项流程归两份 `2026-04-28` 文档，内部正式模型归 `2026-04-29-内部模型总览.md`，外部接口归 `2026-04-29-外部规则接口-CSharp-设计稿.md`。

## 2. 设计目标

当前已经存在两类明确删除需求：

1. 删除实例对象 `s` 及其相关代码
2. 删除从入口不可达的函数链

这两类需求虽然命中方式不同，但共享同一套底层约束：

1. 都必须先基于 `NewJoern` 图证据命中
2. 都可能从一个原始命中扩展到更大删除单位
3. 都可能在同一目标上叠加多个标记
4. 都需要把“为什么删”和“删到什么级别”解释清楚

所以规则体系必须从一开始就把“标记、传播、决策、计划、执行”分开。

## 3. 规则语义子流程

### 3.1 子流程

```text
图分析 -> 标记 -> 传播 -> 决策 -> 计划 -> 执行
```

### 3.2 各阶段职责

| 阶段 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| 标记 | 产生原始待删标记 | 直接给最终删除动作 |
| 传播 | 基于已有标记形成新标记 | 直接替代决策阶段 |
| 决策 | 聚合多标记并产出唯一最终动作 | 真正改源码 |
| 计划 | 把语义动作映射成技术计划 | 重新定义规则语义 |
| 执行 | 真实改源码 | 重新裁决删什么 |

### 3.3 核心约束

这里固定一条核心规则：

```text
优先级 != 决策
优先级 ∈ 决策阶段
```

优先级只能用于：

1. 决策阶段里两个已成立候选动作的排序
2. 同类目标上的稳定 tie-break
3. 保证同样输入得到同样输出

优先级不能用于：

1. 替代标记来源分析
2. 替代传播链分析
3. 替代删除单位裁决
4. 让传播标记直接压过尚未分析清楚的原始标记

## 4. 标记对象与删除单位

规则体系允许处理的目标粒度如下：

| 粒度 | 说明 | 典型示例 |
| --- | --- | --- |
| 表达式 | 可做局部重写 | `s.c`、`s.GetValue()` |
| 语句 | 整句删除 | `x = s.GetValue() + 2;` |
| 定义 | 函数或成员定义 | 不可达函数 `F` |

一个规则在某阶段命中的粒度，不一定等于最终删除单位。

典型例子：

1. 原始标记先命中表达式
2. 决策阶段发现删表达式会破坏父结构
3. 最终动作升级成删语句

## 5. 规则分类

### 5.1 标记规则

标记规则负责回答：

```text
先把什么东西打上待删标记
```

它必须说明：

1. 依赖哪些图节点、边或属性
2. 命中什么粒度
3. 命中后产出什么原始标记

### 5.2 传播规则

传播规则负责回答：

```text
已有标记在什么条件下扩到别的位置
```

它必须显式说明：

1. 传播起点是什么
2. 扩散条件是什么
3. 传播后形成的新目标是什么
4. 什么情况下停止传播

### 5.3 决策阶段

决策阶段不再回答“有没有命中”，而是回答：

1. 同一目标上有哪些标记
2. 哪些是原始标记，哪些是传播标记
3. 哪些只是证据，哪些足以触发删除
4. 最终删什么单位
5. 无法稳定裁决时是升级还是跳过

## 6. 规则最小字段

每条规则至少要表达下面这些信息：

| 字段 | 说明 |
| --- | --- |
| `RuleId` | 稳定规则编号，例如 `DEL-SOBJ-001` |
| `RuleName` | 人类可读名称 |
| `RuleKind` | `Marking` 或 `Propagation` |
| `MatchCondition` | 命中条件 |
| `MarkTarget` | 命中目标视图或命中目标粒度说明 |
| `PropagationSource` | 传播起点，标记规则可为空 |
| `PropagationCondition` | 传播条件 |
| `PotentialBreakageSyntaxKinds` | 当前规则命中后可能破坏的 Roslyn 语法种类清单 |
| `DeletionUnit` | 候选删除单位 |
| `Examples` | 最少两个示例 |
| `Verification` | 如何验证 |

## 7. 第一批共享规则

第一批共享规则先收口为 7 条：

| 规则 ID | 类型 | 作用 |
| --- | --- | --- |
| `DEL-SOBJ-001` | Marking | 标记实例对象 `s` 的直接相关表达式 |
| `DEL-SOBJ-002` | Propagation | `s` 相关标记导致结构残缺时扩展为整句标记 |
| `DEL-DEAD-001` | Marking | 标记从入口不可达的函数定义 |
| `DEL-PROP-001` | Propagation | 已标记目标传播到唯一依赖它的语句 |
| `DEL-PROP-002` | Propagation | 已标记语句传播到更大结构 |
| `DEL-PROP-003` | Propagation | 已标记目标沿跨文件依赖传播 |
| `DEL-GENERIC-001` | Guard | 无法稳定回写时保守跳过 |

下面只固定共享语义，不重复展开两个专项文档的全部上下文。

## 8. 规则定义

### 8.1 `DEL-SOBJ-001`

- `RuleKind`：`Marking`
- 作用：标记以实例对象 `s` 为根的直接相关表达式
- 典型命中：
  - `s`
  - `s.c`
  - `s.F()`
  - `s.A.B.C`
  - `Foo(a, s.c, b)` 中的参数表达式
- 候选删除单位：表达式
- 默认策略：先删子表达式，父表达式还能稳定成立时保留剩余结构
- 失败兜底：若保留后结构非法，则交给 `DEL-SOBJ-002` 或决策阶段升级

示例：

```csharp
z = s.c + 1;
```

删除后：

```csharp
z = 1;
```

```csharp
Foo(a, s.c, b);
```

删除后：

```csharp
Foo(a, b);
```

### 8.2 `DEL-SOBJ-002`

- `RuleKind`：`Propagation`
- 作用：当表达式删除会破坏父结构时，把父语句整体标记
- 传播起点：`DEL-SOBJ-001` 产出的表达式标记
- 典型场景：
  - `if` 条件
  - `for` 初始化项
  - `return` 表达式
  - 无合法重写路径的控制语句头部
- 候选删除单位：语句

示例：

```csharp
if (s.IsReady)
{
  Run();
}
```

删除后：

```csharp

```

### 8.3 `DEL-DEAD-001`

- `RuleKind`：`Marking`
- 作用：标记从入口不可达的函数定义
- 命中前提：
  - 调用图已建立
  - 目标函数不在入口可达集内
  - 下游函数没有来自可达区域的活引用
- 候选删除单位：函数定义
- 默认策略：整函数定义删除
- 失败兜底：若函数可能被外部入口、回调、反射或白名单使用，则保守保留
- 典型 `PotentialBreakageSyntaxKinds`：
  - `SyntaxKind.MethodDeclaration`：若方法声明本身被整段删除，则会直接破坏成员定义结构
  - `SyntaxKind.InvocationExpression`：若函数体内只删局部调用片段会导致调用表达式残缺，则需要升级删除单位

### 8.4 `DEL-PROP-001`

- `RuleKind`：`Propagation`
- 作用：把已标记目标传播到唯一依赖它的语句
- 传播条件：
  - 目标语句关键输入仅来自已标记对象
  - 删除该输入后目标语句无法独立成立
- 候选删除单位：语句
- 默认策略：能局部保留则保留，不能保留则删整句

### 8.5 `DEL-PROP-002`

- `RuleKind`：`Propagation`
- 作用：把集中出现的语句标记提升为作用域或条件分支标记
- 传播条件：
  - 同一 block 或分支中的多条语句已标记
  - 删除后外层结构只剩空壳或失去语义
- 候选删除单位：作用域 / 条件分支

### 8.6 `DEL-PROP-003`

- `RuleKind`：`Propagation`
- 作用：沿跨文件依赖传播已有标记
- 传播条件：
  - 目标依赖发生在别的文件
  - 该依赖没有别的活来源
- 候选删除单位：语句 / 定义
- 风险级别：高

### 8.7 `DEL-GENERIC-001`

- 类型：通用守卫规则
- 作用：图证据不足、源码范围冲突或回写不安全时，明确跳过
- 默认策略：不改代码，只输出跳过原因

### 8.8 `PotentialBreakageSyntaxKinds` 的使用边界

`PotentialBreakageSyntaxKinds` 属于规则定义时的静态风险声明，不属于运行期 `Mark` 结果。

它的职责是：

1. 让规则作者预先声明“这条规则最可能破坏哪些 Roslyn 语法种类”。
2. 让决策阶段快速知道哪些结构需要额外关注。
3. 当结构不可修复时，明确要求升级为删除更大结构，而不是硬保留半截代码。

如果某次实际命中是否真的破坏了结构，还需要结合运行期证据、调用图和决策输入再判断，不能把静态清单直接当作最终事实。

当前建议写法是直接使用：

1. `IReadOnlyCollection<SyntaxKind> PotentialBreakageSyntaxKinds`
2. 这里只放 `SyntaxNode` 对应的 kind，不放 token 和 trivia

## 9. 决策阶段模型

### 9.1 决策阶段要解决什么

真实冲突场景通常不是“谁优先”这么简单，而是：

1. 标记是否真的成立
2. 标记来自原始命中还是传播
3. 一个标记只是证据，还是足以触发删除
4. 最终删表达式、删语句、删作用域还是删定义

### 9.2 决策阶段输入

决策阶段至少读取：

- 原始标记
- 传播标记
- 传播链证据
- 目标当前粒度
- 可重写性判断
- 结构残缺信息

这些输入的正式对象归属见 `2026-04-29-内部模型总览.md`。

### 9.3 决策阶段输出

决策阶段必须保证：

```text
同一 TargetId 最终只产出一个 FinalAction
```

允许的动作包括：

- `Delete`
- `Skip`

### 9.4 决策原则

1. 原始标记通常强于传播标记，但不自动覆盖一切
2. 删除单位更大的动作不天然更正确，只表示影响范围更大
3. 若多个动作都成立，先比较删除语义，再比较风险，再比较优先级
4. 无法稳定裁决时，保守跳过优先于激进删除

## 10. 规则执行顺序

推荐顺序固定如下：

1. 图分析给出候选事实
2. 标记规则先产生原始标记
3. 传播规则循环执行，直到不再产出新标记
4. 决策阶段聚合同一目标上的全部标记
5. 计划阶段把语义动作转为技术计划
6. 执行阶段真实改源码
7. 若执行前发现结果不稳定，则触发回退策略

## 11. 非功能约束

### 11.1 可解释性

每轮删除至少要输出：

- 命中规则 ID
- 原始标记数
- 传播新增标记数
- 决策冲突数
- 最终动作数
- 跳过数和跳过原因

### 11.2 可验证性

每条规则至少要有：

1. 一个局部保留示例
2. 一个删整句或整定义示例
3. 传播规则至少一个“原始标记 -> 新标记”示例

### 11.3 保守性

下列情况默认跳过：

1. 图证据不足
2. 命中范围无法唯一映射回源码
3. 删除后语法无法保证合法
4. 外部活引用无法排除
