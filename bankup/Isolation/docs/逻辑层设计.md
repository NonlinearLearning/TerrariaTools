# 逻辑层设计

## 1. 目标

本文按当前代码事实，定义 `Logic` 层的职责、主要上下文、阶段能力落点和与其他层的边界。

## 2. 当前主线事实

当前 `src/Logic` 主要上下文：

- `Analysis`
- `Decision`
- `Marking`
- `Propagation`
- `Rules`
- `Rewrite`
- `Workflow`
- `Workspaces`

当前关键事实：

- `Logic.Analysis` 负责快照装配和输入描述
- `Logic.Analysis.Engine` 负责语言层、Pass、Layer、查询执行、语义执行
- `Logic.Workflow` 已成为 Rewrite workflow 的稳定阶段能力承载区
- `Logic.*.Events` 负责把领域事件串成可发布序列

## 3. Logic 层应该承载什么

- 单阶段可复用能力
- 消费 Domain 模型后产出中间结果的能力
- 多个用例都可能复用的内部组合逻辑
- 阶段间装配、事件序列、工作流支撑对象
- 不适合暴露为 DTO、也不应沉到 Infrastructure 的内部 API

### 4.4 Logic.Workspaces

关键锚点：

- `src/Logic/Workspaces/WorkspaceContextBuilder.cs`
- `src/Logic/Workspaces/WorkspaceRuleDefaultsBuilder.cs`

当前结论：

- 默认规则输入的装配已经留在 `Logic.Workspaces`
- `WorkspaceRuleDefaultsBuilder` 已通过 `IEnabledRuleFactory` 退化为薄适配器
- `Application` 通过 builder 进入领域对象构建链

### 4.5 Logic.Rules

当前代码已落地 `src/Logic/Rules/`，当前承载：

- 规则目录 `RuleCatalog`
- 规则描述 `RuleDescriptor`
- 启用规则工厂 `EnabledRuleFactory`
- 预设提供器 `IRulePresetProvider`
- 工作区默认预设 `WorkspaceDefaultRulePreset`
- RewriteWorkflow 预设 `RewriteWorkflowRulePreset`
- 目录接口 `IRuleCatalog`
- 工厂接口 `IEnabledRuleFactory`

当前还未落地：

- 更多 workflow 细分预设

目标边界：

- `Domain.Rules` 继续定义 `RuleCode / EnabledRule / RuleScope / RuleExecutionPolicy / RuleSet`
- `Logic.Rules` 负责“系统支持哪些规则、默认怎么装、不同场景启哪组规则”
- `Logic.Workspaces` 退化为工作区输入适配与规则集装配层

当前 workspace 默认路径已经固定为：

- `WorkspaceDefaultRulePreset` 定义默认规则组合
- `WorkspaceContextBuilder` 负责 preset + 显式输入合并与去重
- `WorkspaceRuleDefaultsBuilder` 负责把输入交给 `EnabledRuleFactory`

当前 marking 主链已经开始复用统一规则入口：

- `RuleTargetCandidateBuilder` 通过 `RuleCatalog` 获取规则描述
- `RuleTargetCandidateBuilder` 通过 `EnabledRuleFactory` 构造启用规则
- `ArchitectureTests` 已守护该主链不再局部 new 默认策略对象

当前 workflow 主链已经开始复用统一规则入口：

- `RewriteWorkflowMarkingPreparer` 通过 `RewriteWorkflowRulePreset` 获取规则集名称
- `RewriteWorkflowMarkingPreparer` 会校验规则码是否属于 RewriteWorkflow 场景
- `RewriteWorkflowAppService` 会通过 `RewriteWorkflowRulePreset` 解析请求规则码
- `RewriteWorkflowAppService` 会通过 `RewriteWorkflowRulePreset` 规范化保护规则
- `ArchitectureTests` 已守护 workflow 规则集名称不再局部硬编码

这样做的收益：

- 默认值集中，新增规则只改一个主入口
- 规则扩展路径显式，避免在多个 builder 中散落硬编码
- 后续更容易把规则边界转成 `ArchitectureTests`

## 5. 与其他层的分界

- “这是什么、有哪些不变量” -> `Domain`
- “这一段能力如何复用和组装” -> `Logic`
- “请求 DTO 如何进来、结果 DTO 如何出去” -> `Application`
- “Roslyn 前端、IO、持久化如何实现” -> `Infrastructure`

## 6. 不进入 Logic 的内容

- DTO 边界和对外契约
- 聚合不变量和核心领域规则
- Roslyn 具体实现、文件系统、持久化
- 把整个工作流重新塞回单一 AppService
