本文按当前代码事实，定义 `Application` 层的职责、当前服务边界、DTO 规则、默认下沉判断和验证口径。

## 3. Application 层职责

- 接收请求 DTO
- 调用 `Domain + Logic` 暴露的稳定能力
- 完成 DTO 与领域对象之间的映射
- 组织一次用例执行顺序
- 返回调用方可消费的契约对象

## 4. 当前关键判断

### 4.1 RewriteWorkflowAppService

当前职责锚点：

- `src/Application/Services/RewriteWorkflowAppService.cs`

当前承担：

- 取工作区、分析快照、规则命中目标
- 进入传播阶段、决策阶段、工件装配阶段
- 映射 `Propagation / Decision / Plan / Result / Evidence / Report / DomainEvents` DTO

当前下沉结果：

- 传播解析在 `Logic.Workflow.RewriteWorkflowPropagationStage`
- 决策解释在 `Logic.Workflow.RewriteWorkflowDecisionStage`
- 工件装配在 `Logic.Workflow.RewriteWorkflowArtifactAssembler`

### 4.2 WorkspaceContextAppService

当前职责锚点：

- `src/Application/Services/WorkspaceContextAppService.cs`

当前承担：

- 接收 `CreateWorkspaceContextRequest`
- 通过 `WorkspaceContextBuilder` 和 `WorkspaceRuleDefaultsBuilder` 构建领域工作区
- 持久化 `WorkspaceContext`
- 返回 `WorkspaceContextDto`

当前结论：

- 默认规则输入装配已经从 AppService 本体下沉到 `Logic.Workspaces`
- 应用层保持 DTO 边界与仓储调用

### 4.3 Analysis 与 Rewrite 契约边界

当前 `Application.Contracts` 已明确区分：

- 分析快照 DTO
- 决策 DTO
- 执行 DTO
- 传播 DTO
- 改写产物 DTO
- 输出证据与报告 DTO
- 工作区与工作流 DTO

## 5. 强约束

- `Application` 只依赖 `Domain + Logic`
- 应用层不承载聚合不变量
- 应用层不直接承载 Roslyn、文件系统、持久化、CLI 等技术实现
- 同模块复用能力优先下沉到 `Logic` 或 `Domain`
- `AppService -> AppService` 串调不作为默认复用方式

## 6. 什么时候继续下沉

出现以下信号时，优先继续下沉：

- 同一规则在多个 AppService 中重复
- AppService 已经开始解释 `RuleSet / RuleScope / RuleExecutionPolicy`
- AppService 已经在做候选推理、传播决策、冲突决策
- 一段逻辑已经脱离 DTO 边界，成为稳定内部能力
