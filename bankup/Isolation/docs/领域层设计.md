## 4. 富领域模型规则

- 聚合根、实体和值对象默认承载业务语义、不变量和行为
- 事实快照、边界记录、兼容载体允许偏贫血表达
- 规则默认优先回收到 `Domain` 或 Domain Policy
- 传播、决策、计划、执行、证据、报告的生命周期边界优先在 Domain 收紧

当前已跑通的模式：

- `RewriteDecision`：领域策略解释输入，聚合施加结果
- `RewritePlan`：编译前校验计划项与排序
- `RewriteResult`：执行结果围绕计划语义回收
- `VerificationEvidence`：零证据不可 collect
- `RunReport`：未挂接 evidence 不可 finalize
- `ChangeCandidate`：传播边界和覆盖态在领域层约束

### 4.1 Rule 模型边界补充

当前规则相关边界按下面划分：

- `Domain.Rules`
  - 定义 `RuleCode`、`EnabledRule`、`RuleScope`、`RuleExecutionPolicy`、`RuleSet`
  - 负责规则对象合法性、兼容性和共享语义
- `Logic`
  - 负责把边界输入装配成可运行规则
  - 负责不同场景的规则目录与规则预设

后续收敛原则：

- 规则对象语义继续留在 `Domain`
- 场景默认值、规则注册表、规则工厂留在 `Logic`
- `Domain` 不吸收“workspace 默认规则”“workflow 规则预设”这类场景策略

## 5. 不进入 Domain 的内容

- DTO、请求、响应、控制器契约
- Roslyn `SyntaxNode` / `SemanticModel`
- 文件系统、DI、持久化实现
- 纯流程编排对象
- 为了组装用例顺序存在的薄协调逻辑

## 6. 与其他层的边界

- “这是什么、有哪些不变量” -> `Domain`
- “这一步如何复用完成” -> `Logic`
- “这个用例如何编排和返回 DTO” -> `Application`
- “Roslyn / IO / 持久化如何兑现” -> `Infrastructure`
