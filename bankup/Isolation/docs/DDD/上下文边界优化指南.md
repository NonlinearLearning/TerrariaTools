# 上下文边界优化指南

## 1. 目的

本文按当前代码事实说明：这个仓库的上下文边界该如何继续优化。

## 2. 当前上下文轮廓

当前主上下文已经稳定表现为：

- `Workspaces`
- `Analysis`
- `Marking`
- `Propagation`
- `Decision`
- `Execution`
- `Rewrite`
- `Output`
- `Rules`

## 3. 当前边界判断

### 3.1 稳定边界

以下边界当前已经比较稳定：

- `Workspaces`：工作区输入、路径、默认规则、项目与引用注册
- `Decision`：风险解释、人工复核、批准/拒绝裁决
- `Execution`：计划、执行结果、执行终态
- `Output`：证据收集、风险摘要、审计报告
- `Workflow` 主链角色边界：assembler / stage / compiler / executor / event-stage / recorder

### 3.2 持续收敛边界

以下边界仍在持续收敛：

- `Propagation`：传播事实、边界解释、覆盖态、传播轨迹
- `Rewrite`：重写产物和传播边界之间的语义关系
- `Analysis` 与 `Analysis.Engine`：对外事实与内部引擎的表达层级

## 4. 当前优化原则

1. 先守住 `Domain -> Logic -> Application -> Infrastructure`
2. 先把边界判断写清，再决定拆类还是拆目录
3. 优先回收不变量，再做目录美化
4. 优先补测试和 ArchitectureTests，再补新抽象
5. 优先让 `RewriteWorkflowAppService` 继续保持薄壳，再谈新的 workflow 外围抽象

## 5. 当前最值钱的下一步

- 继续把剩余历史 DDD 文档改成当前事实版
- 继续把目录整改结论写回项目级约束文档
- 继续减少局部规则构造、局部字符串和跨阶段语义在单个类型中的堆积

## 6. 当前已经落地的优化结果

截至当前代码状态，以下优化已经完成：

1. `Application.Services` 不互调 AppService
2. 稳定规则码解析已经从 `RewriteWorkflowAppService` 和 `WorkspaceContextAppService` 收口
3. `Logic.Workflow` 已拆出传播、决策、计划、执行、证据、报告、事件七段主链
4. `RewriteWorkflowArtifactAssembler` 已保持单一公开装配入口
5. `WorkflowEventSequenceBuilder`、`IDomainEventRecorder`、`RewriteWorkflowStageResults` 已形成稳定收尾链
6. 上述事实已进入 `tests/ArchitectureTests/Program.cs`
